## Executive Decision

Devil IDE should not define MVP as an AI-native, plugin-capable, collaborative, remote IDE. The documentation and evidence support a narrower and safer MVP: a local-first deterministic IDE core with trustworthy workspace open/edit/save behavior, scalable viewport-first text handling, proposal-mediated mutation, projection-only UI, and metadata-only observability.

The production path should then move in strict dependency order:

| Stage | Release label | Outcome | Primary gate |
| --- | --- | --- | --- |
| 0 | Foundation lock | Architecture truth, ADR status, and dependency policy are unambiguous | Phase 0/1 accepted, Phase 2 partial, Phase 3 not accepted recorded clearly |
| 1 | MVP | Safe local IDE core with generalized proposal execution for current mutation classes | Phase 2B complete and all global gates pass |
| 2 | Core alpha | User-facing editor shell and release-quality native UX evidence | Renderer, IME, clipboard, focus, accessibility, and input-to-paint evidence exists |
| 3 | Language beta | Semantic fabric and supervised LSP become accepted | Phase 3 evidence complete; LSP output is DTO/proposal-only |
| 4 | AI beta | Policy-bound AI execution with privacy inspector and replayable context manifests | AI cannot mutate directly; air-gap and provider egress tests pass |
| 5 | Production RC | Durable storage, diagnostics, policy profiles, replay, migration, privacy, and CI hardening | Active feature set has production evidence and `cargo deny check` in CI |
| 6 | Production 1.0 | Local-first AI IDE production release | R0-R4 complete plus R8 hardening for active features |
| 7 | Platform production expansions | Plugins, collaboration, and remote development | Each feature ships only after its own ADR, policy, protocol, tests, evidence, and hardening gates |

Recommended production definition: Production 1.0 should mean a local-first AI IDE with safe editing, semantic/LSP support, proposal-only AI edits, privacy inspection, durable metadata, and operational diagnostics. WASM plugins, real-time collaboration, and edge remote development should remain disabled or clearly post-1.0 unless their full Phase 5-7 gates are completed.

## Non-Negotiable Constraints

1. UI remains projection-only. It renders projections and emits `CommandDispatchIntent`; it does not own editor sessions, text buffers, workspace actors, proposal execution, LSP workers, index state, or durable writes.
2. Editor/text owns text mutation. Edits flow through typed editor APIs, transactions, snapshots, undo groups, and buffer versions.
3. Workspace/VFS owns durable file authority. Saves and file mutations require trust, path policy, fingerprint/version preconditions, workspace generation, capability, principal, correlation, and causality.
4. All non-user-direct mutation is proposal-mediated. AI, LSP, plugins, terminal commands, collaboration, remote, and semantic refactors must produce proposals instead of mutating buffers or disk directly.
5. Unsupported routes fail closed. Terminal, plugin, remote, collaboration, command-like LSP, and mixed routes stay denied until their own activation gates exist.
6. Observability and audit are metadata-only by default. Raw source, secrets, unbounded previews, full command payloads, embeddings, vectors, provider outputs, and replay source snapshots are not persisted by default.
7. Background work must not block input, viewport projection, proposal validation, or saves. Indexing, LSP, AI retrieval, collaboration replay, and storage consumers use bounded queues, cancellation, stale suppression, and degradation metadata.
8. Placeholder crates remain inert. `devil-agent`, `devil-tracker`, `devil-memory`, `devil-plugin`, `devil-lsp`, `devil-terminal`, `devil-collaboration`, and `devil-remote` require accepted ADRs, dependency-policy entries, protocol contracts, tests, ownership gates, and evidence before runtime behavior lands.
9. Vector indexing remains deferred. Embeddings, vector stores, model-backed retrieval, provider-output persistence, and vector dependencies require a later ADR and evidence package.
10. Full repository gates run after every execution package: `cargo run -p xtask -- check-deps`, `cargo fmt --all --check`, `cargo check --workspace --all-targets`, `cargo test --workspace --all-targets`, and `cargo clippy --workspace --all-targets -- -D warnings`.

## MVP Definition

The MVP is a safe local IDE core, not the complete AI-native 2026 platform.

### MVP Includes

| Capability | MVP behavior |
| --- | --- |
| Workspace trust and open flow | Open a trusted local workspace with workspace identity, generation, path policy, and shallow tree metadata |
| File navigation | Projection-fed file tree and open-buffer projection sufficient for edit/save workflows |
| Editing | Editor-owned transactions, snapshots, undo/redo, buffer versions, dirty tracking, and viewport-first projection |
| Large files | Files above the full-cache budget open in degraded mode; UI receives bounded viewport slices, not full source |
| Saving | Save through `AppComposition::save_active_buffer` -> `SaveWorkflowService` -> `WorkspaceActor::save_file_with_proposal` |
| Conflict safety | Expected fingerprint, file content version, workspace generation, buffer version, snapshot id, capability, principal, correlation, and causality are required |
| Generalized proposals | Lifecycle service supports validate, preview, approve, reject, cancel, apply, rollback, stale, conflict, denied, failed, and cancelled states |
| Non-save mutation | Open-buffer and closed-file proposal apply are implemented only through editor/workspace authorities; unsupported routes deny before side effects |
| Batch mutation | Batch planner validates route support, ordering, dependency edges, atomicity claims, rollback policy, preflight, audit, and partial-failure records |
| UI approval projection | UI can render proposal list/preview/warnings/result and emit approve/reject/cancel intents without owning lifecycle state |
| Audit | Metadata-only lifecycle and proposal audit records are emitted before success is reported |
| Evidence | All MVP gates pass and Phase 2 evidence is updated to show runtime apply gaps are closed |

### MVP Excludes

| Excluded from MVP | Reason |
| --- | --- |
| AI-generated edits | Requires generalized proposals, semantic fabric, privacy manifests, tracker records, provider policy, and air-gap tests |
| LSP runtime behavior | Phase 3 and LSP supervision are not accepted; runtime needs bounded supervised workers and DTO/proposal routing evidence |
| Semantic fabric acceptance | Current Phase 3 document is an evidence scaffold, not accepted implementation evidence |
| WASM plugin runtime | Requires plugin ADR, ABI, WASI sandbox, capability host calls, quotas, crash isolation, and malicious extension tests |
| Real-time collaboration | Requires operation log or CRDT ADR, convergence tests, storage retention, replay, and shared proposals |
| Remote development | Requires local proposal/conflict semantics, collaboration identity, enterprise policy, transport, edge agent, and remote conflict evidence |
| Vector indexing and embeddings | Explicitly deferred by ADR/evidence until later contracts and privacy/storage decisions exist |

## Roadmap

### Stage 0: Foundation Lock

Goal: remove ambiguity before MVP implementation continues.

Deliverables:

1. Create a concise phase status ledger under `plans/` mapping evidence to each phase.
2. Record Phase 0 and Phase 1 as accepted, Phase 2 as partially accepted, Phase 3 as not accepted, and Phases 4-8 as future-gated.
3. Reconcile ADR status ambiguity for `ADR-0002-ui-editor-rendering.md` and `ADR-0005-storage-backends.md`.
4. Reconcile older ADR identifiers in `ide-core-architecture-spec-v0.1.md` with newer ADRs `ADR-0016`, `ADR-0017`, and `ADR-0018`.
5. Decide whether hardcoded `xtask` dependency checks remain duplicated or become fully represented in `dependency-policy.md`.
6. Annotate stale historical save/UI claims as historical where they contradict current proposal-mediated saves and projection-only UI behavior.

Exit criteria:

1. `cargo run -p xtask -- check-deps` still passes.
2. Status ledger explicitly distinguishes accepted evidence from scaffolds.
3. ADR ambiguity is either resolved or recorded as a named blocker.
4. No runtime surface activation begins before unresolved governance items are closed.

Release result: Foundation lock. This is not yet MVP.

### Stage 1: MVP Safety Substrate

Goal: finish Phase 2B generalized proposal execution so all current and future mutation clients use a common safety model.

Deliverables:

| Package | Scope | Acceptance evidence |
| --- | --- | --- |
| Proposal lifecycle state service | Turn `AppProposalCoordinator` into a real app-domain lifecycle service with recoverable state and enforced transitions | Missing lifecycle context rejects; state transitions are covered by integration tests |
| Generic save proposal apply | Make generic `ProposalPayload::SaveFile` apply equivalent to manual save, or keep denied with documented migration rationale | Save equivalence test preserves existing preconditions and dirty text |
| Deny-by-default validation | Validate text edit, create, delete, rename, save, format, code action, workspace edit, terminal command, and batch payloads | Missing principal/capability/correlation/causality/trust/version/rollback rejects before mutation |
| Open-buffer apply | Route open-buffer proposal edits through editor transactions only | Stale snapshot rejection, rollback, audit, and dirty-buffer preservation tests pass |
| Closed-file apply | Route create/delete/rename/format/workspace edit through workspace VFS authority | Conflict, path-policy, trust, fingerprint, and dirty-open-buffer protection tests pass |
| Batch planner | Implement prepare, preflight, apply, commit, audit, rollback, finalize | All-or-nothing or exact partial-failure tests pass |
| Audit-before-success | Require metadata-only audit/event persistence before success | Storage-failure path fails closed without reporting mutation success |
| UI proposal projections | Add projection DTOs for proposal list, preview, affected targets, warnings, actions, and result | UI remains intent-only; no editor/workspace ownership regression |

Exit criteria:

1. `plans/evidence/phase-2/proposal-mutation-substrate.md` is updated to show remaining Phase 2 gaps are closed.
2. Manual save behavior is preserved from the caller perspective: rejected stale/conflict/denied outcomes preserve dirty editor text.
3. Unsupported terminal, plugin, remote, collaboration, command-like, and mixed routes reject before side effects.
4. Full global gates pass.
5. Focused checks pass:
   - `cargo test -p devil-protocol --test dto_contracts`
   - `cargo test -p devil-app --test workspace_vfs_integration workspace_vfs_integration_external_overwrite_between_open_and_save_yields_conflict`
   - `cargo test -p devil-app --test workspace_vfs_integration workspace_vfs_integration_untrusted_save_is_denied_without_disk_mutation`
   - `cargo test -p devil-project --test path_boundary`

Release result: MVP. This release is safe for internal dogfooding and focused local editing workflows. It should be labeled as not AI-active, not LSP-active, not plugin-active, not collaboration-active, and not remote-active.

### Stage 2: User-Facing Core Alpha

Goal: turn the safe core into a usable local IDE alpha without broad runtime expansion.

Deliverables:

1. Renderer-backed shell evidence for input-to-paint p50/p95, frame variance, CPU sampler output, GPU utilization, focus, clipboard, IME, and accessibility tree.
2. Projection-only file explorer, tabs, active editor projection, status bar, command palette, proposal preview, and session restore.
3. Explicit degraded-mode UX for large files: visible status, skipped overlays, bounded search behavior, and no hidden full-source projection.
4. Packaging and launch path appropriate for alpha distribution on the validated platform.
5. User-facing limitations document that names inactive features: AI execution, LSP runtime, plugins, collaboration, remote, vector indexing, and cloud provider paths.

Exit criteria:

1. UI tests prove the shell remains projection-only.
2. Renderer-backed interaction evidence exists, not just projection tests.
3. Large-file projection tests continue to prove no full-source UI/app projection above the full-cache budget.
4. Existing save/conflict and proposal regression tests remain green.
5. Platform support is labeled accurately. Current documentation validates Windows evidence first; macOS/Linux parity cannot be claimed without platform evidence.

Release result: Core alpha. This can be shown to users as a local deterministic editor/workspace alpha, not as the full AI IDE.

### Stage 4: AI-Native Beta

Goal: implement policy-bound AI execution only after generalized proposals, semantic fabric, and LSP supervision are accepted.

Deliverables:

1. Revise or accept ADRs for AI provider router, context manifests, Privacy Inspector, agent state machine, tracker run ledger, memory retention, and enterprise policy.
2. Expand provider contracts for streaming, structured output, embeddings where allowed, reranking, tool planning, context windows, cost metadata, cancellation, and provider health.
3. Implement local providers before cloud providers.
4. Gate cloud providers behind explicit allowlists, redaction, policy, air-gap tests, and provider egress evidence.
5. Implement `devil-agent` as a state machine: observing, planning, proposing, waiting for approval, applying, verifying, recovering, and blocked.
6. Add context manifests that cite editor snapshots, semantic symbols, retrieved chunks, tracker tasks, memory records, diagnostics, terminal summaries, and user selections.
7. Implement Privacy Inspector data from tracker and event metadata.
8. Persist AI run metadata, selected context, tool calls, approvals, proposal IDs, verification outputs, and redacted provider metadata.
9. Route all generated edits through generalized proposals and all command execution through capability policy.

Exit criteria:

1. AI cannot mutate editor buffers, disk, terminal, tracker, memory, settings, storage, or network paths directly.
2. Every model call has redacted event metadata and a context manifest.
3. Users can inspect what context was sent, why it was selected, and what policy allowed it.
4. Air-gap mode prevents cloud providers, hosted telemetry, update checks, hosted embeddings, remote gateways, and non-approved outbound access.
5. Agent runs can be cancelled, resumed, and replayed from metadata.
6. Cloud activation requires security review, ADR/policy updates, contract tests, and evidence.

Release result: AI beta. This is the first release that can be marketed as AI-native without contradicting the architecture documents.

### Stage 5: Production Release Candidate

Goal: convert active features into production-ready software through storage, diagnostics, privacy, replay, policy, migration, and CI hardening.

Deliverables:

1. Move active production surfaces from in-memory repositories to durable, migrated stores. Scope includes sessions, trust, file metadata, proposal audit, event metadata, tracker, AI run records, semantic metadata, and any other active feature state.
2. Expand `devil-cli` diagnostics for dependency graph, protocol schema, event summaries, proposal audit, storage health, index health, AI run manifests, and release support bundles.
3. Add privacy-preserving metrics with budgets for edit, render, index, LSP, AI, proposal, storage, and UI operations.
4. Add enterprise policy profiles for air-gap, local-only AI, approved providers, plugin allowlists where applicable, remote restrictions where applicable, collaboration retention where applicable, and audit export.
5. Add recovery drills: corruption recovery, replay, downgrade/migration, provider egress, privacy retention, index rebuild, LSP restart storm, proposal rollback, and save conflict.
6. Add CI coverage for all global gates plus `cargo deny check`.
7. Normalize Rust toolchain resolution in CI to avoid the Phase 2 clippy toolchain mismatch recurring.
8. Produce release documentation that clearly labels active, beta, disabled, and future-gated features.

Exit criteria:

1. Metadata-only replay reconstructs critical user flows without retaining raw source by default.
2. Storage migrations are reversible or recoverable.
3. Privacy and retention defaults are enforced by tests.
4. Every active subsystem has event coverage and operational diagnostics.
5. Enterprise policy profiles validate automatically in CI.
6. Cross-platform support claims match evidence. If only Windows evidence exists, production support must be Windows-only or explicitly experimental elsewhere.
7. Full global gates and `cargo deny check` pass in CI.

Release result: Production RC.

### Stage 6: Production 1.0

Goal: ship the smallest production-grade Devil IDE that satisfies the product thesis without claiming incomplete platform features.

Recommended Production 1.0 scope:

1. Local-first editor/workspace core.
2. Projection-only native UI with measured renderer/accessibility/IME/clipboard/focus evidence for advertised platforms.
3. Scalable viewport/chunk text substrate and explicit degraded large-file UX.
4. Proposal-mediated save, open-buffer, closed-file, and batch mutation safety.
5. Supervised semantic and LSP substrate, if Stage 3 is accepted.
6. Policy-bound AI execution, if Stage 4 is accepted.
7. Privacy Inspector, context manifests, metadata-only audit, and air-gap enforcement.
8. Durable active-feature storage, replay, diagnostics, and CI gates.

Production 1.0 must not claim:

1. Production plugin marketplace unless Stage 7 plugin expansion is complete.
2. Real-time collaboration unless Stage 7 collaboration expansion is complete.
3. Remote development unless Stage 7 remote expansion is complete.
4. Vector semantic retrieval unless a later vector ADR, storage/privacy model, dependency policy, and evidence package are accepted.
5. Cross-platform parity without corresponding platform evidence.

Exit criteria:

1. All active-feature gates from Stages 0-5 pass.
2. Known degraded modes are documented and surfaced in-product.
3. Unsupported future routes deny before side effects.
4. Security/privacy review passes for local, AI, provider, storage, observability, and policy behavior.
5. Release support flow can diagnose dependency, storage, event, proposal, index, LSP, AI, and privacy issues without exposing raw source by default.

Release result: Production 1.0 local-first AI IDE.

### Stage 7: Production Platform Expansions

These stages are not required for Production 1.0 unless the release promise includes them.

#### WASM Plugin Runtime

Gate requirements:

1. Proposal execution complete.
2. Plugin ADR and dependency-policy activation accepted.
3. Durable plugin state and sandbox audit contracts accepted.
4. WASM ABI, WASI sandbox, host-call schemas, manifests, compatibility ranges, quotas, cancellation, crash isolation, and redacted plugin events implemented.

Exit criteria:

1. Malicious extension tests prove no ambient source, network, process, storage, editor, workspace, AI, or memory access.
2. Extension crashes do not crash the IDE.
3. Extension mutations are proposal-mediated.
4. ABI compatibility is covered by golden manifests and host-call schema tests.

#### Real-Time Collaboration

Gate requirements:

1. Scalable text substrate accepted.
2. Generalized proposals accepted.
3. Deterministic operation replay and storage retention decisions accepted.
4. Collaboration ADR chooses CRDT versus operation log, identity, permissions, conflict policy, offline behavior, undo semantics, and retention.

Exit criteria:

1. Concurrent edits converge deterministically.
2. Undo semantics are explicit for local and collaborative operations.
3. Dirty buffers are never silently overwritten by remote operations.
4. Shared proposals record approvers, denials, policy decisions, and applied operation IDs.
5. Collaboration does not block local keystrokes or viewport rendering.

#### Edge Remote Development

Gate requirements:

1. Local proposal and conflict semantics generalized and tested.
2. Collaboration identity and operation model accepted.
3. Enterprise policy and transport ADRs accepted.
4. Remote edge workspace agent, projection model, transport, auth, secrets, storage, process isolation, and egress policy are accepted.

Exit criteria:

1. Local UI remains responsive during latency, reconnect, and degraded network events.
2. Remote writes cannot bypass proposal and fingerprint preconditions.
3. Remote terminals and LSP servers cannot launch in untrusted workspaces or outside policy.
4. Remote events correlate with local events using non-zero causality and correlation IDs.
5. Offline resume reconciles operation logs without silent data loss.

## Dependency and Parallelization Rules

| Work | May parallelize with | Must not parallelize with |
| --- | --- | --- |
| Stage 0 foundation lock | Test/evidence cleanup | Runtime activation relying on unresolved ADRs |
| Stage 1 proposal execution | UI proposal projection design | LSP apply, AI edits, plugin writes, collaboration writes, remote writes |
| Stage 2 core alpha UX | Non-mutating diagnostics and renderer measurement harnesses | Moving editor/workspace ownership into UI |
| Stage 3 semantic/LSP | Phase 3 evidence artifact authoring | AI control-plane activation before Phase 3 acceptance |
| Stage 4 AI beta | Privacy Inspector UI design and tracker schema work | Plugin/collaboration/remote mutation shortcuts |
| Stage 5 RC hardening | Non-invasive diagnostics across active subsystems | Treating hardening as a substitute for missing phase evidence |
| Stage 7 plugins | Plugin diagnostics scaffolding | Untrusted extension execution without sandbox quotas |
| Stage 7 collaboration | Presence projection prototypes | Remote writes or collaborative AI approvals before convergence tests |
| Stage 7 remote | Enterprise policy profile design | Remote mutation before local and collaborative conflict semantics are accepted |

Fail-stop conditions:

1. Any generalized proposal path mutates editor or workspace state without preconditions, rollback metadata, policy, and audit.
2. Production `devil-index` scans disk, mints workspace file identity, persists source, or depends directly on workspace/editor/app/UI crates.
3. LSP or semantic workers write buffers, write files, execute commands, block saves, block input, or report fresh query status across stale privacy/version boundaries.
4. AI, plugin, collaboration, remote, or terminal code bypasses the proposal service.
5. Observability, storage, tracker, memory, semantic caches, or evidence artifacts persist raw source by default.
6. Any placeholder runtime surface activates without accepted ADR, dependency-policy entry, protocol contracts, tests, ownership gates, and evidence.

## Final Recommendation

Treat the MVP as a trust and mutation-safety milestone, not as a feature-complete IDE. Ship internal dogfooding only after generalized proposal execution closes Phase 2B. Move to external alpha only after renderer/native UX evidence exists. Move to beta only after Phase 3 semantic/LSP evidence is complete. Keep plugins, collaboration, remote development, vectors, and cloud expansion out of production claims until their own gates are independently accepted.

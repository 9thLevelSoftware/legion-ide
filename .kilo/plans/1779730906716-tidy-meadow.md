# Phase 5 Implementation Plan: WASM Isolated Extension Ecosystem

Prepared: 2026-05-25
Plan status: Ready for implementation review
Scope authority: `plans/implementation-plan.md`, `plans/remaining-implementation-tasks-plan-v0.1.md`, `plans/phase-status-ledger.md`, `plans/dependency-policy.md`

## Scope

This plan implements Phase 5 from the current phase ledger and product implementation plan: a WASM isolated extension ecosystem. Phase 0 through Phase 4 are accepted. Phase 5 remains future-gated until this work creates the ADR, dependency-policy activation, protocol contracts, runtime implementation, ownership tests, and evidence.

This plan intentionally targets the Phase 5 WASM plugin scope, not the older foundational-core Phase 5 multi-tab/session-restore plan. The current ledger classifies Phase 5 as plugins, with collaboration, remote development, and hardening deferred to later phases.

## Non-goals

- Do not add VS Code extension compatibility, a marketplace, Node-based extension execution, or arbitrary host scripting.
- Do not grant plugins ambient filesystem, process, network, editor, workspace, storage, AI, tracker, memory, terminal, collaboration, or remote-development access.
- Do not allow plugin edits, file operations, formatter outputs, scanner writes, or generated actions to bypass generalized proposals.
- Do not persist raw source, full plugin payloads, secrets, prompts, provider responses, or unbounded panel output by default.
- Do not activate collaboration, remote development, terminal execution, cloud providers, hosted telemetry, hosted embeddings, vector retrieval, or unrestricted process/network access as part of Phase 5.

## Current Baseline

- No `crates/devil-plugin` crate exists in the workspace.
- `plans/dependency-policy.md` lists `devil-plugin` only as a planned runtime surface. Its allowed internal dependencies are `devil-observability`, `devil-platform`, `devil-protocol`, `devil-security`, and `devil-storage`.
- `crates/devil-protocol/src/lib.rs` has thin plugin placeholder DTOs: `PluginId`, `PluginManifest`, `PluginActivationEvent`, `PluginCommandDescriptor`, `ContributionDescriptor`, `PluginStateNamespace`, `PluginActionProposal`, `ContextProviderDescriptor`, `PluginRequest`, and `PluginResponse`.
- `crates/devil-security/src/lib.rs` has a placeholder `PluginCapabilityPolicy`; it is not manifest-aware, WASI-aware, quota-aware, or host-call-aware.
- `crates/devil-app/src/lib.rs` currently denies plugin targets and plugin-sourced workspace edits with fail-closed diagnostics such as `proposal.plugin_source_denied`.
- `crates/devil-storage/src/lib.rs` has no plugin state namespace or quota persistence.
- `crates/devil-observability/src/lib.rs` has reusable metadata-only event validation but no plugin-specific activation, host-call, crash, denial, or quota event helpers.
- `crates/devil-ui/src/ui.rs` remains projection-only and has no plugin contribution projection model yet.
- `xtask/src/main.rs` validates Phase 3 and Phase 4 evidence, but has no Phase 5 acceptance governance yet.

## Implementation Strategy

Implement Phase 5 as a deny-by-default runtime surface. The first accepted runtime slice should support deterministic loading of signed or explicitly trusted WASM plugins, declarative contributions, capability-checked host calls, proposal-only mutations, scoped read-only context access, per-plugin metadata storage, quotas, cancellation, crash isolation, and redacted observability.

The dependency direction should be:

- `devil-protocol` owns cross-domain plugin DTOs, ABI records, host-call envelopes, contribution projections, storage request/response variants, validation helpers, and `PluginPort`.
- `devil-security` owns manifest trust, declared capability validation, host-call capability decisions, air-gap/network denial, sandbox policy, and per-plugin budget decisions.
- `devil-plugin` owns manifest parsing, ABI compatibility checks, WASM/WASI instantiation, lifecycle state, host-call dispatch envelopes, fuel/epoch cancellation, memory limits, crash isolation, and contribution registration.
- `devil-storage` owns plugin-scoped metadata namespaces, quota accounting, migration/corruption behavior, and cross-plugin isolation.
- `devil-observability` owns metadata-only plugin event helpers and redaction validation.
- `devil-app` is the composition root. It activates trusted plugin manifests, routes plugin commands, mediates capability decisions, stores metadata, emits events, and registers plugin-produced proposals with existing proposal orchestration.
- `devil-ui` remains projection-only. It renders declarative plugin contributions and emits command intents; it never executes plugin code or owns plugin state.

## Work Package 2: Security, Manifest Trust, And Capability Policy

Files likely touched:

- `crates/devil-security/src/lib.rs`
- `crates/devil-security/tests/*`
- `crates/devil-protocol/src/lib.rs`
- `crates/devil-protocol/tests/dto_contracts.rs`

Steps:

1. Replace placeholder plugin policy with manifest-aware, deny-by-default policy.
2. Validate declared capabilities against manifest contributions and host-call requests.
3. Require trusted workspace for plugin activation unless the ADR records a narrower read-only exception.
4. Deny plugin filesystem, process, network, terminal, AI, memory, tracker, raw storage, collaboration, and remote capabilities by default.
5. Keep air-gap behavior strict: no outbound network for plugins, no hosted telemetry, no hosted embeddings, and no gateway calls.
6. Add explicit namespace isolation: a plugin can request only its own namespace, and cross-plugin namespace reads/writes are denied.
7. Add quota/budget policy for CPU/fuel, wall-clock timeout, memory pages, storage bytes, host-call count, event count, and output byte count.
8. Add signature/trust metadata decisions: trusted first-party, explicitly allowed local, revoked, checksum mismatch, unknown signer, and incompatible ABI.
9. Add security tests for malicious manifests, untrusted workspaces, undeclared capabilities, namespace escape, network/process/filesystem denial, quota denial, air-gap denial, and deny-by-default unknown capabilities.

Exit criteria:

- Capability decisions require manifest context, plugin identity, host-call context, trust state, and correlation metadata.
- Malicious or malformed plugin requests fail closed before runtime execution or host-call dispatch.
- Unknown plugin capabilities remain denied by default.

## Work Package 3: Create `devil-plugin` Runtime Crate

Files likely touched:

- `Cargo.toml`
- `crates/devil-plugin/Cargo.toml`
- `crates/devil-plugin/src/lib.rs`
- `crates/devil-plugin/tests/*`
- `plans/dependency-policy.md`

Steps:

1. Add `crates/devil-plugin` only after Work Package 0 policy and ADR updates are in place.
2. Use a sandbox runtime that supports WASM/WASI with no ambient preopened directories, no inherited environment unless explicitly whitelisted, no network, no process launch, no stdio authority by default, fuel or epoch interruption, memory limits, and deterministic teardown.
3. Prefer a Wasmtime/WASI implementation if the ADR and dependency review accept its supply-chain footprint; record fallback decisions if cargo-deny or platform constraints block it.
4. Implement manifest parsing and validation before module instantiation.
5. Implement lifecycle states: discovered, rejected, loaded, activated, running, idle, cancelling, cancelled, crashed, disabled, and unloaded.
6. Implement deterministic activation events for startup, command, file open metadata, language id, workspace scanner trigger, and passive context provider trigger.
7. Implement host-call dispatch as data-only protocol envelopes. The runtime never receives references to app, editor, workspace, storage, AI, or UI internals.
8. Enforce fuel/epoch cancellation, wall-clock timeout, memory limits, call-count limits, bounded output, and crash isolation around each invocation.
9. Convert runtime failures into typed metadata responses and events without panicking the IDE.
10. Add tests using minimal WASM fixtures for successful load, incompatible ABI, denied WASI ambient access, denied host call, cancellation, fuel exhaustion, memory quota, trap/crash isolation, and deterministic unload.

Exit criteria:

- A plugin can be instantiated and invoked without ambient filesystem, process, network, or host object access.
- Plugin traps, infinite loops, quota exhaustion, or invalid host calls do not crash the IDE.
- Runtime APIs expose protocol DTOs only.

## Risks And Mitigations

| Risk | Mitigation |
| --- | --- |
| WASI runtime accidentally receives ambient host authority | Construct runtime with no preopens, no inherited env, no network/process, bounded stdio, and adversarial sandbox tests. |
| Plugin writes bypass proposals | Allow write-like host calls to create proposals only; app/editor/workspace authorities remain the only apply path. |
| Dependency-policy drift | Add `xtask` Phase 5 governance and keep `devil-plugin` isolated from app/UI/editor/project internals. |
| Wasmtime/WASI dependency footprint is large | Make dependency approval explicit in ADR, run cargo-deny, and document fallback if supply-chain policy blocks activation. |
| Raw source or secrets leak through storage/events | Reuse metadata-only validation and add plugin-specific forbidden marker tests. |
| Quotas fail under malicious or stuck WASM | Enforce fuel/epoch, timeout, memory, output, host-call count, and storage quotas with fail-closed tests. |
| UI becomes plugin host by convenience | Keep UI projection-only and test that plugin command handling remains app-owned. |
| Plugin activation weakens air-gap guarantees | Deny plugin network and hosted telemetry by default; test air-gap denial explicitly. |

## Phase 5 Exit Criteria

Phase 5 is complete only when all of the following are true:

- Plugin ADR is accepted and dependency policy activates `devil-plugin` with explicit allowed dependencies.
- `xtask check-deps` enforces Phase 5 evidence and protocol symbol requirements.
- `devil-plugin` exists and runs WASM/WASI plugins without ambient filesystem, process, network, editor, workspace, storage, AI, memory, terminal, collaboration, or remote authority.
- Plugin manifests, ABI compatibility, contributions, capabilities, trust metadata, quotas, and storage namespaces are versioned and contract-tested.
- Host calls are capability-checked, cancellable where applicable, quota-bound, observable, and metadata-only by default.
- Plugin storage is namespace-isolated and quota-bound.
- Plugin crashes, traps, infinite loops, and quota failures do not crash the IDE or apply pending mutations.
- Plugin mutations are generalized proposals and apply only through existing editor/workspace authorities.
- UI remains projection-only and renders plugin contributions without executing plugin code.
- Malicious extension tests prove no ambient source, network, process, storage, editor, workspace, AI, memory, terminal, collaboration, or remote access.
- Future surfaces remain deferred unless separately gated.
- Required global gates pass and evidence artifacts are captured under `plans/evidence/phase-5/`.

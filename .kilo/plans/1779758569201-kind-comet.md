# Phase 8 Production GA Completion Plan

Status: planning draft  
Created: 2026-05-25  
Scope: develop and implement the work required to move Phase 8 from deterministic metadata-only fixture scaffold to production GA acceptance.

Implementation update 2026-05-25: the safe governance slice is implemented. `xtask`, `devil-cli evidence check --phase phase8`, and the Phase 8 architecture map now share the full GA artifact list below; stale CLI doctor/setup diagnostics were reconciled with the accepted Phase 4 and future-gated Phase 8 status. Phase 8 remains not accepted for production GA.

## 1. Current Baseline

Phase 8 is currently not accepted for production GA. The accepted repository state contains deterministic, metadata-only fixture slices for `devil-remote-transport`, `devil-terminal`, `devil-telemetry`, and `devil-retention`, plus governance checks in `xtask` and `plans/evidence/phase-8/phase-8-architecture-map.md`.

Current fixture capabilities:

| Area | Current state | Production gap |
| --- | --- | --- |
| Remote transport | `RemoteTransportFixture` validates metadata-only handshake/frame descriptors. | No real network I/O, TLS/mTLS, agent packaging, endpoint policy runtime, resume/backpressure, or platform evidence. |
| Terminal | `TerminalFixtureRuntime` validates launch metadata and bounded redacted fixture output. | No native PTY/process execution, input/resize lifecycle, kill cleanup, platform backend, or platform evidence. |
| Telemetry | `TelemetryFixtureSpool` stores in-memory metadata records and fake acknowledgements. | No hosted export, durable spool, retry/drop behavior, endpoint/proxy/cert policy, operational dashboard, or consent operations. |
| Raw-source retention | `RetentionFixtureVault` stores descriptor-only metadata and tombstones. | No encrypted raw-source content vault, key management, durable lifecycle, scoped raw capture, restore, or deletion guarantees. |
| Storage/migrations | `FileBackedStorage` has JSON schema versioning, serde-default migration to schema 2, and corrupt-file quarantine. | No migration registry, dry-run/rollback strategy, schema manifests, backup/restore drills, or production storage operational evidence. |
| Operations/release | Phase 8 evidence map, `xtask`, and `devil-cli` list the required GA artifact set while preserving not-accepted status. | No accepted threat model, fault drills, platform matrix, release gates, rollout/rollback playbooks, canary evidence, or GA signoff. |

Non-negotiable boundaries:

| Boundary | Requirement |
| --- | --- |
| UI projection-only | UI may render projections and send intents; it cannot own sessions, PTYs, transports, storage, editor text, or mutation authority. |
| Proposal-mediated mutation | Any file/editor mutation from remote, terminal, telemetry import/export, retention, plugin, AI, or diagnostics must route through existing proposal/editor/workspace authorities. |
| Deny-by-default | Production transport, PTY, hosted telemetry, and raw-source retention remain disabled until policy, capability, consent, and evidence gates explicitly enable them. |
| Air-gap authority | Air-gap mode denies non-loopback production transport, hosted telemetry, hosted raw-source upload, update checks, and hosted providers. |
| Metadata-only default | Normal observability, telemetry, storage, remote, terminal, plugin, AI, and collaboration records must reject raw source, transcripts, process output, transport payloads, prompts, provider payloads, and secrets. |
| Runtime activation gates | Runtime activation must include accepted ADR updates, dependency-policy updates, protocol contracts, contract tests, ownership tests, owners, and evidence. |

## 2. Program Shape

Recommended duration: 20 weeks, organized as 10 two-week gates. This is a production hardening program, not a small follow-up, because GA requires real transport, native platform behavior, hosted operations, encrypted raw-source storage, migrations, drills, and release evidence.

Required workstreams:

| Workstream | Primary owner | Output |
| --- | --- | --- |
| A. Production remote transport | Distributed systems and security | Secure, typed-envelope-only production transport with endpoint policy, identity binding, resume, flow control, and evidence. |
| B. Native terminal runtime | Platform and terminal runtime | App-composed local terminal/PTY runtime with policy, lifecycle, cleanup, and platform matrix evidence. |
| C. Hosted telemetry operations | Observability, privacy, backend | Consent-aware hosted metadata export with durable spool, classifier, egress controls, retries, dashboards, and evidence. |
| D. Encrypted raw-source vault | Storage, security, privacy | Default-deny encrypted vault for scoped support/debug bundles with access audit, TTL, deletion, restore, and isolation. |
| E. Operational GA hardening | Release, QA, storage, security | Migrations, diagnostics, fault drills, enterprise policy CI, performance budgets, release playbooks, and GA evidence. |

## 4. Workstream A: Production Remote Transport

Objective: convert the fixture-only `devil-remote-transport` crate into a production carrier for `RemoteTransportEnvelope` traffic without granting direct local mutation authority.

Implementation plan:

1. Update ADR-0025 with production design decisions: transport protocol, TLS/mTLS or equivalent, peer identity, endpoint policy, schema negotiation, replay window, reconnect/resume, flow control, and remote agent packaging.
2. Update `plans/dependency-policy.md` and `xtask` for production transport activation, allowed dependencies, forbidden edges, and required protocol symbols.
3. Extend `devil-protocol` with production DTOs for certificate/key references, endpoint policy, handshake result, frame acknowledgement, flow-control window, resume token, heartbeat, degraded state, and remote agent compatibility.
4. Add contract tests before runtime code for invalid schema, zero identity values, downgrade, missing principal, expired credential, wrong workspace, oversized frames, duplicate frames, replay, and causal gaps.
5. Implement a transport abstraction in `devil-remote-transport` with deterministic in-memory conformance tests and real network implementations behind explicit feature flags.
6. Bind connection authorization to `devil-security`: workspace trust, principal, `remote.session.connect`, endpoint allowlist, air-gap/local-only policy, certificate policy, and remote authority/agent identity.
7. Keep all raw sockets and network dependencies out of app/UI. `devil-app` composes the transport through ports; `devil-ui` renders projections only.
8. Dispatch successful frames only as typed `RemoteTransportEnvelope` values into the existing `devil-remote` runtime. File mutations still require proposal IDs and all write preconditions.
9. Add metadata-only observability/storage for handshake, denial, health, latency, reconnect, resume, backpressure, and audit summaries. Reject raw transport payload bodies.
10. Implement remote agent packaging evidence: version compatibility, integrity hash, startup health, shutdown, upgrade, rollback, and fail-closed mismatch behavior.
11. Add fault tests for network loss, frame split/reorder/duplicate/drop, tamper, downgrade, backpressure, cancellation, reconnect, invalid resume, and clock/sequence gaps.
12. Produce Windows 11 and at least one Unix-like platform evidence before claiming production network parity.

Acceptance criteria:

| Area | Criteria |
| --- | --- |
| Security | Unknown authority, wrong principal, expired credential, downgrade, schema mismatch, non-allowlisted endpoint, and air-gap egress fail closed. |
| Authority | No transport path mutates editor/workspace/disk directly; remote writes still route through proposals and write preconditions. |
| Reliability | Reconnect/resume, duplicate detection, causal gaps, cancellation, shutdown, and degraded states are deterministic and tested. |
| Privacy | No raw transport payload or secrets persist in observability, telemetry, or storage. |
| Evidence | `remote-production-transport-security-tests.txt`, remote agent packaging evidence, threat model, platform matrix, and full gates are archived. |

## 5. Workstream B: Native Terminal Runtime

Objective: activate a local terminal/PTY runtime through `TerminalPort` while preserving app-owned composition, policy gates, projection-only UI, and proposal-mediated mutation.

Implementation plan:

1. Update ADR-0026 with production terminal lifecycle, approval model, shell policy, cwd/env policy, backend matrix, output retention defaults, and unsupported-platform behavior.
2. Update dependency policy and `xtask` so `devil-terminal` may use `devil-platform`, `devil-security`, `devil-observability`, and metadata-only storage while preserving forbidden app/UI/editor/project dependencies.
3. Extend terminal protocol contracts for launch, input, resize, close, kill, timeout, output chunk, truncation, exit, backend error, policy denial, and health projection.
4. Build a session registry/state machine in `devil-terminal`: starting, running, degraded, exiting, exited, failed, denied, killed, and timed out.
5. Enforce `devil-security` decisions per action: trusted workspace, principal, terminal capability, cwd containment, shell allowlist, environment filtering/redaction, timeout, input/output limits, and rate limits.
6. Implement `devil-platform::PtyService` production backend for Windows ConPTY first, then Unix PTY. Unsupported backends return `PtyUnavailable` with diagnostics and fail closed.
7. Stream output as bounded, sequenced, redacted chunks for projection. Persist metadata summaries only unless a scoped raw-source/terminal retention grant exists.
8. Route terminal-originated mutation candidates into proposal flows. Terminal runtime must never write editor buffers or workspace files directly.
9. Integrate with `devil-app` composition and UI projections without granting UI process authority.
10. Add cleanup and resource controls: kill tree, orphan detection, timeout shutdown, resize behavior, input after exit denial, crash cleanup, and app shutdown cleanup.
11. Add read-only CLI diagnostics for sessions, backend availability, denials, truncation, orphan cleanup, limits, and event summaries.
12. Produce native PTY platform evidence for each claimed GA platform.

Acceptance criteria:

| Area | Criteria |
| --- | --- |
| Security | Launch/input/resize/close deny in untrusted workspaces, without capability, outside cwd policy, on blocked shell/env, after exit, or beyond limits. |
| Lifecycle | Launch, input, resize, close, kill, exit, timeout, truncation, backend error, and cleanup are deterministic and tested. |
| Authority | Terminal cannot mutate workspace/editor/disk directly; mutation candidates become proposals. |
| Privacy | Raw command bodies, transcripts, process output, secrets, and environment values are not persisted by default. |
| Evidence | `terminal-runtime-policy-tests.txt`, `terminal-pty-platform-tests.txt`, cleanup evidence, platform matrix, and full gates are archived. |

## 6. Workstream C: Hosted Telemetry Operations

Objective: deliver opt-in hosted telemetry for metadata-only operational diagnostics without violating air-gap or privacy defaults.

Implementation plan:

1. Update ADR-0027 with telemetry categories, consent hierarchy, endpoint identity, proxy/region/certificate policy, privacy budgets, spool TTL, retry/drop semantics, upload acknowledgement, deletion hooks, and operational SLOs.
2. Update dependency policy and `xtask` for production telemetry activation and required protocol symbols.
3. Extend telemetry DTOs for consent grant/revocation, category, endpoint, tenant/project identity, export batch, spool record, privacy classification, upload outcome, retry-after, deletion request, purge summary, and exporter health.
4. Implement structured privacy classification before spooling. Use schema allowlists plus adversarial tests for raw source, transcripts, process output, transport payloads, prompts, provider payloads, paths, tokens, and secrets.
5. Implement durable, bounded, consent-tagged telemetry spool with TTL, crash recovery, encryption for sensitive metadata where needed, purge-on-revoke, backpressure, drop summaries, and migration support.
6. Implement hosted exporter with typed batches, endpoint allowlist, auth token references, certificate/proxy/region policy, acknowledgement handling, retry-after, exponential backoff, and failure classification.
7. Enforce `devil-security` egress policy for `telemetry.export.hosted`: trusted workspace, explicit consent, category policy, endpoint allowlist, and unconditional air-gap denial.
8. Integrate metadata sources from active subsystems: proposals, editor/app, storage, security, plugin, collaboration, remote, terminal, telemetry, retention, and CLI diagnostics. Keep every event bounded and schema-versioned.
9. Add app/CLI projections for telemetry status, consent, queue length, drops, denials, last upload, endpoint health, and purge state. UI cannot export directly.
10. Add operational tests for offline queue, network outage, endpoint 4xx/5xx, retry-after, proxy failure, certificate mismatch, consent revocation race, crash recovery, spool full, and exporter backpressure.

Acceptance criteria:

| Area | Criteria |
| --- | --- |
| Consent | Export is denied without active scoped consent; revocation purges or tombstones queued records within budget. |
| Air-gap | Hosted telemetry export never succeeds under air-gap or non-allowlisted endpoint policy. |
| Privacy | Hosted batches contain only classified, bounded metadata; raw source/transcripts/process output/transport payloads/prompts/provider payloads/secrets are rejected. |
| Reliability | Exporter failures never block editor input, saves, terminal lifecycle, remote transport dispatch, or proposals. |
| Evidence | `hosted-telemetry-consent-policy-tests.txt`, failure-mode tests, privacy classifier audit, diagnostics evidence, and full gates are archived. |

## 7. Workstream D: Encrypted Raw-Source Vault

Objective: implement a controlled exception to metadata-only defaults for scoped support/debug bundles, with encryption, explicit consent, access audit, TTL, deletion, and isolation from telemetry/observability.

Implementation plan:

1. Update ADR-0028 with accepted purposes, local-only versus hosted bundle posture, consent model, scope limits, TTL defaults, encryption/key management, access model, deletion/tombstone semantics, backup/restore, and legal-hold stance.
2. Decide whether production content vault APIs live in `devil-retention` or `devil-storage`; keep raw content isolated from generic storage and observability records.
3. Update dependency policy and `xtask` for production retention activation, protocol symbols, and forbidden edges.
4. Extend protocol contracts for retention policy, purpose, consent grant, capture request, lease, bundle descriptor, encryption metadata, key reference, access audit, revocation, deletion, tombstone, backup/restore marker, and hosted export linkage.
5. Implement security policy: default denial, trusted workspace, principal, purpose, path scope, TTL, max bytes, enterprise override, break-glass flow, capability checks, and air-gap hosted-upload denial.
6. Implement encrypted local vault: chunked bounded capture, content encryption, integrity hash, metadata index, key reference, TTL scanning, access audit, tombstones, compaction, backup/restore, migration, and corruption recovery.
7. Capture raw source only through app/editor/workspace-authorized snapshots or proposal/debug-bundle flows. Respect large-file budget, path policy, vendor/binary exclusions, and explicit extra consent above thresholds.
8. Ensure normal telemetry, observability, remote, terminal, plugin, AI, collaboration, and storage records reference bundle descriptors only and still reject raw-source markers by default.
9. Add optional hosted debug bundle upload only if ADR-0028 accepts it. It requires separate raw-source consent, endpoint policy, encryption decision, access audit, and air-gap denial.
10. Add consent projections and read-only CLI diagnostics for grants, bundles, TTL, access logs, deletion, revocation, restore, corruption, and export status.
11. Add lifecycle tests for capture, read, export, revoke, delete, tombstone, TTL expiry, backup/restore, migration, key loss, corrupted bundle, and hosted upload denial.

Acceptance criteria:

| Area | Criteria |
| --- | --- |
| Default denial | Raw-source capture/read/export is denied without scoped consent and capability. |
| Security | Retained bundles are encrypted, integrity-checked, TTL-bound, access-controlled, and access-audited. |
| Isolation | Normal metadata stores and hosted telemetry never inline raw source; only descriptors may flow outside the vault. |
| Lifecycle | Delete, revoke, TTL, tombstone, migration, backup/restore, key loss, and corruption recovery are tested. |
| Evidence | `raw-source-retention-policy-tests.txt`, lifecycle tests, privacy audit, migration/recovery evidence, and full gates are archived. |

## 8. Workstream E: Operational GA Hardening

Objective: make all active Phase 8 capabilities diagnosable, recoverable, measurable, policy-governed, and release-ready.

Implementation plan:

1. Update ADR-0029 with storage migration policy, diagnostics coverage, enterprise profile requirements, SLOs, performance budgets, release stages, kill switches, evidence capture, rollback, incident runbooks, and GA stop conditions.
2. Build a migration framework with schema registry, manifests, dry-run validation, backups, checksums, forward migration, recover/roll-forward strategy, corruption quarantine, tombstone compaction, and compatibility tests.
3. Apply migrations to active stores: event metadata, proposal audit, remote transport metadata, terminal metadata, telemetry spool, raw-source vault metadata/content index, policy profiles, and diagnostics indexes.
4. Expand read-only diagnostics in `devil-cli`: phase gates, activation gates, dependency policy, protocol schema, storage health, event summaries, proposal audit, remote transport, terminal sessions, telemetry spool/exporter, retention vault, policy profiles, and evidence generation.
5. Add metadata-only replay drills for save/proposal flow, remote transport reconnect, terminal lifecycle, telemetry outage/purge, retention capture/delete, and migration recovery. Replay must reconstruct critical flows without raw source by default.
6. Add enterprise policy profile CI: air-gap, local-only AI, telemetry opt-in, telemetry denied, remote allowed, remote denied, terminal restricted, plugin allowlist, collaboration retention, raw-source denied, raw-source break-glass, audit export.
7. Add subsystem health events and projections for app, proposal, workspace, editor, storage, observability, security, plugin, collaboration, remote, remote transport, terminal, telemetry, retention, and CLI diagnostics.
8. Define and enforce performance budgets for editor input, save/proposal path, remote transport dispatch, terminal output projection, telemetry spool/exporter, raw-source capture, storage migration, and diagnostics.
9. Build fault drills for network loss, packet reorder/drop, endpoint outage, PTY hang, orphan process, output flood, disk full, permission denied, interrupted migration, corrupt storage, key loss, telemetry spool full, vault deletion failure, and rollback.
10. Create release operations package: feature flags, kill switches, alpha/beta/RC gates, canary plan, rollback playbooks, incident response, cargo-deny baseline review, support/debug bundle posture, release notes, and residual-risk signoff.

Acceptance criteria:

| Area | Criteria |
| --- | --- |
| Migrations | Migration, dry-run, interruption, corruption, backup/restore, downgrade/recover, tombstone, and metadata replay drills pass. |
| Diagnostics | Every active subsystem has read-only CLI/app health diagnostics and metadata-only event coverage. |
| Policy | Enterprise policy profiles run in CI and match actual runtime broker decisions. |
| Fault drills | All required fault scenarios have deterministic tests or documented platform drill evidence. |
| Release | Alpha, beta, RC, canary, rollback, incident, cargo-deny, and residual-risk evidence are complete. |

## 11. GA Definition Of Done

| Area | Done condition |
| --- | --- |
| Governance | ADR-0025 through ADR-0029 are accepted for production scope; owners are recorded; phase ledger and architecture map are updated only after evidence passes. |
| Dependency boundary | `plans/dependency-policy.md` and `xtask` align with active production crates, protocol symbols, and forbidden edges. |
| Remote transport | Real production transport is secure, typed-envelope-only, reconnectable, flow-controlled, policy-gated, metadata-audited, and unable to bypass proposals. |
| Terminal runtime | Native PTY runtime is app-composed, capability-gated, bounded, redacted, cleanup-safe, platform-evidenced, and unable to mutate workspace/editor directly. |
| Hosted telemetry | Hosted export is opt-in, air-gap denied, endpoint-policy-gated, metadata-only, bounded, classified, resilient, diagnosable, and non-blocking. |
| Raw-source vault | Raw retention is default-deny, explicit-consent, purpose-bound, encrypted, TTL-bound, access-audited, deletable, recoverable, and isolated from normal metadata paths. |
| Migrations | Durable stores have schema manifests, tested migrations, backups, recovery, corruption handling, and metadata-only replay evidence. |
| Fault drills | Required production failure scenarios are tested with deterministic results and documented evidence. |
| Release evidence | Platform matrix, performance budgets, cargo-deny review, diagnostics, rollout, rollback, incident runbooks, canary results, and residual-risk signoff are complete. |

## 12. Stop Conditions

Do not mark Phase 8 accepted if any stop condition occurs:

| Stop condition | Required action |
| --- | --- |
| Any direct mutation path bypasses proposals | Stop release; fix boundary and add ownership regression tests. |
| Any raw source, transcript, process output, transport payload, prompt, provider payload, or secret leaks into metadata-only stores or hosted telemetry | Stop release; fix classifier/schema/storage and expand adversarial tests. |
| Any non-loopback egress succeeds under air-gap | Stop release; fix security broker and exporter/transport policy enforcement. |
| Native PTY leaves orphan processes after close/crash/shutdown | Stop release for affected platform; fail closed or remove platform from GA matrix. |
| Migration cannot recover from interruption/corruption | Stop release; fix migration/recovery before GA. |
| Cargo-deny reports unreviewed production network/crypto/process dependency risk | Stop release until security review is complete or dependency is removed. |
| Evidence claims a platform or capability that lacks passing tests | Remove claim or block GA until evidence exists. |

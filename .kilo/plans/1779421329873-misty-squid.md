# Devil IDE Control-First Roadmap Plan

## Purpose

Move Devil IDE from the current safe local-core substrate toward the target control-first adaptive IDE: fast manual editing remains primary, while LSP, AI, agents, terminal, plugins, collaboration, and remote work are opt-in, permissioned, reviewable, audited, and rollback-capable.

This plan treats `plans/` plus current source/tests as authority. There is no `docs/` directory in this repository. Older architecture-review claims are useful historical context only when they match current code and evidence.

## Current Truth

The supplied assessment is directionally correct. The most important verified facts are:

- Phase status is: Phase 0 accepted, Phase 1 accepted, Phase 2 partially accepted, Phase 3 not accepted, Phases 4-8 future-gated.
- Manual saves are proposal-mediated through `AppComposition`, `SaveWorkflowService`, and `WorkspaceActor::save_file_with_proposal`.
- UI is projection-only and currently emits command intents rather than owning editor/workspace/proposal authority.
- `devil-editor::request_save` still rejects degraded buffers and still requires `TextSnapshot::try_full_text()` for save payload assembly.
- Generalized proposal DTOs, lifecycle state, registered text edits, closed-file create/delete/rename, registered save apply, and side-effect-free batch planning exist.
- Batch runtime mutation, multi-file atomicity, format/code-action execution, generic workspace edits, terminal/plugin/remote/collab routes, and universal rollback remain incomplete or denied.
- App-level audit-failure rollback for closed-file mutations currently uses direct filesystem operations in `crates/devil-app/src/lib.rs`; this should move behind workspace/platform authority.
- `devil-index` contains meaningful semantic scaffolding, but Phase 3 and LSP supervision remain not accepted.
- AI providers are stubs; `devil-agent`, `devil-tracker`, and `devil-memory` remain placeholder/inert crates.
- File-backed storage persists only a subset of metadata; proposal audit, event metadata, assisted-AI audit, and delegated-task linkage durability need explicit migration work.
- CI does not fully match the documented local phase gates because `cargo test --workspace` and `cargo check --workspace` lack `--all-targets`.

## Non-Negotiable Invariants

Every implementation package must preserve these invariants:

- UI renders projections and emits intents only; it never owns editor buffers, workspace VFS state, proposal lifecycle state, durable storage, or mutation authority.
- Editor owns text, transactions, undo/redo, snapshots, dirty state, viewport projections, snapshot leases, and save payload assembly.
- Workspace owns file identity, trust/path boundaries, watcher state, disk fingerprints, workspace generation, and durable file mutation.
- All non-user-direct mutation enters as `WorkspaceProposal` and progresses through app/protocol lifecycle handling before editor/workspace mutation.
- Saves preserve expected fingerprint, file content version, workspace generation, buffer version, snapshot id, principal, capability, correlation, and causality preconditions.
- Unsupported future runtime routes fail closed before side effects.
- Audit and observability are metadata-only by default and reject zero/invalid core identifiers.
- Placeholder runtime crates remain inert until their ADR, dependency policy, protocol contracts, tests, ownership proof, and evidence exist.
- Semantic, LSP, AI, plugin, terminal, collaboration, and remote work must not block editor input, viewport projection, proposal validation, or saves.

## Target Architecture

The end state should have these authority boundaries:

- `devil-ui`: projection renderer, command-intent surface, trust panels, proposal ledger views, task views, and status badges.
- `devil-app`: composition root and app-domain orchestration for commands, saves, proposal lifecycle, trust projections, and subsystem integration.
- `devil-editor`: editor/text authority through transactions, snapshots, chunk leases, viewport projections, undo/redo, and save payload sources.
- `devil-project`: workspace/VFS authority through discovery, path policy integration, file identities, watcher deltas, save/write operations, and rollback checkpoints.
- `devil-platform`: native filesystem/process/PTY primitives, all exposed through fail-closed higher-level services.
- `devil-protocol`: shared IDs, DTOs, ports, proposal payloads, lifecycle states, semantic/LSP DTOs, AI/tracker/task DTOs, storage records, and projection-safe contracts.
- `devil-security`: capability broker and policy enforcement for filesystem, LSP, terminal, plugin, network, AI/provider, and remote operations.
- `devil-observability`: metadata-only events, audits, redaction, ordering, and replay-safe envelopes.
- `devil-storage`: durable metadata stores with migrations, corruption quarantine, atomic flush, and no raw source/prompt/terminal persistence by default.
- `devil-index`: descriptor-first semantic fabric with bounded queues, cancellation, freshness, privacy scopes, metadata persistence, and proposal-only refactor outputs.
- Future LSP/runtime/AI/agent/plugin/terminal/collab/remote surfaces: clients of protocol, policy, proposals, events, and storage, not mutation authorities.

## Program Shape

Use two immediate tracks that converge before Phase 3 acceptance:

- Core IDE credibility track: renderer-backed UX, degraded-mode visibility, chunked/degraded saves, conflict UX, durable settings/session/trust metadata.
- Control substrate track: universal proposal lifecycle, rollback authority, ledger projections, durable audit/event metadata, app modularization.

The core track can run in parallel with proposal-service work where it only consumes projections/intents. Runtime surfaces that can produce edits, commands, network egress, or durable side effects must wait for the control substrate.

## Phase 1: Foundation UX And Large-File Hardening

Goal: make Devil credible as a non-AI IDE while preserving the current projection-only architecture.

### 1.1 Renderer-backed projection shell

Implementation work:

- Build the production renderer adapter as a consumer of `ShellProjectionSnapshot` and active-buffer viewport projections.
- Add file tree, tabs or buffer list, command palette, status area, proposal panel placeholders, conflict/degraded banners, keyboard/focus handling, clipboard, IME, accessibility tree, and renderer metrics.
- Keep all editor/workspace actions flowing as `CommandDispatchIntent` values resolved by app/editor/workspace authority.
- Add viewport scrolling, cursor/selection projection, dirty/save/conflict indicators, and degraded-mode feature omissions.

Acceptance criteria:

- Renderer never depends on `devil-editor`, `devil-project`, `devil-storage`, or app internals.
- UI tests render snapshots and drain intents without mutating editor/workspace state.
- Large files render via viewport slices only; no full-source projection regression.
- Evidence records p50/p95 input-to-paint, frame variance, focus, clipboard, IME, and accessibility behavior.

### 1.2 Chunked/degraded save support

Problem: degraded buffers currently cannot save because save assembly requires full-source materialization.

Design direction:

- Keep `SaveWorkflowService -> WorkspaceActor::save_file_with_proposal` as the only durable save path.
- Extend editor save assembly from inline full text only to a payload source model:
  - small eligible buffers: inline text remains allowed within the existing full-cache budget;
  - degraded/large buffers: a chunked snapshot save source streams ordered chunks from editor/text-owned snapshot data.
- Add a protocol/app descriptor for save payload metadata: snapshot id, buffer version, byte length, content hash, chunk count, chunk hashes, degraded flag, and lease/correlation metadata.
- Add a workspace/project save content abstraction that can atomically write inline or chunked payloads without depending on editor types.
- Add a platform atomic writer that stages bytes to a same-directory temp file, verifies byte count/hash as it writes, flushes/syncs where supported, then atomically replaces the target.
- Preserve workspace preflight before mutation: expected disk fingerprint, file content version, workspace generation, capability, principal, correlation, causality, path policy, and conflict checks.
- Do not enable non-atomic fallback.
- Revisit file-write policy limits so trusted manual saves of large existing files can be allowed under explicit policy while AI/plugin/agent/generated writes remain tightly capped.

Likely code areas:

- `crates/devil-text/src/lib.rs`: chunk iteration/lease APIs and tests.
- `crates/devil-editor/src/lib.rs`: save request payload descriptor/source and degraded-save state handling.
- `crates/devil-protocol/src/lib.rs`: save payload descriptor DTOs if needed by app/workspace contracts.
- `crates/devil-app/src/lib.rs` or extracted app-domain save modules: `SaveWorkflowService` integration.
- `crates/devil-project/src/lib.rs`: `WorkspaceSaveRequest` and streaming/atomic write orchestration.
- `crates/devil-platform/src/lib.rs`: atomic write from chunks/reader/closure.
- `crates/devil-security/src/lib.rs`: write-size context and policy tests.

Acceptance tests:

- A degraded large file saves successfully without calling `TextSnapshot::try_full_text()`.
- A stale external overwrite still rejects and preserves dirty editor text.
- A denied/untrusted/oversize-policy save rejects without disk mutation.
- Atomic writer failure preserves the original target and cleans up temp files where possible.
- Chunk byte count/hash mismatch fails closed before replace.
- UI never receives full source as part of save UX.

### 1.3 Conflict and degraded-mode UX

Implementation work:

- Project stale/conflict/save-failed state into UI snapshots with actionable metadata.
- Add intents for reload disk, compare, keep mine, accept disk, retry save, and cancel conflict resolution.
- Route any disk-mutating resolution through save/proposal/workspace authority.
- Surface degraded-mode limitations: viewport-only, bounded search, stale/omitted overlays, chunked-save status, and feature unavailability.

Acceptance criteria:

- Conflict UX never clobbers disk without proposal/workspace preconditions.
- Dirty editor text is preserved across stale, conflict, denial, and failed resolution attempts.
- Large-file limitations are visible and testable in projection snapshots.

## Phase 2: Universal Proposal Lifecycle, Rollback, And Ledger

Goal: complete Phase 2B so every future mutation source can use one validated, previewed, approved, audited, and rollback-aware substrate.

### 2.1 Modularize app-domain proposal/save code

Implementation work:

- Extract `crates/devil-app/src/lib.rs` proposal/save/projection responsibilities into app-domain modules without creating a new crate unless an ADR authorizes one.
- Suggested modules: `proposal/lifecycle`, `proposal/validation`, `proposal/preview`, `proposal/executor`, `proposal/batch`, `proposal/rollback`, `save_workflow`, `projection`, and `trust_projection`.
- Preserve public behavior and existing integration tests during extraction.

Acceptance criteria:

- Save/conflict tests remain green after each extraction step.
- No dependency-policy changes are needed unless a new crate is deliberately introduced.
- App modularization does not move mutation authority into UI, editor, storage, or future runtime crates.

### 2.2 Lifecycle state machine and generic save migration

Implementation work:

- Keep `AppProposalCoordinator` as the app-owned lifecycle coordinator until a later ADR approves a different boundary.
- Require create/register, validate, preview, approve where needed, apply, audit, and final response ordering.
- Reject missing lifecycle context and invalid transitions with typed diagnostics.
- Make generic `ProposalPayload::SaveFile` apply equivalent to manual save after chunked save payload source support exists, or keep it denied with an explicit migration rationale until equivalence is proven.

Acceptance criteria:

- Manual save and generic save proposal execution either share implementation or have documented safe separation.
- Created-to-applied direct transitions are rejected with no side effect.
- Existing manual save caller behavior remains `Ok(AppSaveOutcome::Rejected(_))` for stale/conflict/denial and preserves dirty text.

### 2.3 Workspace-authorized rollback checkpoints

Problem: app currently performs some audit-failure rollback through direct filesystem calls.

Design direction:

- Add workspace/project APIs for preparing mutation checkpoints and applying compensation:
  - create: expected-absence checkpoint;
  - delete/save/closed-file edit: ephemeral file checkpoint or inverse write plan owned by workspace/platform;
  - rename: inverse rename plan with source/destination preconditions;
  - open-buffer text edit: editor undo group/transaction rollback.
- Keep checkpoint content ephemeral and non-audit by default; durable audit records should store checkpoint IDs, hashes, byte counts, and disposition, not raw source.
- Add cleanup/expiry for abandoned checkpoints.
- Remove app-level direct filesystem rollback once workspace compensation APIs exist.

Acceptance criteria:

- Audit failure after create/delete/rename/save rolls back through workspace/platform authority.
- Rollback failure emits metadata-only diagnostics and preserves user buffers.
- Static search shows no app-domain direct file writes/removes/renames in production rollback paths.
- Large rollback checkpoints avoid unnecessary full-source app materialization.

### 2.4 Route-specific proposal executors

Implementation work:

- Open-buffer text edits route through editor transaction APIs with buffer version, snapshot id, edit range validation, undo group, and stale rejection.
- Closed-file create/delete/rename/save route through workspace APIs with expected absence/fingerprint/version/generation/path policy/capability preconditions.
- Workspace edit support expands in stages:
  - single closed-file operation stays supported;
  - multiple closed-file operations with ordered non-atomic semantics;
  - multiple open-buffer text edits with editor undo groups;
  - mixed open-buffer plus closed-file edits only after rollback/checkpoint proof exists.
- Format and code-action payloads are executable only when they already contain safe edit payloads; command-only or mixed command actions stay denied until command/terminal gates exist.

Acceptance criteria:

- Stale open-buffer edit proposal rejects before text mutation.
- Closed-file conflict rejects before disk mutation.
- Open-file delete/rename protections remain intact.
- Format/code-action command-only payloads remain denied.
- Unsupported terminal/plugin/remote/collab routes remain fail-closed.

### 2.5 Batch runtime execution

Implementation work:

- Convert side-effect-free batch contracts into a real runner with explicit phases: prepare, preflight, mutate, commit, audit, finalize, rollback.
- Start with homogeneous supported routes under `PrepareAllBeforeMutate`.
- Add `OrderedNonAtomic` with mandatory partial-failure records.
- Add `AllOrNothing` only after exact rollback proof works for every included route.
- Deny impossible atomicity claims before mutation.
- Deny mixed editor/workspace/terminal/plugin/remote/collab batches until every contained route has an accepted executor and cross-authority rollback semantics.

Acceptance criteria:

- All-success batch applies in deterministic order.
- Mid-batch failure rolls back or records exact partial-failure metadata matching declared atomicity.
- Rollback failure is typed and audited.
- Batch apply success is impossible without audit-before-success.
- Unsupported mixed route has zero side effects.

### 2.6 Proposal ledger and trust projections

Implementation work:

- Add projection DTOs for proposal list, selected preview, affected targets, warnings, lifecycle state, policy decisions, rollback availability, partial failures, and result summaries.
- Add UI intents for select, preview refresh, approve, reject, cancel, apply, and rollback.
- Persist/recover enough proposal lifecycle metadata to build a ledger after restart.
- Keep preview bounded/redacted; no raw source, raw prompts, raw terminal output, or secrets in durable ledger records by default.

Acceptance criteria:

- UI renders proposal ledger snapshots without editor/workspace dependencies.
- Ledger survives restart for metadata-only proposal/audit state.
- Audit/event storage failure prevents reporting success where audit-before-success is required.

## Phase 3: Durable Metadata, Storage, And Replay Foundations

Goal: make audit, event, session, trust, semantic, and future tracker storage production-ready before activating high-power runtime surfaces.

Implementation work:

- Expand file-backed persisted state to include protocol proposal audit, event metadata, assisted-AI audit, delegated-task linkage, protocol trust/session/file metadata, and semantic metadata as appropriate.
- Add schema versions and forward migrations for each persisted namespace.
- Make file-backed flush crash-safer: write temp file, flush/sync, atomic replace, quarantine corrupt prior state.
- Add metadata-only validation that rejects raw source, raw prompt, raw terminal output, provider payload bodies, and secrets by default.
- Add replay drills that reconstruct save/proposal flows from metadata IDs, hashes, states, and event order.

Acceptance criteria:

- Proposal audit and event metadata round-trip through file-backed storage.
- Corrupt storage is quarantined without losing the ability to start with safe defaults.
- Migration tests cover old schema to new schema.
- Replay can reconstruct critical save/proposal lifecycle without source bodies.

## Phase 4: Semantic Fabric And Supervised LSP

Goal: accept Phase 3 only after semantic/LSP work is non-blocking, descriptor-first, privacy-aware, and proposal-only for mutations.

### 4.1 Semantic boundary remediation

Implementation work:

- Ensure production semantic inputs are workspace-authored discovery DTOs, watcher deltas, snapshot descriptors, changed ranges, and bounded chunk leases.
- Keep any direct repository scanner behavior test-only or remove production filesystem traversal from `devil-index`.
- Keep `devil-index` free of app/editor/project/UI mutation authority and within dependency-policy limits.
- Replace normal full-source indexing with descriptor/chunk/changed-range paths; small-buffer full-text may remain an explicit bounded optimization.
- Enforce syntax/semantic cache freshness by file identity, content hash, snapshot id, workspace generation, grammar version, model metadata version, privacy scope, and schema version.
- Persist semantic records as metadata only: symbols, graph edges, ranges, hashes, provenance, freshness, privacy scope, tombstones.
- Keep vector indexing and embeddings deferred.

Acceptance criteria:

- Large snapshots can be indexed without `try_full_text()` as the default path.
- Privacy downgrade tombstones or invalidates records before query exposure.
- Same content in different files cannot reuse file-specific semantic outcomes unsafely.
- Semantic storage rejects source bodies and vectors.
- Phase 3 evidence artifacts are produced but acceptance remains not accepted until the full checklist is complete.

### 4.2 Semantic actor and query APIs

Implementation work:

- Harden `IndexingActor` outcomes: accepted, coalesced, cancelled, rejected, degraded, stale, unavailable, resync-required.
- Prioritize open-buffer/recent-edit work over background workspace discovery.
- Add lexical maps, graph records, normalized query DTOs, freshness/degradation status, and bounded query APIs for navigation, completion ranking, AI context selection, agent planning, test impact, and refactor previews.
- Ensure semantic refactors produce proposal-ready payloads only.

Acceptance criteria:

- Queue saturation never blocks editor input, viewport projection, proposal validation, or saves.
- Query results carry freshness/privacy/degraded metadata.
- Refactor preview tests prove no direct editor/workspace mutation.

### 4.3 LSP supervisor

Implementation work:

- Add LSP runtime only after dependency policy and evidence allow it; prefer a dedicated crate only if policy/ADR explicitly authorizes it.
- Supervise server process lifecycle, JSON-RPC transport, capability negotiation, restart budget, circuit breaker, shutdown, and health state.
- Gate server launch on workspace trust, binary allowlist, path policy, and no-network-refresh/update policy.
- Add bounded queues per server/workspace/feature class, cancellation, timeout budgets, stale-response suppression, and degraded/unavailable outcomes.
- Normalize diagnostics, completion, hover, definition, references, semantic tokens, formatting, rename, and code actions into protocol DTOs.
- Translate formatting/rename/organize-imports/quick-fix/refactor/workspace edits into `WorkspaceProposal`; deny command-only or unsafe mixed actions.

Acceptance criteria:

- Untrusted workspaces cannot launch LSP.
- Slow/crashing/malformed servers cannot corrupt editor, workspace, save, or index state.
- LSP queues under pressure do not block input/save paths.
- Edit-producing LSP features route through proposal lifecycle only.
- `plans/evidence/phase-3/predictive-semantic-fabric.md` final checklist is complete before Phase 3 is marked accepted.

## Phase 5: Trust Surfaces And Policy-Bound Assisted AI

Goal: implement AI assistance as inspectable, local-first, context-manifest-backed help, not autonomous mutation.

Implementation work:

- Add or finalize protocol DTOs for mode badge, air-gap status, context manifest, privacy inspector, permission budget, risk budget, cost budget, provider route, proposal ledger linkage, and approval checklist.
- Implement trust projections from proposal, event, storage, policy, semantic, provider, and tracker metadata.
- Strengthen `DenyByDefaultBroker` enforcement for structured LSP binary and network host allowlist/blocklist contexts before provider activation.
- Expand provider contracts for streaming, cancellation, structured outputs, context window, cost metadata, provider health, local/cloud route metadata, and redaction status.
- Implement local-provider routing first; cloud providers require explicit allowlist, workspace trust, user approval, redaction, cost budget, air-gap tests, and provider audit metadata.
- Generated edits become proposals only; explain-only responses cite context manifest entries and create no mutation-capable proposal.
- Activate tracker storage minimally for AI run metadata: selected context, provider route, approvals, proposal IDs, verification output, redacted errors, cost/risk summaries.

Acceptance criteria:

- Provider call cannot occur without approved context manifest and policy grant.
- Air-gap mode blocks unapproved outbound access.
- Users can inspect what context was used or excluded and why.
- AI edit output cannot mutate editor/workspace directly.
- Prompt/source/provider payloads are redacted from durable storage by default.

## Phase 6: Delegated Agents And Terminal Governance

Goal: support cancellable task objects with scoped tools, not uncontrolled chat/tools.

Implementation work:

- Add/revise ADRs and dependency policy for agent runtime, tracker ledger, terminal supervisor, command taxonomy, and verification records.
- Implement terminal supervisor before broad agent activation: command classification, working directory policy, environment policy, timeout, cancellation, bounded output, redacted summaries, and no unrestricted PTY access.
- Treat terminal-caused file changes as external watcher changes unless explicitly converted into proposals.
- Activate `devil-tracker` as metadata-only run/task ledger: task goal, scope, context manifest, policy grants, tool calls, proposal IDs, approvals, verification output, recovery attempts, blocked states.
- Activate `devil-agent` as a state machine: observing, planning, proposing, waiting approval, applying through proposals, verifying, recovering, blocked, cancelled.
- Forbid self-approval for risky actions.

Acceptance criteria:

- Agent cannot mutate editor, disk, terminal, network, memory, tracker, or settings directly.
- Terminal commands require policy grants and produce redacted bounded output summaries.
- Agent proposals are reviewable and cancellable.
- Verification output links to tracker/proposal ledger metadata.
- Task runs can be replayed from metadata without raw source snapshots.

## Phase 7: Sandboxed Plugin Runtime

Goal: add extensibility without ambient authority.

Implementation work:

- Add plugin ADR and dependency-policy activation before creating/activating runtime code.
- Define WASM/WASI ABI, manifest schema, compatibility ranges, contribution points, signatures/trust metadata, capability declarations, quotas, cancellation, and crash isolation.
- Expose host calls only through protocol DTOs and capability checks.
- Give plugins scoped projection-safe context; writes become proposals.
- Namespace plugin storage and emit redacted plugin events.

Acceptance criteria:

- Malicious plugin tests prove no ambient filesystem, process, network, editor, workspace, AI, memory, or storage access.
- Plugin writes route through proposal lifecycle.
- Plugin crash/timeout cannot corrupt editor/workspace/app state.

## Phase 8: Collaboration

Goal: support convergent multiplayer without bypassing editor/workspace/proposal ownership.

Implementation work:

- Decide CRDT versus operation log in an ADR before implementation.
- Add protocol DTOs for collaborator identity, presence, version vectors, operations, authorship, causality, shared proposal approvals, and conflict records.
- Merge remote collaborative operations into editor transactions without blocking local input.
- Preserve dirty buffers, proposal semantics, save preconditions, and metadata-only audit.
- Define shared proposal approval and rollback semantics.

Acceptance criteria:

- Concurrent edits converge deterministically.
- Undo semantics are clear for local, remote, and shared operations.
- Collaborative file operations do not bypass workspace authority.
- Shared approvals, conflicts, and authorship are replayable from metadata.

## Phase 9: Remote Development And Production Hardening

Goal: distributed IDE substrate with operational readiness.

Implementation work:

- Add remote edge workspace agent ADR, encrypted transport, identity/trust model, latency/failure semantics, and capability-scoped remote filesystem/process/LSP/index/terminal clients.
- Preserve local projections and proposal-mediated writes for remote operations.
- Complete durable migrated stores for sessions, trust, audit, tracker, memory, semantic index metadata, plugin state, collaboration logs, and remote sessions.
- Expand CLI diagnostics beyond version output: configuration inspection, storage health, trust state, dependency policy, renderer health, LSP health, provider health, index health, and replay checks.
- Add privacy-preserving metrics, supply-chain checks, policy profiles, storage migration rollback tests, and disaster-recovery/replay drills.

Acceptance criteria:

- Remote writes cannot bypass proposals, capabilities, fingerprints, or conflict checks.
- Metadata-only replay reconstructs save/proposal/AI/task/collab critical flows.
- Storage migrations are recoverable.
- Every active subsystem has health diagnostics and CI coverage.

## Stop Conditions

Stop implementation and re-plan if any of these occur:

- A future runtime surface can mutate editor/workspace state without a proposal.
- UI gains editor/workspace/proposal/storage authority.
- A save can clobber external disk changes or skip fingerprint/version preconditions.
- Degraded large-file save requires UI full-source access.
- App-domain rollback writes/removes/renames files directly after workspace compensation APIs are available.
- Audit success can be reported before metadata emission/persistence when audit-before-success is required.
- Raw source, raw prompts, secrets, terminal output, or provider payloads become durable defaults.
- Phase 3 or future runtime phases are marked accepted without the required evidence artifacts.
- Semantic/LSP/AI/plugin/terminal/remote/collab work blocks editor input, viewport projection, proposal validation, or save workflows.

# Phase 6 Development and Implementation Plan - Real-Time Multiplayer Collaboration

Prepared: 2026-05-25  
Plan status: Ready for implementation planning review  
Target phase: Current ledger Phase 6, real-time multiplayer collaboration  
Current phase state: Not started / future-gated  
Primary authorities: `plans/phase-status-ledger.md`, `plans/implementation-plan.md`, `plans/remaining-implementation-tasks-plan-v0.1.md`, `plans/dependency-policy.md`

## 1. Executive Summary

Phase 6 advances Devil IDE from an accepted local, proposal-mediated, plugin-capable IDE substrate into a safe real-time multiplayer collaboration substrate. The implementation must preserve the repository's non-negotiable ownership rule: UI, collaboration, plugins, AI, LSP, terminal, and remote surfaces must not directly own editor text or workspace writes. User text edits from authenticated collaborators may become editor-owned transactions only through the accepted collaboration operation layer and active session policy. Durable file operations, AI/LSP/plugin outputs, and workspace mutations remain proposal-mediated.

The current repository state is ready to plan Phase 6 but not ready to claim any Phase 6 acceptance:

- Phases 0 through 5 are accepted in `plans/phase-status-ledger.md`.
- Phase 6 through Phase 8 remain future-gated and require ADRs, dependency-policy activation, protocol contracts, runtime tests, ownership tests, and evidence.
- There is no `crates/devil-collaboration` workspace member today.
- `devil-protocol` has collaboration placeholders such as `SnapshotConsumerKind::Collaboration`, `ProposalTargetKind::CollaborationSession`, and future-surface planning gates, but no active collaboration runtime DTO set.
- `devil-app` intentionally rejects collaboration proposal targets as unsupported until activation gates are complete.
- Durable collaboration/replay storage requires a follow-up storage-retention decision because ADR-0005 is accepted with reservations.

Recommended implementation strategy: activate Phase 6 as an operation-log collaboration system with deterministic ordering, version vectors, bounded non-blocking editor integration, metadata-only durable audit, and explicit conflict/undo semantics. A CRDT alternative must be considered in the ADR, but the default execution path should prefer an operation-log substrate for the first accepted Phase 6 slice because it fits the existing proposal, audit, storage, and deterministic replay model.

## 2. Scope

### In Scope

- Accepted collaboration ADRs covering operation-log versus CRDT strategy, identity, permissions, conflict policy, offline/reconnect behavior, undo semantics, transport boundary, and retention.
- `devil-collaboration` activation with explicit dependency policy and no direct dependency on app, UI, editor, project, or remote internals.
- Protocol DTOs for collaboration sessions, participants, document operations, version vectors, presence, cursors, selections, acknowledgements, gaps, conflicts, undo records, shared proposal approvals, and audit metadata.
- App-owned collaboration composition and transport adapters that exchange protocol envelopes without granting remote filesystem, process, terminal, or workspace authority.
- Editor integration that converts local transactions into collaboration operations and validated remote participant operations into editor-owned transactions without blocking local input.
- Shared proposal workflows for multi-user AI actions, LSP refactors, plugin commands, and file operations.
- Projection-only UI awareness for presence, selections, proposal approvals, comments, agent activity metadata, and conflict states.
- Metadata-only operation ordering, event, storage, and replay records without storing full source snapshots by default.
- Phase 6 evidence artifacts and `xtask` governance mirroring accepted Phase 3 through Phase 5 patterns.

### Out Of Scope

- Phase 7 edge remote development, remote filesystem authority, remote process execution, PTY/terminal expansion, remote LSP server launching, and edge workspace agents.
- Marketplace, VS Code compatibility, Node extension execution, arbitrary host scripting, hosted telemetry, hosted embeddings, and cloud provider activation.
- Unrestricted source retention, full source snapshot persistence by default, or raw collaboration transcripts in observability records.
- Direct UI mutation of editor/workspace state.
- Direct workspace writes from collaboration runtime. Durable file changes still require proposal/workspace preconditions.

## 4. Key Assumptions

- Calendar assumes implementation starts on 2026-05-26. If approval starts later, all dates shift by the same number of business days.
- Phase 0 through Phase 5 remain accepted and green during Phase 6 work.
- Any new external dependency for operation transforms, fuzzing, transport, or storage requires supply-chain review and must pass `cargo deny check`.
- The accepted initial runtime can use app-owned transport adapters and deterministic simulation harnesses before broad network rollout, but the final Phase 6 acceptance must prove real-time multi-participant operation exchange through the approved collaboration transport boundary.
- Collaboration text operations are active-session participant edits. They are not durable workspace writes. Durable file operations and non-user-generated edits stay proposal-mediated.
- Phase 6 can update dependency policy to allow `devil-app` to compose `devil-collaboration`; `devil-collaboration` itself should remain isolated to approved dependencies unless the ADR justifies an explicit policy change.

## 14. Deployment And Rollout Strategy

| Stage | Timing | Audience | Enablement | Exit Criteria | Rollback |
| --- | --- | --- | --- | --- | --- |
| Stage 0 Hidden | M2 to M3 | Developers only | Compile/test only, runtime disabled | DTO and governance tests pass | Remove/disable feature flag path before runtime use. |
| Stage 1 Simulation Alpha | M3 | Engineering team | Deterministic in-process harness | Convergence and security tests pass | Disable runtime crate integration, keep DTOs if accepted. |
| Stage 2 Internal Trusted Workspace Alpha | M4 | Phase 6 implementers | Feature flag on for trusted local sessions | Editor/app regression and dirty-buffer tests pass | Feature flag off; close sessions; preserve local buffers. |
| Stage 3 Dogfood Beta | M7 | Selected internal users | Opt-in trusted workspaces only | No critical/high defects, redaction audit clean, rollback drill complete | Feature flag off; metadata retained for audit; no source recovery needed. |
| Stage 4 Accepted Phase 6 | M8 | Main development branch | Default policy remains conservative; collaboration available under accepted config | All evidence and global gates pass | Disable accepted config if regression found; normal save/proposal flows unaffected. |

Rollout principles:

- Collaboration remains opt-in until final acceptance.
- Feature disablement must not corrupt or discard open buffers.
- Session shutdown must preserve local editor state and emit metadata-only audit events.
- No migration may be irreversible before Phase 8 hardening unless explicitly accepted by ADR.
- Remote workspace and terminal/process capabilities remain unavailable even when collaboration is enabled.

# Phase 7 End-to-End Implementation Plan - Edge-Executed Remote Development

Prepared: 2026-05-25  
Status: Planning baseline  
Target phase: Phase 7, edge-executed remote development  
Primary authority: `plans/phase-status-ledger.md`, `plans/implementation-plan.md`, `plans/remaining-implementation-tasks-plan-v0.1.md`, and accepted `plans/evidence/phase-6/` artifacts

## 1. Planning Baseline

Phase 7 is currently future-gated. Phase 0 through Phase 6 are accepted, and Phase 7 must not activate runtime behavior until its own ADRs, dependency-policy entries, protocol contracts, contract tests, ownership tests, security tests, and evidence artifacts exist.

Current authoritative scope defines Phase 7 as edge-executed remote development. Older foundational documents used the label Phase 7 for integrated terminal emulation; that label is treated as historical for this plan. Terminal, process, PTY, and LSP behavior in this plan is limited to policy-gated remote-development execution through the edge workspace agent and does not authorize a standalone local terminal runtime unless the Phase 7 ADRs explicitly approve that boundary.

Planning horizon: 16 calendar weeks for full acceptance. A 10 to 12 week delivery is possible only if the accepted scope is reduced to a remote-filesystem and proposal-mediated write MVP without remote process, PTY, LSP, semantic-index, and offline-resume acceptance.

Primary milestone: M7, edge workspace agent and local projection accepted.  
Primary dependency gate: remote writes cannot bypass proposals, fingerprint checks, workspace-generation checks, buffer-version checks, snapshot checks, capability checks, correlation IDs, causality IDs, or metadata-only audit requirements.

## 2. Scope And Objectives

### 2.1 In Scope

| Area | Scope |
| --- | --- |
| Governance | Phase 7 ADRs, dependency-policy activation, `xtask` checks, phase evidence, final ledger update after acceptance. |
| Protocol | Remote workspace lifecycle DTOs, remote transport envelopes, remote filesystem snapshots and operations, remote process and PTY descriptors, remote LSP descriptors, remote semantic query descriptors, latency hints, offline resume manifests, remote audit records. |
| Remote runtime | Capability-scoped edge workspace agent owning remote filesystem, process, LSP, semantic-index, and terminal execution on the remote side. |
| App integration | App-owned composition, remote command routing, local projection updates, proposal bridge, collaboration identity integration, and remote session lifecycle control. |
| Mutation safety | Generalized proposals for remote edits and file operations, fingerprint and version preconditions, stale/conflict rejection, dirty-buffer preservation. |
| Security | Encrypted transport, authentication, authorization, secrets boundaries, path policy, process policy, egress policy, air-gap behavior, untrusted workspace denial. |
| Latency and resilience | Local optimistic projections, remote acknowledgements, operation logs, version vectors, reconnect behavior, degraded-network state, offline resume and reconciliation. |
| Observability and storage | Metadata-only remote audit records, redacted event envelopes, non-zero correlation/causality/event sequence validation, storage of metadata and replay manifests without raw-source retention by default. |
| QA and evidence | Contract, ownership, security, integration, resilience, performance, redaction, and global phase-gate evidence. |

### 2.2 Out Of Scope

| Area | Explicit Non-Goal |
| --- | --- |
| Direct mutation | Remote agent, UI, plugins, AI, LSP, terminal, and collaboration runtime must not directly mutate local editor text or durable workspace state outside accepted editor/workspace/proposal authority. |
| Raw network helpers | No ad hoc remote networking APIs outside approved protocol ports, transport envelopes, security policy, and event correlation. |
| Hosted telemetry | No hosted telemetry activation. Events remain metadata-only by default. |
| Hosted AI/cloud provider expansion | No hosted AI provider, hosted embedding, vector-store, marketplace, or cloud-provider expansion beyond remote-development transport and approved egress policy. |
| Raw source retention | No durable raw source snapshots, raw terminal transcripts, raw process output bodies, or raw collaboration transcripts by default. |
| Standalone local terminal | No standalone terminal/process runtime outside the Phase 7 remote-development agent scope unless separately approved by ADR, policy, tests, and evidence. |
| Phase 8 hardening | Durable production operations, broad ecosystem readiness, and full enterprise rollout hardening remain Phase 8 unless required as Phase 7 acceptance gates. |

### 2.3 Measurable Objectives

| ID | Objective | Measurable Acceptance Target |
| --- | --- | --- |
| O1 | Activate Phase 7 through governance rather than ad hoc implementation. | `xtask check-deps` fails closed when Phase 7 ADR, policy, protocol, owner, or evidence markers are missing or inconsistent. |
| O2 | Provide a capability-scoped remote workspace runtime. | Edge agent lifecycle tests cover create, connect, authenticate, authorize, active, degraded, reconnecting, offline, closing, closed, and denied states. |
| O3 | Preserve proposal-mediated mutation. | All remote write, create, delete, rename, format, code-action, and workspace-edit flows either produce accepted proposals or are denied; stale and conflict cases reject without dirty-buffer loss. |
| O4 | Preserve projection-only UI. | UI tests prove remote UI actions emit intents/projections only and do not depend on app, editor, project, storage, remote, terminal, or LSP internals. |
| O5 | Enforce remote security policy. | Tests prove deny-by-default behavior for untrusted workspaces, missing capabilities, path escape, unsafe process launch, egress denial, air-gap mode, invalid secrets, and zero identifiers. |
| O6 | Maintain responsiveness under remote latency. | Approved performance harness shows local typing, viewport projection, and command dispatch remain within the Phase 6 accepted editor overhead budget plus a Phase 7-approved remote overhead allowance. |
| O7 | Reconcile offline and reconnect behavior safely. | Offline resume tests prove operation logs reconcile deterministically or surface explicit conflicts without silent data loss. |
| O8 | Keep observability metadata-only by default. | Storage and observability tests reject raw-source/raw-transcript audit markers and reject zero `CorrelationId`, nil `CausalityId`, or zero `EventSequence`. |
| O9 | Integrate with collaboration semantics. | Multi-participant remote-session tests share collaboration identity, proposal approval, operation ordering, and replay metadata without granting remote workspace authority to collaboration runtime. |
| O10 | Produce final acceptance evidence. | Every expected Phase 7 evidence artifact exists, is current, references passing validation output, and is approved by the sign-off owners. |

## 11. Formal Final Acceptance Criteria And Sign-Off

### 11.1 Final Acceptance Criteria

Phase 7 is accepted only when all criteria below are true.

| Category | Acceptance Criterion |
| --- | --- |
| Governance | Phase 7 ADRs are accepted, dependency policy explicitly activates approved Phase 7 boundaries, and `xtask check-deps` enforces Phase 7 evidence and required protocol symbols. |
| Dependency boundaries | `devil-remote` and any Phase 7 runtime code use only approved dependencies and do not depend on app, UI, editor, project, terminal, LSP, collaboration internals, or other feature internals outside protocol-port mediation. |
| Protocol contracts | Remote lifecycle, transport, filesystem, proposal bridge, process, PTY, LSP, semantic query, latency, offline resume, and audit DTOs have contract tests and validation helpers. |
| App ownership | App owns remote session composition and command routing; UI remains projection-only; remote runtime does not own local editor text or durable workspace mutation. |
| Mutation safety | Remote writes and file operations are proposal-mediated and enforce expected fingerprint, file content version, workspace generation, buffer version, snapshot id, capability, principal, correlation ID, and causality ID. |
| Conflict safety | Stale, conflict, denial, and external-overwrite cases reject or surface explicit conflict states without dirty-buffer loss or silent disk clobbering. |
| Security | Encrypted transport, authentication, authorization, secrets boundaries, egress controls, air-gap behavior, untrusted workspace denial, path policy, process policy, and output limits are tested and enforced. |
| Remote execution | Remote process, PTY, terminal, LSP, and semantic-index execution is policy-gated, bounded, cancellable, observable, and unable to mutate outside proposals. |
| Resilience | Latency, reconnect, duplicate/reordered envelopes, causal gaps, offline resume, operation logs, and version vectors reconcile deterministically or produce explicit conflicts. |
| Collaboration | Remote development integrates with accepted Phase 6 collaboration identity, shared proposal approvals, operation ordering, replay metadata, and presence projections without granting collaboration direct remote authority. |
| Observability | Remote events are metadata-only by default and reject zero `CorrelationId`, nil `CausalityId`, zero `EventSequence`, raw-source retention, and raw-transcript audit markers. |
| Performance | Local editor input, viewport projection, and command dispatch remain within approved Phase 7 budgets under simulated remote latency and reconnect scenarios. |
| Evidence | Every Phase 7 evidence artifact exists, is current, and references reproducible validation output. |
| Global gates | `cargo run -p xtask -- check-deps`, `cargo fmt --all --check`, `cargo check --workspace --all-targets`, `cargo test --workspace --all-targets`, `cargo clippy --workspace --all-targets -- -D warnings`, and CI `cargo deny check` pass or have explicitly accepted warning-level baseline notes. |
| Defects | No P0 or P1 issue remains open; every residual P2 has documented risk acceptance and owner. |
| Ledger | `plans/phase-status-ledger.md` marks Phase 7 accepted only after all other acceptance criteria and sign-offs are complete. |

### 11.4 Final Acceptance Record

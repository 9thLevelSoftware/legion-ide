# Phase 4 Implementation Plan: Native Agentic AI Execution Context

Prepared: 2026-05-24
Plan status: Ready for implementation review
Scope authority: `plans/implementation-plan.md`, `plans/remaining-implementation-tasks-plan-v0.1.md`, `plans/phase-status-ledger.md`, `plans/dependency-policy.md`

## Scope

This plan implements Phase 4 from `plans/implementation-plan.md`: a policy-bound native agentic AI execution context. It assumes Phase 0 through Phase 3 remain accepted as recorded in `plans/phase-status-ledger.md`, including generalized proposals, proposal-only mutation, semantic fabric activation, LSP supervision, metadata-only observability, and vector-index deferral.

Phase 4 must activate AI runtime behavior without giving AI direct mutation authority over editor buffers, disk, terminal, tracker, memory, settings, storage, or workspace state.

## Non-goals

- Do not activate Phase 5 plugins, Phase 6 collaboration, Phase 7 remote development, or Phase 8 hardening as part of Phase 4.
- Do not add vector indexing, vector storage, model-backed retrieval, or durable embeddings unless a separate vector ADR, dependency-policy update, provenance contract, privacy-scope contract, invalidation contract, and evidence suite are accepted.
- Do not activate cloud providers by default. Cloud provider execution remains blocked until an explicit provider-specific ADR, security review, dependency-policy update, allowlist policy, redaction tests, and air-gap tests are accepted.
- Do not let `devil-agent`, `devil-ai`, `devil-ai-providers`, `devil-tracker`, or `devil-memory` depend on `devil-app`, `devil-ui`, editor internals, workspace internals, or direct filesystem/process/network authority.
- Do not persist raw prompts, raw provider responses, secrets, source snapshots, or full terminal output by default.

## Current Baseline

- `plans/evidence/phase-3/predictive-semantic-fabric.md` accepts Phase 3 and explicitly leaves provider egress, embeddings, vector storage, terminal execution, plugins, collaboration, and direct mutation inactive.
- `crates/devil-ai/src/lib.rs` contains a minimal synchronous `ModelProvider`, `ProviderRegistry`, chat request/response DTOs, embedding request/response DTOs, and simple capabilities.
- `crates/devil-ai-providers/src/lib.rs` contains `OllamaStub` and `OpenAiStub`; both reject requests as not implemented.
- `crates/devil-agent/src/lib.rs`, `crates/devil-tracker/src/lib.rs`, and `crates/devil-memory/src/lib.rs` are docs-only placeholders, and `xtask` currently enforces that state.
- `crates/devil-protocol/src/lib.rs` already includes metadata-only `ContextManifest*`, `PrivacyInspector*`, `PermissionBudget*`, `AssistedAi*`, `DelegatedTask*`, and future-surface planning DTOs. Runtime provider invocation is currently represented as `NotEncoded` only.
- `crates/devil-storage/src/lib.rs` already persists metadata-only proposal audit, assisted-AI audit, delegated-task audit linkage, event metadata, and semantic metadata records.
- `crates/devil-observability/src/lib.rs` already validates non-zero correlation, non-nil causality, non-zero event sequence, metadata-only redaction, assisted-AI audit records, and forbidden raw provider/prompt markers.
- `crates/devil-security/src/lib.rs` already has `DenyByDefaultBroker`, path policy, network policy, command taxonomy, and capability-request context, but AI provider, credential, air-gap, and host allowlist semantics need Phase 4 expansion.
- `crates/devil-ui/src/ui.rs` already exposes projection-only surfaces for context manifests, privacy inspector, permission budget, assisted AI, delegated task plans, and proposal lifecycle intents.

## Implementation Strategy

Implement Phase 4 as a sequence of narrow activation gates. The first accepted runtime slice is local-provider, proposal-only, metadata-audited, cancellable agent execution. Cloud providers remain disabled unless a later provider-specific gate is accepted.

The architecture should use this dependency direction:

- `devil-protocol` owns cross-domain DTOs, IDs, ports, validation helpers, and projection contracts.
- `devil-security` owns provider, egress, credential, air-gap, and tool capability decisions.
- `devil-ai` owns provider contracts and the policy-bound provider router.
- `devil-ai-providers` owns provider adapters behind `devil-ai` traits.
- `devil-agent` owns run state machines and produces proposal/tool/provider intents, not mutations.
- `devil-tracker` owns run/task/approval ledger APIs backed by storage metadata.
- `devil-memory` owns opt-in retention metadata and memory candidate review, not vector retrieval.
- `devil-app` may become the composition root only after dependency policy explicitly allows the required Phase 4 edges; app integration must still route mutations through proposal/workspace/editor authorities.
- `devil-ui` remains projection-only and must not gain runtime ownership.

## Work Package 2: Security, Air-gap, And Capability Policy

Files likely touched:

- `crates/devil-security/src/lib.rs`
- `crates/devil-security/tests/*`
- `crates/devil-protocol/src/lib.rs`
- `crates/devil-protocol/tests/dto_contracts.rs`

Steps:

1. Add AI-specific policy structs for provider invocation, local loopback use, remote egress, BYOK credentials, prompt categories, context privacy scopes, permission budgets, memory retention, and tool planning.
2. Add explicit capability ids and namespaces for provider invoke, provider stream, provider cancel, context assemble, tracker write, memory candidate write, memory retain, tool plan, and proposal creation.
3. Use `CapabilityRequestContext.network_target` to enforce host, scheme, and port allowlists for provider calls.
4. Enforce air-gap mode by denying hosted telemetry, cloud providers, hosted embeddings, update checks, gateway calls, and non-approved outbound access.
5. Allow local provider execution only for local process or tightly scoped loopback targets such as `localhost` or `127.0.0.1`, and only when the provider class is local or local-loopback.
6. Deny all provider and network capabilities for untrusted or unknown workspace trust unless a specific read-only metadata route is explicitly allowed.
7. Ensure credential identifiers can be referenced as metadata but credential material never crosses protocol, observability, storage, or tracker records.
8. Add denial event helpers or metadata records for provider-policy refusals.

Exit criteria:

- Air-gap tests prove remote provider, hosted telemetry, hosted embeddings, and gateway access are denied.
- Local loopback provider tests prove only approved local targets are allowed.
- Capability tests prove unknown capability ids remain deny-by-default.
- Credential tests prove secret/token/password values are rejected or redacted.

## Work Package 7: Agent Runtime State Machine

Files likely touched:

- `crates/devil-agent/src/lib.rs`
- `crates/devil-agent/Cargo.toml`
- `crates/devil-agent/tests/*`
- `crates/devil-protocol/src/lib.rs`

Steps:

1. Implement `devil-agent` as a deterministic state machine over protocol DTOs and injected ports, not as an app/editor/workspace owner.
2. Supported states are observing, planning, proposing, waiting for approval, applying, verifying, recovering, blocked, cancelled, completed, and failed.
3. Agent inputs are user intent metadata, editor snapshot descriptors, semantic query DTOs, context manifest references, privacy inspector decisions, permission-budget decisions, tracker records, memory references, diagnostics summaries, and selected ranges.
4. Agent outputs are provider route requests, proposal requests, tracker ledger writes, memory candidate writes, verification requests, cancellation records, and projection DTOs.
5. The agent must never call editor mutation, workspace write, terminal execution, settings write, storage mutation, or provider network APIs directly.
6. Generated edits must become `AssistedAiEditProposalOutput` and then `WorkspaceProposal` values.
7. Applying an approved proposal remains owned by `devil-app`, `devil-editor`, and `devil-project` through existing proposal apply paths.
8. Add cancellation, recovery, replay, and idempotent transition tests.

Exit criteria:

- State-machine tests prove illegal transitions are refused.
- Agent tests prove every mutation-capable output is proposal-only.
- Agent tests prove cancellation preserves run metadata and does not apply pending proposals.
- Replay tests reconstruct state from tracker metadata without raw provider payloads.

## Risk Controls

- Direct mutation bypass: Keep AI/agent crates unable to depend on app/UI/editor/project and require proposal-only output contracts.
- Privacy regression: Reject raw prompt/provider/source markers in protocol validation, observability, storage, tracker, and memory tests.
- Egress regression: Enforce air-gap and local-only policy in `devil-security` before any provider adapter can invoke runtime behavior.
- Nondeterministic replay: Persist state transitions, context manifest hashes, provider route metadata, policy decisions, approvals, proposal ids, verification records, and cancellation records.
- Dependency creep: Require dependency-policy and `cargo deny` review before adding HTTP, keychain, streaming, or provider SDK dependencies.
- Vector-index scope creep: Keep embeddings/reranking as capability metadata until vector storage/retrieval gates are separately accepted.
- UI ownership regression: Keep UI projection-only and preserve existing `xtask` tests that forbid UI dependencies on app/editor/project/storage.
- Save safety regression: Keep save conflict and dirty-buffer preservation tests in the required evidence set.

# Phase 6 Completion Plan - Remaining Acceptance Gaps

## Purpose

Complete the remaining Phase 6 acceptance work that is still explicitly unchecked in `plans/evidence/phase-6/collaboration-architecture-map.md`:

- App-owned real-time collaboration transport/composition
- Shared proposal executor integration for collaboration sessions
- Complete reconnect/session-shutdown acceptance scenarios
- Measured p95/p99 editor overhead evidence
- Phase ledger update to accepted after all gates pass

This plan assumes the current uncommitted Phase 6 local slice is the baseline: `devil-collaboration` exists, protocol DTOs exist, storage/observability/security/editor/UI scaffolding exists, evidence artifacts exist, and global gates including `cargo deny check` currently pass with warning-level duplicate dependency findings.

## Current Baseline

- `crates/devil-collaboration` provides deterministic in-memory operation-log behavior, convergence tests, duplicate/gap/conflict outcomes, metadata-only replay manifests, and presence handling.
- `crates/devil-protocol` has collaboration DTOs for sessions, participants, operations, acknowledgements, causal gaps, presence, shared proposal approval, replay manifest, transport envelope, and metadata-only audit.
- `crates/devil-app` does not yet depend on `devil-collaboration`; collaboration UI intents currently route to `AppCommandRequest::Noop`, and `AppComposition::shell_projection_snapshot()` emits an empty `collaboration_presence_projections` list.
- `ProposalExecutionRoute::for_payload()` still treats `ProposalTargetKind::CollaborationSession` as unsupported. This is correct until shared proposal gating is implemented.
- `devil-collaboration` has no explicit disconnect, reconnect resume, participant leave, closing, closed-session rejection, or shutdown-drain API.
- Editor performance tests currently report p95 in some CI tests, but there is no collaboration-specific baseline-vs-collaboration overhead test and no p99 evidence.
- Phase 6 evidence remains `Phase 6 acceptance: Not accepted.` and the phase ledger still marks Phase 6 future-gated.

## Implementation Strategy

Finish Phase 6 by adding an app-owned composition layer around the existing collaboration runtime rather than moving editor/workspace authority into `devil-collaboration` or UI. The collaboration runtime remains isolated and deterministic; app owns session orchestration, transport boundaries, proposal gating, editor application, persistence/audit linkage, and projection output.

Runtime activation must remain policy-gated. Durable file writes and non-user-generated edits must continue through existing proposal/workspace preconditions.

## Work Plan

### 1. App-Owned Collaboration Composition And Transport

Files likely touched:

- `crates/devil-app/Cargo.toml`
- `crates/devil-app/src/lib.rs`
- `crates/devil-app/tests/workspace_vfs_integration.rs`
- `plans/dependency-policy.md` if dependency-policy text needs to reflect active app composition
- `plans/evidence/phase-6/dependency-boundary.txt`
- `plans/evidence/phase-6/future-surface-deferral-audit.md`

Implementation steps:

1. Add `devil-collaboration` as a direct dependency of `devil-app`. Keep the dependency one-way: `devil-collaboration` must not depend on app, UI, editor, project, or workspace crates.
2. Add an app-owned `CollaborationComposition` or `CollaborationSessionRegistry` inside `devil-app`.
3. Store enabled `CollaborationSessionRuntime` instances keyed by `CollaborationSessionId`.
4. Add app-owned transport abstraction for Phase 6 acceptance. Keep it local/deterministic and protocol-envelope based. Do not add remote workspace, terminal/process, hosted telemetry, or arbitrary network authority.
5. Extend `AppCommandRequest` with explicit collaboration variants instead of routing collaboration UI intents to `Noop`:
   - `JoinCollaborationSession { session_id }`
   - `LeaveCollaborationSession { session_id }`
   - `PublishCollaborationPresence { session_id, participant_id }`
6. Update `CommandDispatcher::route_intent()` to return these app-owned requests.
7. Add `AppComposition` methods for session create/join, leave/close, publish presence, receive transport envelope, and expose projections.
8. Wire `AppComposition::shell_projection_snapshot()` to include current collaboration presence projections from app state.
9. Apply incoming collaboration document operations through existing editor authority using `EditorEngine::apply_protocol_edits()` with `TransactionSource::CollaborationParticipant`.
10. Persist and emit metadata-only collaboration audit/replay data after accepted app-owned operations.
11. Deny or no-op safely when runtime sessions are disabled by security policy or the workspace is untrusted.

Required tests:

- Command dispatcher routes collaboration intents to explicit app command requests, not `Noop`.
- Joining a session creates an app-owned runtime without granting UI/editor ownership.
- Publishing presence updates shell projections and does not mutate editor text.
- Receiving a document operation envelope applies through editor authority only after runtime/security preconditions pass.
- Untrusted workspace or disabled runtime policy denies session join and operation publish before side effects.
- Existing save, proposal, plugin, AI, LSP, and UI projection tests remain green.

Acceptance notes:

- This must be app-owned composition, not collaboration-owned workspace/editor mutation.
- Transport evidence can be deterministic/in-process for Phase 6, but it must use protocol envelopes and prove multi-participant exchange through the approved boundary.

### 2. Shared Proposal Executor Integration For Collaboration Sessions

Files likely touched:

- `crates/devil-app/src/lib.rs`
- `crates/devil-app/tests/workspace_vfs_integration.rs`
- `crates/devil-protocol/src/lib.rs` if additional requirement/status DTOs are needed
- `crates/devil-protocol/tests/dto_contracts.rs`
- `crates/devil-storage/src/lib.rs`
- `crates/devil-observability/src/lib.rs`
- `plans/evidence/phase-6/shared-proposal-approval-tests.txt`
- `plans/evidence/phase-6/storage-observability-redaction-audit.md`

Implementation steps:

1. Do not create a separate proposal executor inside `devil-collaboration`. Shared collaboration proposals should authorize existing app proposal execution paths.
2. Add an app-level shared approval registry keyed by `(CollaborationSessionId, ProposalId)`.
3. Store required approvers, received approvals/denials, capability decisions, participant roles, linked operation IDs, stale/superseded state, and metadata-only audit references.
4. If protocol needs to expose required approvers or aggregate state, add DTOs such as:
   - `CollaborationSharedProposalRequirement`
   - `CollaborationSharedProposalState`
   - `CollaborationSharedProposalProjection`
5. Update proposal route classification so proposals with a `CollaborationSession` target plus supported mutation targets are treated as shared-gated proposals, not unsupported direct collaboration execution.
6. Keep pure unknown collaboration mutation targets unsupported unless the executor can delegate to an existing safe route.
7. Before applying a shared proposal, require the app approval registry to prove quorum/required approvals and no denial or stale state.
8. After approval, reuse existing proposal apply paths for text edit, workspace edit, save, plugin/AI/LSP-lowered proposals, and batch proposals. Do not bypass existing preconditions.
9. Link successful application to collaboration metadata:
   - `CollaborationSharedProposalApproval.applied_operation_ids`
   - `CollaborationAuditRecord.proposal_id`
   - existing proposal audit/event metadata
10. Preserve audit-before-success behavior. If collaboration audit storage/emission fails, proposal apply must fail closed or roll back according to existing proposal rollback rules.

Required tests:

- Shared proposal apply is denied without required approvals.
- Approval from an unauthorized participant is denied before side effects.
- Explicit denial blocks apply and preserves editor/workspace state.
- Stale/superseded approval does not count toward quorum.
- Approved shared proposal reuses existing executor and produces normal applied proposal lifecycle metadata.
- Dirty-buffer/file preconditions still reject stale saves or edits with dirty text preserved.
- Collaboration approval/audit records remain metadata-only and reject raw source/transcript markers.
- Existing unsupported collaboration targets remain fail-closed when no accepted executor route exists.

Acceptance notes:

- Collaboration approval authorizes app-owned proposal execution; it must not replay operation-log text independently.
- This is the highest-risk remaining item because it touches proposal routing and rollback invariants.

### 3. Reconnect, Disconnect, Leave, And Session Shutdown Semantics

Files likely touched:

- `crates/devil-collaboration/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`
- `crates/devil-protocol/tests/dto_contracts.rs`
- `crates/devil-app/src/lib.rs`
- `crates/devil-app/tests/workspace_vfs_integration.rs`
- `plans/evidence/phase-6/disconnect-reconnect-replay-tests.txt`

Implementation steps:

1. Add explicit metadata DTOs if needed:
   - `CollaborationDisconnectNotice`
   - `CollaborationReconnectRequest`
   - `CollaborationReconnectResponse`
   - `CollaborationSessionShutdownNotice`
   - `CollaborationParticipantLifecycleEvent`
2. Include required metadata on these DTOs: session ID, participant ID, document epoch or version vector, correlation ID, causality ID, event sequence where persisted, retention label, redaction hints, schema version, and reason codes.
3. Add runtime lifecycle APIs:
   - `session_state()`
   - `disconnect_participant(...)`
   - `begin_reconnect(...)`
   - `complete_reconnect(...)`
   - `leave_participant(...)`
   - `begin_shutdown(...)`
   - `finish_shutdown(...)`
4. Enforce state behavior:
   - `Active` accepts validated operations and presence.
   - `Reconnecting` accepts metadata/presence/replay requests but rejects or resyncs new text operations.
   - `Closing` drains existing accepted operations and rejects new mutation.
   - `Closed` rejects operation/presence mutation and transport envelopes.
   - `Denied` remains fail-closed.
5. Reconnect resume must use version vectors and replay manifests, not raw source reconstruction.
6. Presence projections should mark reconnecting participants with `reconnecting = true`.
7. App composition must handle lifecycle transport envelopes and update projections/audit metadata.
8. Session shutdown must preserve local editor text and normal save/proposal flows.

Required tests:

- Participant disconnect marks presence reconnecting and preserves text.
- Reconnecting session rejects new text operations but accepts metadata/presence.
- Reconnect resume emits metadata-only replay manifest with correct operation IDs and gap counts.
- Out-of-order operation after reconnect requires resync before accepting sequence.
- Closing session drains accepted operations and rejects new operations.
- Closed session rejects operations, presence updates, and transport envelopes.
- App-owned transport handles disconnect/reconnect/shutdown envelopes without UI/editor ownership leakage.
- Metadata replay identifies correct resync point and does not persist raw source.

Acceptance notes:

- Runtime-only reconnect tests are insufficient; app-owned transport/composition tests must cover these paths too.

Implementation steps:

6. Print numeric evidence in test output and archive the exact command output in `performance-budget-tests.txt`.

Required tests/evidence:

- `ci_collaboration_edit_overhead_p95_p99_vs_user_baseline`

## Non-Negotiable Invariants

- UI remains projection-only and must not own editor text, app state, storage, project, workspace, or collaboration runtime internals.
- `devil-collaboration` must not depend on app, UI, editor, project, workspace, remote, terminal, or process internals.
- App composition owns runtime session orchestration and transport boundaries.
- Durable file writes remain proposal-mediated through existing save/workspace preconditions.
- Collaboration must not activate remote workspace authority, terminal/process execution, hosted telemetry, cloud providers, or raw source retention.
- Metadata-only audit/replay defaults must reject raw source, raw transcripts, secrets, and unbounded payload markers.
- Existing plugin, AI, LSP, save, batch, workspace edit, rollback, and UI projection regressions must remain green.

## Main Risks And Mitigations

| Risk | Mitigation |
| --- | --- |
| Proposal route regression | Add shared-collaboration route as a gating wrapper around existing executors, not a new executor. Preserve existing precondition checks. |
| Dirty-buffer clobbering | Apply remote operations only through editor authority after operation/runtime/security preconditions pass; reuse existing stale/conflict tests. |
| Runtime ownership leak | Keep session registry in app; keep collaboration runtime isolated; keep UI as projections/intents only. |
| Performance flakiness | Measure synchronous collaboration overhead separately from total edit latency; use deterministic sample sizes and evidence output. |
| Reconnect raw-source leak | Use version vectors, operation IDs, byte counts, hashes, and metadata-only replay manifests; never persist full source/transcripts. |
| Premature acceptance | Keep Phase 6 not accepted until every unchecked blocker is implemented, evidenced, and `xtask check-deps` passes in accepted mode. |

## Definition Of Done

Phase 6 can be marked accepted only when:

- App-owned collaboration composition routes session lifecycle, presence, document operations, and transport envelopes through app authority.
- Shared collaboration proposals require authorized approvals before existing proposal executors apply changes.
- Disconnect, reconnect, participant leave, closing, closed-session rejection, and shutdown scenarios are implemented and tested in runtime and app composition.
- Editor performance evidence shows collaboration synchronous overhead meets p95 <= 2 ms and p99 <= 5 ms acceptance thresholds.
- All Phase 6 evidence artifacts are current and the architecture map has no unchecked final checklist items.
- `plans/phase-status-ledger.md` marks Phase 6 accepted only after accepted Phase 6 evidence passes `xtask`.
- All global gates pass, including `cargo deny check`.

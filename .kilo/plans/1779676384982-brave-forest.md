Scope: Activate Phase 3 semantic fabric and LSP supervision only. Do not activate Phase 4+ AI, plugins, terminal, collaboration, remote development, vector indexing, embeddings, or model-backed retrieval.

- LSP remains DTO/governance-only. There is no accepted supervised runtime process model, JSON-RPC worker loop, document sync, request queue, timeout/circuit-breaker implementation, or runtime evidence.
- Tree-sitter parsing is not implemented; current parsing is lexical fallback.
- Changed-range incremental indexing exists structurally but still needs real incremental refresh semantics.
- Durable semantic metadata contracts exist, but Phase 3 must still prove end-to-end persistence, tombstone, freshness, and privacy behavior before acceptance.

## 2. Acceptance Objective

Phase 3 is accepted only when the repository proves all of the following:

- Semantic indexing is actor-owned, bounded, cancellable, freshness-aware, privacy-scoped, and non-blocking.
- Workspace discovery remains workspace-authored and crosses into indexing only through protocol DTOs.
- Source indexing is descriptor-first and uses bounded chunk leases or changed ranges instead of unbounded full-source ownership.
- Lexical maps, tree-sitter-derived records, normalized graph records, semantic queries, and metadata persistence preserve freshness and privacy metadata.
- LSP operations are supervised, bounded, cancellable, timeout-aware, stale-response-safe, and DTO-only.
- LSP edit-producing outputs become `WorkspaceProposal` or `WorkspaceEditProposalPayload` values and never mutate editor buffers or workspace files directly.
- Existing proposal-mediated save conflict, dirty-buffer preservation, workspace trust, observability ID, and projection-only UI invariants remain green.
- `plans/evidence/phase-3/predictive-semantic-fabric.md` has no unchecked final checklist items, all required artifacts exist, and `xtask` passes with Phase 3 and LSP acceptance markers changed only after evidence is complete.

## 3. Non-Negotiable Guardrails

- Keep `devil-index` isolated to approved internal dependencies: `devil-protocol`, `devil-storage` if needed, and `devil-text`.
- Do not make `devil-index` depend on `devil-project`, `devil-editor`, `devil-app`, `devil-ui`, `devil-ai`, provider crates, vector storage, terminal, plugin, collaboration, or remote crates.
- Keep UI projection-only. UI may render semantic/LSP snapshots and dispatch intents, but never own editor sessions, index state, workspace state, server processes, or proposal application.
- Keep all writes proposal-mediated. Semantic and LSP workers may produce metadata, diagnostics, queries, or proposals only.
- Keep saves fail-closed through the existing proposal/workspace path with expected fingerprint, file content version, workspace generation, buffer version, snapshot id, capability, principal, correlation, and causality.
- Keep semantic and LSP work unable to block editor input, viewport projection, proposal validation, or save workflows.
- Keep observability and storage metadata-only by default; do not persist full source snapshots, chunk payloads, syntax trees with source, provider output, embeddings, vectors, or unbounded previews.
- Keep vector indexing, embeddings, vector databases, model-backed retrieval, AI provider routing, terminal execution, plugins, collaboration, and remote development inactive.

## 5. Work Package 1: Contract And Service Boundaries

Goal: Make semantic and LSP runtime integration callable without adding hidden ownership or direct crate coupling.

Implementation tasks:

1. Decide whether Phase 3 needs a protocol-level `SemanticPort` or `IndexPort` trait, because semantic DTOs exist but there is no service boundary equivalent to workspace, storage, or proposal ports.
2. Expose workspace-authored discovery through an approved port or app adapter instead of direct cross-crate ownership. Existing `WorkspaceActor::semantic_discovery_snapshot()` and `WorkspaceActor::semantic_discovery_delta_from_watcher_events()` are the starting point.
3. Add or extend protocol DTO contract tests for semantic scheduling, source descriptor references, metadata persistence, query responses, LSP request/response DTOs, and LSP edit proposal conversion.
4. Update `plans/dependency-policy.md` and `xtask` protocol-symbol checks together if new literal required protocol symbols are introduced.
5. Preserve current proposal DTOs for mutation outputs: `WorkspaceEditProposalPayload`, `WorkspaceTextEdit`, `WorkspaceFileOperation`, `WorkspaceEditSourceKind`, and `ProposalPayload::WorkspaceEdit`.

Exit criteria:

- Runtime integration has explicit ports or adapters instead of ad hoc calls.
- Protocol tests cover all new or revised DTOs.
- `cargo run -p xtask -- check-deps` remains green.

## 6. Work Package 2: Semantic Actor Hardening

Goal: Make `IndexingActor` acceptance-worthy rather than scaffold-only.

Implementation tasks:

1. Verify and harden bounded queue admission outcomes: accepted, coalesced, cancelled, rejected, degraded, and resync-required.
2. Ensure priority policy favors current open-buffer edits, visible queries, save-adjacent invalidations, and direct user requests over background discovery and low-priority refresh.
3. Ensure every work item carries cancellation identity tied to workspace generation, file identity, file content version, snapshot id, content hash, privacy scope, grammar version, schema version, and causality.
4. Add stale-result suppression for queued, in-flight, parser, LSP-fusion, graph, ranking, and query-refresh work.
5. Emit metadata-only scheduling diagnostics and health snapshots for queue pressure, cancellation, stale results, degradation, and resync requirements.
6. Add stress tests proving index pressure does not block editor input, viewport projection, proposal validation, or save workflows.

Exit criteria:

- Actor tests cover priority ordering, queue saturation, cancellation, supersession, stale result suppression, and resync behavior.
- Save and editor regressions remain green while indexing work is queued or cancelled.

## 7. Work Package 3: Workspace Discovery Integration

Goal: Keep workspace discovery authority in `devil-project` while allowing `devil-index` to consume discovery DTOs.

Implementation tasks:

1. Route `WorkspaceDiscoverySnapshot` and `WorkspaceDiscoveryDelta` through the selected service boundary.
2. Ensure discovery records include workspace id, workspace generation, file identity, canonical path, language hint, trust result, privacy scope, policy decision, skip reason, file fingerprint, content hash when allowed, and change kind.
3. Keep ignored, hidden, generated, binary, vendored, oversized, external, deleted, untrusted, and privacy-restricted paths excluded or metadata-only according to workspace policy.
4. Ensure `RepositoryDiscoveryImporter` schedules content work only for `ContentAllowed` records with valid semantic identity.
5. Tombstone or invalidate records before replacements become query-authoritative after deletion, external overwrite, privacy downgrade, path denial, or workspace generation changes.

Exit criteria:

- Tests prove `devil-index` does not scan disk or mint workspace file identity in production flows.
- Tests cover allowed, metadata-only, excluded, deleted, ignored, denied, external, oversized, binary, generated, vendored, and untrusted discovery records.
- Evidence artifact `repository-discovery-ignore-fingerprint.md` can be produced from real tests and code paths.

## 8. Work Package 4: Descriptor-First And Incremental Source Inputs

Goal: Make source ingestion safe for large files and live editing.

Implementation tasks:

1. Treat descriptor-only, leased chunk, changed-range, and bounded full-text inputs as explicit modes with distinct freshness and degradation states.
2. Implement real changed-range refresh behavior for affected lexical records and graph records where possible.
3. Use `devil-text` snapshot descriptors and chunk lease descriptors for bounded reads; do not require `TextSnapshot::try_full_text()` for large or policy-restricted inputs.
4. Keep bounded full-text as a small-file optimization or fixture path with explicit byte budget, policy reason, and degraded metadata where applicable.
5. Add cancellation checks between chunk batches and changed-range refresh steps.
6. Ensure no durable storage, observability event, debug evidence, or query response persists raw source or chunk payloads by default.

Exit criteria:

- Large-file tests prove indexing does not require full text materialization.
- Lease-based chunk tests produce expected lexical metadata without source retention.
- Changed-range tests prove partial refresh, stale unaffected records, and cancellation behavior.
- Evidence artifact `lexical-symbol-map-tests.txt` can cite these tests.

## 9. Work Package 5: Syntax Worker And Tree-Sitter Layer

Goal: Replace lexical-only parsing with a supervised syntax worker layer while preserving lexical fallback.

Implementation tasks:

1. Introduce tree-sitter parser worker integration behind the existing `ParserWorker` abstraction or a narrowly scoped extension of it.
2. Choose the first supported grammar set deliberately. Add external parser dependencies only after dependency, license, and supply-chain review.
3. Keep lexical fallback available for unsupported languages, parser failures, timeouts, oversized files, and privacy-restricted inputs.
4. Split reusable content-addressed parser artifacts from file-specific semantic extraction where practical.
5. Keep syntax caches keyed or freshness-validated by content hash, grammar version, language, parser options, schema version, file identity, snapshot id, file content version, workspace generation, model metadata version, and privacy scope.
6. Add parser timeout, crash, malformed tree, grammar upgrade, schema upgrade, and privacy downgrade tests.

Exit criteria:

- Tree-sitter tests prove cache reuse only under valid content/grammar/freshness boundaries.
- Lexical fallback remains deterministic and non-blocking when tree-sitter is unavailable.
- Evidence artifact `tree-sitter-cache-tests.txt` can be produced.

## 10. Work Package 6: Normalized Graph Records And Queries

Exit criteria:

- Query tests cover freshness policy, privacy denial, stale cache rejection, degraded fallback, pagination, and proposal-ready preview output.

## 11. Work Package 7: Metadata-Only Semantic Persistence

Goal: Persist semantic metadata without violating source retention or privacy constraints.

Implementation tasks:

1. Wire semantic index outputs to `SemanticMetadataRepository` through approved storage ports or app-owned orchestration.
2. Save `SemanticMetadataBatch` records with namespace, workspace, file, language, content hash, disk fingerprint reference, file content version, workspace generation, snapshot id, grammar version, model metadata version, extraction version, schema version, privacy scope, provenance, freshness, symbol hashes, bounded labels, graph endpoints, redaction hints, and tombstones.
3. Reject full source text, chunk payloads, syntax tree source, embeddings, vectors, provider outputs, model summaries, proposal edit bodies, and replay source snapshots by default.
4. Tombstone or quarantine records on privacy downgrade, file identity replacement, path denial, deletion, workspace generation replacement, schema upgrade, content hash mismatch, or corrupt persisted metadata.
5. Ensure semantic reads respect freshness policy and never report fresh across invalid version or privacy boundaries.

Exit criteria:

- `devil-storage` tests cover save, read, tombstone, corrupt persistence, schema mismatch, privacy downgrade, and source-payload rejection.
- `devil-index` or integration tests prove persisted metadata can refill query state without source bodies.
- Evidence artifact `privacy-redaction-audit.md` can cite storage and query behavior.

## 12. Work Package 8: LSP Runtime Supervision

## 17. Parallelization Rules

Safe to parallelize:

- Protocol DTO contract tests and semantic storage tests after boundary names are stable.
- Descriptor-first source input tests and semantic actor scheduling tests.
- Evidence artifact drafting while implementation tests are running, as long as acceptance status remains not accepted.
- LSP DTO conversion tests before process supervision, as long as runtime activation remains disabled.

Do not parallelize:

- LSP runtime activation before proposal routing and supervisor contracts are accepted.
- Tree-sitter dependency additions before dependency/license review and cache freshness tests are designed.
- Phase 3 acceptance marker changes before all required artifacts and full validation outputs exist.
- AI, plugin, terminal, collaboration, remote, vector, or model-provider runtime work with Phase 3 implementation.

## 18. Stop Conditions

Stop Phase 3 acceptance work if any of these occur:

- `devil-index` gains direct dependency on workspace, editor, app, UI, provider, vector, terminal, plugin, collaboration, or remote crates.
- Any semantic or LSP worker mutates an editor buffer, writes workspace files, executes commands, or applies proposals directly.
- UI code starts owning editor sessions, index state, LSP workers, workspace state, or proposal lifecycle state.
- Saves wait for semantic freshness, LSP diagnostics, formatting, server health, document sync, or graph queries.
- Queries report `Fresh` across mismatched content hash, grammar version, model version, schema version, file identity, workspace generation, snapshot id, or privacy scope.
- Storage, observability, tests, or evidence persist raw source, chunk payloads, syntax trees with source, embeddings, vectors, provider output, secrets, or unbounded previews by default.
- LSP launches in untrusted or policy-denied workspaces.
- Command-like code actions execute before a separate command execution surface is accepted.
- `xtask` fails after acceptance markers are changed.

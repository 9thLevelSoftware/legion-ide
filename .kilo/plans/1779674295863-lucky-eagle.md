The current accepted baseline is: lifecycle state, deny-by-default validation, generic save-file apply, open-buffer text edit apply, closed-file create/delete/rename apply, single-file workspace-edit delegation, audit-before-success rollback, app-owned lifecycle recovery snapshots, live proposal ledger projection, batch preflight/contracts, and workspace-authorized rollback checkpoints.

## Completion Definition

Phase 2 can be marked accepted only after the repository can prove all of the following:

- Every supported mutation path enters as `WorkspaceProposal` and transitions through `ProposalRequest` to `ProposalResponse`.
- Runtime batch execution exists for the accepted item routes and either honors declared atomicity or emits exact rollback and partial-failure metadata.
- Multi-file and multi-edit workspace edits execute only through editor/workspace authorities with prepare-all-before-mutate preflight.
- Audit-before-success is enforced for every successful mutation response, including batch and workspace-edit routes.
- Rollback uses editor-owned undo for open buffers and workspace-authorized rollback checkpoints for disk-backed mutations.
- Dirty editor text is preserved on rejected, denied, stale, conflicted, failed, cancelled, and rolled-back outcomes.
- Raw `FormatFileProposal` remains denied unless it is first lowered to concrete edit material, because the DTO has file/snapshot/options but no formatter result edits.
- `CodeActionProposal` edit-only payloads can execute only by lowering to the same text/workspace-edit pipeline; command-like code actions remain future-gated.
- Terminal, plugin, remote, collaboration, AI, agent, and LSP runtime apply paths remain denied until their own ADR, dependency-policy, protocol, ownership-test, and evidence gates exist.
- `plans/evidence/phase-2/proposal-mutation-substrate.md` records accepted routes, rejected routes, validation outputs, and the final Phase 2 acceptance rationale.
- `plans/phase-status-ledger.md` is updated from partially accepted to accepted only after all implementation and validation evidence is complete.

## Non-Goals

- Do not activate Phase 3, semantic runtime acceptance, or LSP supervision.
- Do not activate AI, plugin, terminal, remote, collaboration, tracker, memory, or agent runtime behavior.
- Do not add a new proposal-service crate unless an ADR and dependency-policy update are completed in the same change.
- Do not make `devil-ui` own proposal lifecycle, editor text, workspace state, or apply authority.
- Do not make `devil-editor` depend on `devil-project`.
- Do not persist raw source, prompts, secrets, terminal output, provider payloads, or unbounded previews in audit or observability records.

## Architecture Direction

Keep generalized execution in the app domain for Phase 2 completion. `AppProposalCoordinator` should remain the lifecycle and validation coordinator. `AppComposition` should remain the owner of editor/workspace authority wiring. Add a small internal execution layer in `devil-app` rather than spreading batch and multi-edit logic across route-specific helpers.

Recommended internal types:

- `ProposalExecutionService`
- `ProposalExecutionPlan`
- `ProposalExecutionItem`
- `ProposalExecutionJournal`
- `ProposalMutationRoute`
- `ProposalMutationCheckpoint`
- `ProposalItemOutcome`
- `ProposalRollbackOutcome`

These can start as private `devil-app` types. Add protocol DTOs only if projection or cross-crate contract stability requires them. If protocol symbols are added, update `plans/dependency-policy.md` and keep `cargo run -p xtask -- check-deps` green.

`ProposalPort::handle(ProposalRequest::Apply(_))` currently lacks editor/workspace authorities. Keep stateless apply fail-closed unless the implementation introduces an explicit authority-bearing execution context. The authoritative apply path should remain `AppComposition::handle_proposal_request()`.

- Add explicit tests that unsupported future runtime routes still do not mutate: terminal, plugin, remote, collaboration, LSP, AI, nested batch, raw format, and command-like actions.

Acceptance checks:

- Unsupported-route tests fail before side effects and return structured diagnostics.
- No dependency-policy changes are needed for this package unless protocol symbols are introduced.

## Work Package 1: Common Execution Pipeline

Goal: make apply behavior share one prepare/preflight/mutate/audit/finalize/rollback structure.

Implementation tasks:

- Introduce `ProposalExecutionService` or equivalent private app-domain module in `crates/devil-app/src/lib.rs` or a sibling module if the file is split.
- Move shared apply sequencing out of individual helpers and into a single pipeline: load lifecycle context, verify state, prepare current versions, preflight route support, capture rollback checkpoints, mutate, audit, finalize, rollback on audit failure.
- Keep existing single-route helpers as route executors initially to minimize risk.
- Ensure every execution item carries proposal id, item id when present, target coverage, required capability, correlation id, causality id, preconditions, and rollback availability.
- Ensure failed prepare/preflight never mutates editor or disk.
- Ensure audit failure after mutation triggers rollback before any success response is returned.

Acceptance checks:

- `AppComposition::apply_workspace_proposal()` becomes a thin lifecycle wrapper around the execution service.
- Manual save and generic save proposal execution remain equivalent.
- Stateless coordinator apply remains rejected or is explicitly wired through authority-bearing app execution.
- Tests prove invalid lifecycle state cannot self-approve or apply.

## Work Package 2: Route Normalization

Goal: convert every proposal payload into accepted mutation primitives or typed fail-closed diagnostics.

Implementation tasks:

- Normalize `TextEdit`, `CreateFile`, `DeleteFile`, `RenameFile`, and `SaveFile` into existing accepted route primitives.
- Normalize `WorkspaceEdit` into ordered file text edits and file operations.
- Normalize edit-only `CodeActionProposal` into a text/workspace-edit plan only when target coverage and version preconditions are complete.
- Reject raw `FormatFileProposal` with a stable diagnostic such as `proposal.format_requires_lowered_workspace_edit` until a future formatter/LSP gate produces concrete edit material.
- Reject terminal, plugin, remote, collaboration, metadata-smuggling, unsupported, and mixed future runtime routes before mutation.
- Keep total target traversal and target coverage deterministic for all payloads and nested batch items.

Acceptance checks:

- Unit tests cover every `ProposalPayload` variant.
- `CodeActionProposal` with concrete edits routes through the same executor as text/workspace edits.
- Raw `FormatFileProposal` and command-like routes are denied without side effects.
- No payload visitor panics on non-save, multi-file, metadata-only, or unsupported targets.

## Work Package 3: Multi-Edit Workspace Edit Execution

Goal: support proposal-ready workspace edits containing multiple file edits and file operations without bypassing editor or workspace authority.

Implementation tasks:

- Extend `apply_workspace_edit_proposal()` beyond the current single-file-operation delegation.
- Preflight every `WorkspaceTextEdit` and `WorkspaceFileOperation` before the first mutation.
- For open-buffer text edits, require buffer id when known, buffer version, snapshot id, byte-range validation, and editor transaction rollback metadata.
- For closed-file text edits, either route through a workspace-authorized safe edit API or deny until a closed-file text-edit authority exists; do not read/write files directly in app code.
- For create operations, support only material that can be supplied through accepted proposal payloads and policy; deny hash-only non-empty creates unless an approved content materialization path exists.
- For delete and rename operations, reuse `WorkspaceActor::{delete,rename}_file_with_proposal()` and workspace rollback checkpoints.
- Capture per-target rollback checkpoints before mutation and record them in the execution journal.
- Apply supported items in deterministic order and roll back in reverse order if any committed item must be compensated.

Acceptance checks:

- Multi-file workspace edit succeeds when all targets are valid and supported.
- Stale buffer version or snapshot id rejects before any mutation.
- Path escape, untrusted workspace, missing capability, and mismatched workspace generation reject before any mutation.
- Audit failure after a mixed workspace edit rolls back all committed supported items or records rollback-failure diagnostics without clobbering external changes.
- Dirty open buffers are preserved on every rejected or failed path.

## Work Package 4: Runtime Batch Execution

Goal: turn the existing batch preflight and contract model into safe runtime execution for accepted item routes.

Implementation tasks:

- Reuse `preflight_batch_proposal()` and `plan_batch_execution_contract()` as the mandatory prepare stage for batch apply.
- Support initial batch item routes that already have accepted executors: `TextEdit`, `CreateFile`, `DeleteFile`, `RenameFile`, and workspace edits that lower entirely into supported primitives.
- Keep batch `SaveFile`, raw `FormatFile`, terminal, plugin, remote, collaboration, AI, LSP, nested batch, and unsupported routes denied unless a later package proves exact semantics.
- Topologically order items by dependency edges and deterministic item order.
- For `ProposalBatchAtomicity::PrepareAllBeforeMutate`, preflight all items before mutation and stop before mutation if any item is invalid.
- For `ProposalBatchAtomicity::AllOrNothing`, require rollback proof for every mutating item and roll back every committed item on mutation, commit, or audit failure.
- For `ProposalBatchAtomicity::OrderedNonAtomic`, allow deterministic partial completion only when every failed or skipped item gets a `ProposalPartialFailureRecord` and every committed item is audited or compensated according to policy.
- Honor `ProposalBatchRollbackPolicy::Required`, `BestEffort`, `NotSupported`, and `NotRequired` exactly; reject combinations that cannot be proven before mutation.
- Record item states in the execution journal: prepared, preflight_failed, skipped_dependency, mutated, committed, audit_persisted, rolled_back, rollback_failed, finalized.
- Make batch apply impossible unless the proposal is in a legal lifecycle state and cannot self-approve from Created.

Acceptance checks:

- Atomic batch success mutates all intended supported targets and audits before success.
- Atomic batch mid-item failure rolls back already committed items in reverse order.
- Rollback refusal caused by external changes produces typed failure and does not clobber the external change.
- Ordered non-atomic batch emits exact partial-failure records and never hides a failed item behind success metadata.
- Dependency-blocked items are recorded as not started or skipped without mutation.
- Batch audit failure triggers rollback or exact failed audit metadata before returning.

## Work Package 5: Audit, Ledger, And Recovery Integration

Goal: make generalized execution visible and recoverable without storing raw source.

Implementation tasks:

- Reuse `SaveWorkflowService::observe_proposal_response()` semantics or extract a shared audit helper for all proposal responses.
- Persist metadata-only audit records before successful apply or rollback responses.
- Ensure observability rejects zero `CorrelationId`, nil `CausalityId`, and zero `EventSequence` in new events.
- Extend the app-owned execution journal enough for lifecycle recovery snapshots to reconstruct in-flight or failed batch/workspace-edit state.
- Surface batch/workspace-edit state in `ProposalLedgerProjection` as bounded summaries, diagnostics, target coverage, rollback availability, and redaction hints.
- Avoid protocol DTO expansion unless UI/projection contracts need stable per-item status; if expanded, add DTO contract tests.

Acceptance checks:

- Ledger projection remains metadata-only and does not expose raw source, full diffs, prompts, terminal commands, or provider payloads.
- Recovery snapshot preserves proposal envelopes, lifecycle states, and safe execution-journal summaries.
- Audit sink failure changes success into typed failure and triggers rollback for already mutated items.
- Event ordering remains compatible with existing save/proposal expectations.

## Work Package 6: Format And Code-Action Boundary Closure

Goal: close the Phase 2 ambiguity around format/code-action without prematurely activating LSP or formatter runtimes.

Implementation tasks:

- Treat raw `FormatFileProposal` as a request descriptor, not executable edit material, and keep apply denied with stable diagnostics.
- Document that future formatting must produce `WorkspaceEdit` or `TextEdit` proposal payloads after Phase 3/LSP gates, then execute through the Phase 2 executor.
- Execute `CodeActionProposal` only when it contains concrete text edits, target coverage is complete, and preconditions match the current editor/workspace state.
- Deny command-like code actions and mixed command/edit actions until a separate command execution surface is accepted.
- Add evidence explaining that Phase 2 accepts the proposal mutation substrate and lowered edit execution, not raw formatter or command runtime activation.

Acceptance checks:

- Code-action edit payload applies through editor/workspace proposal execution and rolls back on audit failure.
- Code-action stale snapshot or missing preconditions rejects before mutation.
- Raw format proposal rejects without mutation and with a diagnostic that points to lowered workspace-edit requirements.
- Phase 3/LSP status remains not accepted.

## Dependency And Policy Constraints

Preserve these constraints throughout implementation:

- `devil-editor` must not depend on `devil-project`.
- `devil-ui` must remain projection-only and must not depend on `devil-app`, `devil-editor`, `devil-project`, or `devil-storage`.
- `devil-protocol` remains the cross-domain DTO and port boundary.
- Placeholder crates remain inert unless activation gates land in the same change.
- `devil-index` Phase 3 dependency limits remain unchanged and Phase 3 stays not accepted.
- Boundary DTO changes require matching policy and contract-test updates because `xtask check-deps` validates protocol symbol presence literally.

## Stop Conditions

Stop implementation and narrow the package if any of these occur:

- Any non-user-direct mutation path writes editor or workspace state without a `WorkspaceProposal` lifecycle.
- Any workspace write bypasses expected fingerprint, file content version, workspace generation, buffer version, snapshot id, principal, capability, correlation, or causality where applicable.
- Any batch path mutates before all required preflight checks for its declared atomicity have passed.
- Any rollback path directly manipulates disk from app code instead of using workspace authority.
- Any rejected, denied, stale, conflicted, failed, cancelled, or rolled-back outcome loses dirty editor text.
- Any audit record stores raw source, full diffs, secrets, terminal output, prompts, or provider payloads by default.
- Any future runtime route starts processes, writes files, mutates buffers, launches LSP, calls providers, or activates placeholder crates.
- Any documentation marks Phase 3, LSP supervision, AI, plugins, terminal, remote, collaboration, or agents accepted as a side effect of Phase 2 completion.

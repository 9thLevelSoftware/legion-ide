# Plan 05-03: Proposal Lifecycle App Routing And Details Population

## Objective

Route proposal lifecycle controls through app-owned proposal authority and populate usable selected-proposal trust details in shell snapshots.

<execution_contract>

- read-before-write: read `.planning/phases/05-control-trust-and-assisted-ai-surfaces/05-CONTEXT.md`, `05-01-RESULT.md`, `05-02-RESULT.md`, `crates/devil-app/src/lib.rs`, `crates/devil-app/tests/workspace_vfs_integration.rs`, `crates/devil-protocol/src/lib.rs`, and `crates/devil-ui/src/ui.rs` before editing.
- evidence-before-action: prove where proposal intents currently become `Noop` and where `handle_proposal_request` already owns lifecycle transitions before modifying dispatch.
- minimal diff: modify app routing/projection helpers and focused app integration tests only.
- verify-before-report: run every frontmatter command and record outcomes in `05-03-RESULT.md`.
- role: app authority implementer with lifecycle/security test coverage.
- task: make preview/approve/reject/apply/rollback/cancel/details usable from UI intents without moving proposal authority into UI or desktop.
- scope: app command dispatch, proposal coordinator helpers, shell projection population, and focused app integration tests.
- allowed tools/actions: focused Rust edits, app integration tests, workspace rollback regression tests, cargo check.
- forbidden actions: UI/desktop rendering changes, protocol contract redesign, editor/workspace direct mutation bypasses, storage schema changes, dependency changes.
- verification criteria: proposal lifecycle commands update app-owned lifecycle/projections, application still uses proposal-mediated mutation, and selected proposal details show lifecycle/risk/privacy/approval/rollback metadata.
- final result format: write `.planning/phases/05-control-trust-and-assisted-ai-surfaces/05-03-RESULT.md` with changed files, command outcomes, and any `BLOCKED` items.

</execution_contract>

<stop_gates>

- BLOCKED if proposal lifecycle commands cannot be routed without bypassing `AppProposalCoordinator` or `AppComposition::handle_proposal_request`.
- BLOCKED if app dispatch would need UI or desktop to provide an owned `WorkspaceProposal` instead of selecting an app-owned proposal id.
- BLOCKED if apply/rollback would bypass existing workspace proposal preconditions, audit ordering, conflict/stale handling, or dirty text preservation.
- BLOCKED if selected proposal details require raw source or unredacted provider output in shell projections.

</stop_gates>

<recovery>

- If generic `CommandDispatcher::route_intent` still maps proposal intents to `Noop`, add an app-level proposal-intent branch in `AppComposition::dispatch_ui_intent` before generic dispatch.
- If a selected proposal is missing, return a typed app error and surface it as a status/error outcome; do not synthesize success.
- If applying or rolling back a proposal fails, keep the lifecycle/projection in a failed/rejected state and preserve editor/workspace authority.
- On interruption, inspect `git diff`, compare modified files against `files_modified` and `files_forbidden`, rerun verification, and continue only from unverified work.

</recovery>

## Task 1: Route Proposal Lifecycle Intents Through App Authority

### Read Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-ui/src/ui.rs`
- `crates/devil-app/tests/workspace_vfs_integration.rs`

### Write Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-app/tests/control_trust_surfaces.rs`

### Implementation Sequence

1. Add or expose an app-owned lookup helper on `AppProposalCoordinator` for a cloned `WorkspaceProposal` by `ProposalId`; do not expose mutable proposal maps.
   > verification: `rg -q "proposal_for_id" crates/devil-app/src/lib.rs`
2. In `AppComposition::dispatch_ui_intent`, detect `PreviewProposal`, `ApproveProposal`, `RejectProposal`, `ApplyProposal`, `RollbackProposal`, `CancelProposal`, and `OpenProposalDetails` before calling generic `CommandDispatcher::route_intent`.
   > verification: `rg -q "route_proposal_intent" crates/devil-app/src/lib.rs`
3. Convert each detected intent with `CommandDispatcher::route_proposal_intent`, call `handle_proposal_request` for lifecycle requests, and add exact app outcomes `AppCommandOutcome::ProposalLifecycleUpdated(ProposalResponse)` and `AppCommandOutcome::ProposalDetailsOpened(ProposalId)` for workflow/status consumers.
   > verification: `cargo test -p devil-app --test control_trust_surfaces proposal_lifecycle -- --nocapture`

### Edge And Error Cases

- Missing or mismatched proposal ids must return typed errors, not `Noop`.
- `OpenProposalDetails` must select/show metadata only and must not mutate lifecycle state.
- `ApplyProposal` must continue using `apply_workspace_proposal` and existing preconditions.

<verify>

```powershell
cargo test -p devil-app --test control_trust_surfaces proposal_lifecycle -- --nocapture
```

</verify>

### BLOCKED Conditions

- A lifecycle action cannot be routed without giving UI/desktop an owned proposal payload, or the exact `AppCommandOutcome` variants above cannot be added without breaking existing callers.

## Task 2: Populate Selected Proposal Trust Details

### Read Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`
- `05-02-RESULT.md`

### Write Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-app/tests/control_trust_surfaces.rs`

### Implementation Sequence

1. Update `shell_projection_snapshot` or a narrow projection helper to derive selected proposal context, privacy, permission budget, approval checklist, and checkpoint/rollback projections from the selected/current proposal when available.
   > verification: `rg -q "approval_checklist_from_trust_projections" crates/devil-app/src/lib.rs`

### Edge And Error Cases

- Empty proposal ledger must keep empty/default trust projections.
- Failed apply or rollback must keep failure status visible rather than resetting to created.
- Privacy denied or redacted target coverage must produce denial/redaction counts.

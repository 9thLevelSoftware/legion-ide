# Plan 05-05: Desktop Control Trust Panels And Actions

## Objective

Make the Phase 5 control, trust, and assisted-AI projections usable in the desktop adapter while keeping desktop projection-only.

<execution_contract>

- read-before-write: read `.planning/phases/05-control-trust-and-assisted-ai-surfaces/05-CONTEXT.md`, `05-02-RESULT.md`, `05-03-RESULT.md`, `05-04-RESULT.md`, `crates/devil-desktop/src/view.rs`, `crates/devil-desktop/src/bridge.rs`, `crates/devil-desktop/src/workflow.rs`, `crates/devil-ui/src/ui.rs`, and `crates/devil-desktop/tests/projection_rendering.rs` before editing.
- evidence-before-action: confirm current desktop view only renders summary rows and current bridge lacks proposal/AI actions before modifying desktop code.
- minimal diff: modify only desktop view/bridge/workflow and focused desktop tests.
- verify-before-report: run every frontmatter command and record outcomes in `05-05-RESULT.md`.
- role: desktop UI implementer with UX architecture review.
- task: render detailed control/trust/assistant rows and translate proposal/AI user actions into `CommandDispatchIntent` values handled by app authority.
- scope: desktop view models, egui panel rendering, desktop action bridge, workflow status mapping, and tests.
- allowed tools/actions: focused Rust edits, desktop tests, cargo check.
- forbidden actions: app/protocol/UI contract changes, provider routing changes, storage writes, editor/workspace mutation, dependency changes.
- verification criteria: desktop tests prove detailed rows, empty/error states, action translation, and no product-state authority in desktop.
- final result format: write `.planning/phases/05-control-trust-and-assisted-ai-surfaces/05-05-RESULT.md` with changed files, command outcomes, and any `BLOCKED` items.

</execution_contract>

<stop_gates>

- BLOCKED if a desktop action requires owning a proposal payload, provider route, editor buffer, or workspace mutation.
- BLOCKED if detailed rows require raw source/provider text or unbounded payloads.
- BLOCKED if desktop imports app-internal modules beyond the existing adapter boundary.
- BLOCKED if app outcomes from 05-03/05-04 are not available for workflow status mapping.

</stop_gates>

<recovery>

- If egui rendering becomes too large, keep the model rows deterministic and bounded first; rendering can reuse row lists without taking authority.
- If a bridge action references a missing proposal/run id, return a typed `DesktopBridgeError` rather than synthesizing an intent.
- If workflow status mapping cannot distinguish refusal/failure/success, show a conservative error/status row from the app outcome.
- On interruption, inspect `git diff`, compare modified files against `files_modified` and `files_forbidden`, rerun verification, and continue only from unverified work.

</recovery>

<verification>

- [ ] Empty, denied, redacted, blocked, failed, and ready states render as distinct rows.
- [ ] Desktop bridge maps proposal and AI actions to `CommandDispatchIntent`.
- [ ] Desktop workflow reports app outcomes without owning lifecycle or provider state.

</verification>

## Task 1: Render Detailed Control And Trust Rows

### Read Targets

- `crates/devil-desktop/src/view.rs`
- `crates/devil-desktop/tests/projection_rendering.rs`
- `crates/devil-ui/src/ui.rs`

### Write Targets

- `crates/devil-desktop/src/view.rs`
- `crates/devil-desktop/tests/control_trust_view.rs`

### Implementation Sequence

1. Expand `DesktopProjectionViewModel` row construction so proposal rows include selected proposal id, lifecycle, diff summary, target coverage, risk, privacy, rollback availability, warnings, and diagnostics.
   > verification: `cargo test -p devil-desktop --test control_trust_view proposal_details -- --nocapture`
2. Expand trust rows so context manifest items/permissions, privacy inspector records/refusals, permission budget contracts/evaluations, approval checklist gates/blockers, and checkpoint/rollback targets/limitations render as bounded metadata rows.
   > verification: `cargo test -p devil-desktop --test control_trust_view trust_details -- --nocapture`
3. Expand assistant rows so providers, routes, requests, refusals, previews, cost/risk labels, and proposal preview links are visible without raw provider output.
   > verification: `cargo test -p devil-desktop --test control_trust_view assisted_ai_details -- --nocapture`

### Edge And Error Cases

- Empty projections must render clear empty states.
- Denied/refused/blocked states must be distinguishable from "no data".
- Keep row counts bounded; do not dump full payloads.

<verify>

```powershell
cargo test -p devil-desktop --test control_trust_view -- --nocapture
```

</verify>

### BLOCKED Conditions

- A required row cannot be rendered from `ShellProjectionSnapshot`.

## Task 2: Add Desktop Proposal And AI Actions

### Read Targets

- `crates/devil-desktop/src/bridge.rs`
- `crates/devil-ui/src/ui.rs`

### Write Targets

- `crates/devil-desktop/src/bridge.rs`
- `crates/devil-desktop/tests/control_trust_bridge.rs`

### Implementation Sequence

1. Add `DesktopAction` variants for preview, approve, reject, apply, rollback, cancel, and open proposal details. Each action must carry or resolve only a projected `ProposalId` plus display-safe reason where needed.
   > verification: `rg -q "ApproveProposal" crates/devil-desktop/src/bridge.rs`
2. Add exact `DesktopAction::StartAiExplain { instruction_label }`, `DesktopAction::StartAiProposal { instruction_label }`, `DesktopAction::CancelAiRun { run_id }`, `DesktopAction::ReplayAiRun { run_id }`, and `DesktopAction::InspectAiRun { run_id }` variants using display-safe instruction labels and projected `AgentRunId`.
   > verification: `rg -q "StartAiExplain" crates/devil-desktop/src/bridge.rs; rg -q "StartAiProposal" crates/devil-desktop/src/bridge.rs`
3. Translate those actions into `CommandDispatchIntent` values and add tests for missing proposal/run inputs, valid translation, and no app-owned payload in desktop.
   > verification: `cargo test -p devil-desktop --test control_trust_bridge -- --nocapture`

### Edge And Error Cases

- Unknown proposal/run ids must produce typed bridge errors.
- Desktop must not construct `WorkspaceProposal` or provider route requests.

<verify>

```powershell
cargo test -p devil-desktop --test control_trust_bridge -- --nocapture
```

</verify>

### BLOCKED Conditions

- Proposal/AI actions cannot be expressed as `CommandDispatchIntent` without desktop authority.

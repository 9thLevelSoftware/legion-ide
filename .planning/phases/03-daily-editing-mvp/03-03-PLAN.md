# Plan 03-03: Bounded File And Workspace Search

## Objective

Add daily-editing search in file and search in workspace through app-owned, bounded lexical services and projection-only desktop UI.

<execution_contract>

- read-before-write: read Plan 03-01/03-02 results, `crates/devil-app/src/lib.rs`, `crates/devil-ui/src/ui.rs`, `crates/devil-desktop/src/view.rs`, `workflow.rs`, `crates/devil-project/src/lib.rs`, and existing index/query DTOs.
- evidence-before-action: use workspace tree and editor/app authority for file content access; do not activate `devil-index`, LSP, vector, AI, or provider-backed search.
- minimal diff: add bounded search contracts and tests only.
- verify-before-report: run app/desktop search tests plus workspace check.
- role: indexing/search specialist constrained to Phase 3 lexical search.
- task: implement visible search-in-file and search-in-workspace behavior.
- scope: search commands, bounded result DTOs, active-buffer search, workspace lexical search, desktop search panel, cancellation/no-result/error states.
- allowed tools/actions: source reads, focused Rust edits, tests, cargo checks.
- forbidden actions: direct mutation, semantic/LSP runtime activation, vector search, hosted providers, raw search result persistence, or unbounded full-source loading for degraded buffers.
- verification criteria: search results are bounded, deterministic, cancellable/no-op when cancelled, and safe for degraded buffers.
- final result format: write `.planning/phases/03-daily-editing-mvp/03-03-RESULT.md`.

</execution_contract>

<stop_gates>

- BLOCKED if workspace search cannot be bounded without changing `devil-project`.
- BLOCKED if active-file search requires `editor.text()` for degraded buffers.
- BLOCKED if implementing search requires a new dependency or LSP/provider activation.

</stop_gates>

<recovery>

- If interrupted, inspect `git diff`, rerun app and desktop search tests, and ensure no forbidden indexing/provider files were modified.
- If search over large files becomes ambiguous, constrain Phase 3 to viewport/chunk-aware active-file search and report exact skipped/limited reasons in projection diagnostics.

</recovery>

- [ ] Active-file search works for small buffers and returns degraded/limited status for degraded buffers.
- [ ] Workspace search scans only workspace-authorized files with bounded result counts/snippet lengths.
- [ ] Desktop search panel shows query, scope, results, no-results, errors, and degraded/limited statuses.

</verification>

<success_criteria>

## Task 1: Add Projection And Command Contracts

### Read Targets

- `crates/devil-ui/src/ui.rs`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-ui/src/ui.rs`

### Implementation Sequence

1. Add `SearchScopeProjection`, `SearchResultProjection`, `SearchStatusProjection`, and `SearchProjection` structs. Fields must include scope, query label, result path/file id, bounded line/range metadata, snippet limited to a small constant, status, omitted counts, and diagnostics.
   > verification: `rg -q "pub struct SearchProjection" crates/devil-ui/src/ui.rs`
2. Add `CommandDispatchIntent::RunSearch { scope, query, limit }` and `CancelSearch { query_id }` using metadata IDs only.
   > verification: `rg -q "RunSearch" crates/devil-ui/src/ui.rs`
3. Add `search_projection` to `ShellProjectionSnapshot`, `Shell::empty`, and replacement paths.
   > verification: `rg -q "search_projection" crates/devil-ui/src/ui.rs`

### Edge And Error Cases

- Empty query returns a typed validation state, not all files.
- Snippets are bounded and redaction-friendly.
- Search projection defaults to idle with empty results.

<verify>

```powershell
rg -q "SearchProjection" crates/devil-ui/src/ui.rs
cargo check -p devil-ui --all-targets
```

</verify>

### BLOCKED Conditions

- Required result metadata cannot be represented without adding raw-source persistence fields.

## Task 2: Implement App-Owned Bounded Search

### Read Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-project/src/lib.rs`
- `crates/devil-editor/src/lib.rs`
- `crates/devil-app/tests/workspace_vfs_integration.rs`

### Write Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-app/tests/daily_editing_search.rs`

### Implementation Sequence

1. Route search intents through `CommandDispatcher` and app execution. Do not use `devil-index` in this plan.
   > verification: `rg -q "RunSearch" crates/devil-app/src/lib.rs`
2. Implement active-file search for non-degraded buffers using current editor text and bounded result counts. For degraded buffers, search only available viewport/chunk metadata or return a degraded/limited status with no full-text read.
   > verification: `rg -q "Degraded" crates/devil-app/src/lib.rs`
3. Implement workspace search by traversing the workspace tree through existing workspace authority, opening/reading candidate files with bounded size/result limits, skipping binary/oversized/unreadable files with diagnostics, and never mutating workspace state.
   > verification: `rg -q "workspace_search" crates/devil-app/src/lib.rs`
4. Add app tests for small active-file search, workspace search across two files, empty query validation, result limit/omitted count, unreadable/oversized skip diagnostics, and degraded-buffer no-full-text behavior.
   > verification: `cargo test -p devil-app daily_editing_search -- --nocapture`

### Edge And Error Cases

- Empty/whitespace query returns validation error.
- Limit zero uses a conservative default.
- Workspace search must not open files outside the trusted workspace root.
- Search failures are projected as status/diagnostics, not panics.

<verify>

```powershell
cargo test -p devil-app daily_editing_search -- --nocapture
```

</verify>

### BLOCKED Conditions

- Workspace search cannot read candidate file content without bypassing path policy.

## Task 3: Add Desktop Search Panel And Workflow Tests

### Read Targets

- `crates/devil-desktop/src/lib.rs`
- `crates/devil-desktop/src/view.rs`
- `crates/devil-desktop/src/workflow.rs`
- `crates/devil-desktop/tests/projection_rendering.rs`

### Write Targets

- `crates/devil-desktop/src/lib.rs`
- `crates/devil-desktop/src/search.rs`
- `crates/devil-desktop/src/view.rs`
- `crates/devil-desktop/src/workflow.rs`
- `crates/devil-desktop/tests/search_workflow.rs`

### Implementation Sequence

1. Add `pub mod search` and a `DesktopSearchViewModel` that derives rows only from `SearchProjection`.
   > verification: `rg -q "pub mod search" crates/devil-desktop/src/lib.rs`
2. Render search query input, scope control, result rows, no-results state, error/degraded status, and omitted-count status in the desktop view.
   > verification: `rg -q "Search" crates/devil-desktop/src/view.rs`
3. Add workflow routing for search submit/cancel actions through `DesktopCommandBridge` and `AppComposition::dispatch_ui_intent`.
   > verification: `rg -q "RunSearch" crates/devil-desktop/src/workflow.rs`
4. Add desktop tests for file search, workspace search, no results, empty query error, cancel/no-op, and degraded/limited projection display.
   > verification: `cargo test -p devil-desktop search_workflow -- --nocapture`

### Edge And Error Cases

- Search text input must not insert into the editor while the search panel has focus.
- Search result activation should reveal/open through existing app path/file intents.
- Workspace search must not block the UI indefinitely; long-running work should be bounded in this phase.

```powershell
cargo check -p devil-desktop --all-targets
```

</verify>

### BLOCKED Conditions

- Desktop search requires direct workspace/editor imports outside `DesktopRuntime`.

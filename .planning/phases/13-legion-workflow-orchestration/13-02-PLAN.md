# Plan 13-02: Protocol Workflow Contracts

## Objective

Add public protocol contracts for Legion workflow orchestration so downstream crates can coordinate workers, dependencies, conflicts, verification, sign-off, and merge approval without inventing private data shapes.

<execution_contract>

- read-before-write: read `13-CONTEXT.md`, `13-01-RESULT.md`, `crates/devil-protocol/src/lib.rs` around delegated-task, artifact-ledger, verification-run, approval, and runtime-readiness DTOs, plus `crates/devil-protocol/tests/dto_contracts.rs`.
- evidence-before-action: verify existing `ProductMode::LegionWorkflows`, `ExecutionSessionArtifact`, `EvidenceArtifact`, `ApprovalArtifact`, `DelegatedTaskRuntimeReadiness`, and proposal lifecycle types before adding new DTOs.
- minimal diff: edit only protocol source, protocol DTO tests, and this plan result artifact.
- verify-before-report: run every command in `verification_commands` and record exact outcomes in `13-02-RESULT.md`.
- role: protocol contract implementer and security/QA reviewer.
- task: define stable metadata-only Legion workflow types and validators.
- scope: session, worker, backend, dependency, conflict, verification, sign-off, merge approval, projection, and validation helpers.
- allowed tools/actions: focused Rust edits, serde round-trip tests, targeted cargo check/test.
- forbidden actions: app composition, agent runtime, tracker/memory, UI/desktop, dependency changes, raw payload retention by default, provider invocation, terminal/process execution, or weakening existing protocol validators.
- verification criteria: DTOs serialize/deserialize, invalid IDs/schema/correlation metadata are rejected, merge readiness is fail-closed, and protocol tests pass.
- final result format: write `.planning/phases/13-legion-workflow-orchestration/13-02-RESULT.md` with status, files changed, command outputs, decisions, issues, and errors.

</execution_contract>

<stop_gates>

- BLOCKED if existing protocol contracts cannot represent Legion Workflow mode without changing public semantics of earlier phases.
- BLOCKED if a required field would need raw prompt, raw source, raw logs, or provider payloads by default.
- BLOCKED if merge readiness cannot be expressed as proposal/approval metadata.
- BLOCKED if adding DTOs requires new dependencies.

</stop_gates>

## Task 1: Add Workflow Session And Worker DTOs

### Read Targets

- `crates/devil-protocol/src/lib.rs`
- `crates/devil-protocol/tests/dto_contracts.rs`

### Write Targets

- `crates/devil-protocol/src/lib.rs`
- `crates/devil-protocol/tests/dto_contracts.rs`

### Implementation Sequence

1. Add transparent id newtypes or structs for `LegionWorkflowSessionId` and `LegionWorkflowWorkerId` using the existing style for delegated-task ids.
2. Add enums for workflow state, worker state, worker role, model backend (`Local`, `ProviderBacked`, `Unavailable`), dependency state, conflict kind/state, verification gate state, sign-off state, and merge readiness state.
3. Add `LegionWorkflowWorkerAssignment` with worker id, role, model backend, display-safe model label, allowed command classes, linked delegated plan id, risk/privacy labels, correlation/causality, redaction hints, and schema version.
4. Add `LegionWorkflowSession` with session id, directive/spec/task graph artifact ids, product mode fixed to `ProductMode::LegionWorkflows`, worker assignments, dependency edges, conflict summaries, verification gates, sign-off records, proposal ids, merge approval metadata, lifecycle state, generated timestamps, redaction hints, and schema version.
   > verification: `rg -q "LegionWorkflowSession" crates/devil-protocol/src/lib.rs`

### Edge And Error Cases

- Empty ids, schema version zero, zero correlation id, nil causality id, and `RedactionHint::None` must be invalid.
- Do not store raw directive text, prompts, source diffs, command bodies, provider payloads, or terminal output bodies.
- Model backend labels are display-safe strings only; credentials, model secrets, token payloads, or provider request bodies are forbidden.

```powershell

```

</verify>

### BLOCKED Conditions

- The session contract cannot be represented without raw payload retention or new dependencies.

## Task 2: Add Dependency, Conflict, Verification, Sign-Off, And Merge Validators

### Read Targets

- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-protocol/src/lib.rs`

### Implementation Sequence

1. Add `validate_legion_workflow_session`, `validate_legion_workflow_worker_assignment`, `validate_legion_workflow_conflict`, `validate_legion_workflow_verification_gate`, `validate_legion_workflow_signoff`, and `evaluate_legion_workflow_merge_readiness`.
2. Merge readiness must return blocked when any of these are present: unresolved conflict, missing dependency, failed/blocked verification, missing required sign-off, stale proposal preconditions, dirty main workspace conflict, missing audit-before-success, missing proposal id, or invalid correlation/causality.
3. Reuse existing `DelegatedRuntimeStopGate` where appropriate and add Legion-specific display labels rather than raw diagnostics.
4. Add projection helper `legion_workflow_projection_from_sessions` that emits counts, rows, display-safe labels, redaction hints, linked proposals, and merge readiness without raw payloads.
   > verification: `rg -q "validate_legion_workflow" crates/devil-protocol/src/lib.rs`

### Edge And Error Cases

- A session with zero workers is blocked, not silently accepted.
- Provider-backed workers without an assisted-AI/provider route reference are blocked.
- Completed sessions without verification and sign-off evidence are blocked.
- Merge readiness must not mark ready when approval or rollback metadata is missing.

```powershell

```

</verify>

### BLOCKED Conditions

- Merge readiness requires mutable app state or filesystem reads inside `devil-protocol`.

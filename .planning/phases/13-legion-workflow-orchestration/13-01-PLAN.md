# Plan 13-01: Governance And Activation Gate

## Objective

Create the governance and activation record for Phase 13 before runtime orchestration code expands beyond the Phase 12 single-plan delegated-task runtime.

<execution_contract>

- read-before-write: read `AGENTS.md`, `.planning/PROJECT.md`, `.planning/ROADMAP.md`, `.planning/STATE.md`, `.planning/phases/12-delegated-task-runtime/12-CONTEXT.md`, all Phase 12 summaries, `plans/dependency-policy.md`, and the Phase 8 delegated-task evidence before editing.
- evidence-before-action: confirm from live files that Phase 12 activated only isolated proposal runtime and that Phase 13 is the first workflow-team orchestration phase.
- minimal diff: edit only the ADR, dependency policy, governance evidence, and this plan result artifact.
- verify-before-report: run every command in `verification_commands` and record the results in `13-01-RESULT.md`.
- role: governance planner, security boundary reviewer, and QA gate owner.
- task: define the accepted Phase 13 activation boundary and the evidence required before workflow orchestration can be claimed.
- scope: ADR, dependency-policy narrative, governance evidence, and result record only.
- allowed tools/actions: read/search, focused markdown edits, and `cargo run -p xtask -- check-deps`.
- forbidden actions: Rust source edits, dependency changes, CI edits, enabling provider calls, enabling terminal/process execution, direct workspace mutation, autonomous merge/apply, or weakening any existing phase gate.
- verification criteria: ADR and evidence exist, policy contains a Phase 13 boundary entry, and `xtask check-deps` still passes.
- final result format: write `.planning/phases/13-legion-workflow-orchestration/13-01-RESULT.md` with status, files changed, commands, decisions, issues, and errors.

</execution_contract>

<stop_gates>

- BLOCKED if the current Phase 12 evidence is missing or contradicts the claim that delegated runtime is isolated and proposal-only.
- BLOCKED if policy changes would authorize direct app/UI/editor/workspace mutation from `devil-agent`, `devil-tracker`, or `devil-memory`.
- BLOCKED if the ADR cannot define approval-gated merge semantics without adding new unapproved dependencies or external services.
- BLOCKED if `cargo run -p xtask -- check-deps` fails due to the governance edits.

</stop_gates>

<recovery>

After compaction, interruption, or context loss, re-read this plan, `13-CONTEXT.md`, and any `13-01-RESULT.md`; inspect `git diff`; compare changed files against `files_modified` and `files_forbidden`; rerun `verification_commands`; then continue only from unverified work.

</recovery>

- `plans/evidence/gui-productization/phase-8-delegated-task-command-center.md`

### Write Targets

- `plans/adrs/ADR-0019-legion-workflow-orchestration.md`

### Implementation Sequence

1. Create ADR-0019 with sections: Status, Context, Decision, Architecture, Model Backend Policy, Conflict And Dependency Tracking, Verification And Sign-Off, Approval-Gated Merge, Security/Privacy, Alternatives Considered, Consequences, and Acceptance Evidence.
2. State that Legion workflow workers may be represented as local or provider-backed, but provider-backed execution is routed through assisted-AI/provider consent metadata and cannot be invoked from UI/desktop.
3. Define approval-gated merge as a proposal/approval readiness state. Main-workspace mutation is forbidden until proposal approval, clean/stale checks, evidence checks, and audit-before-success checks all pass.
4. Include explicit non-goals: autonomous merge, direct workspace mutation, raw prompt/source/log retention by default, and UI/desktop runtime ownership.
   > verification: `rg -q "Approval-gated merge" plans/adrs/ADR-0019-legion-workflow-orchestration.md`

### Edge And Error Cases

- If Phase 12 evidence is incomplete, record that as a blocker instead of inventing accepted runtime behavior.
- If ADR numbering already has ADR-0019, use the next available ADR number and update this plan result with the exact path.
- If policy requires broader crate edges than currently allowed, stop and record the needed dependency-policy decision before implementation proceeds.

# Plan 13-06: Desktop Workflow Command Center

## Objective

Expose Legion workflow orchestration to users through projection-only UI and desktop command-center flows while preserving app-owned workflow authority and proposal-mediated merge gates.

<execution_contract>

- read-before-write: read `13-CONTEXT.md`, `13-05-RESULT.md`, `crates/devil-ui/src/ui.rs`, `crates/devil-desktop/src/bridge.rs`, `crates/devil-desktop/src/workflow.rs`, `crates/devil-desktop/src/view.rs`, `crates/devil-desktop/src/health.rs`, existing delegated-task desktop tests, and Plan 13-05 app request/outcome APIs before editing.
- evidence-before-action: verify existing delegated task command-center patterns for projection rows, bridge validation, workflow outcomes, health labels, and proposal-preview routing before adding Legion workflow rows.
- minimal diff: edit only UI/desktop files, one focused desktop test file, and this result artifact.
- verify-before-report: run every command in `verification_commands` and record exact outcomes in `13-06-RESULT.md`.
- role: projection/desktop implementer and QA boundary verifier.
- task: show workflow sessions, workers, dependencies, conflicts, verification, sign-off, and merge readiness without moving authority into UI/desktop.
- scope: projection snapshot fields, command parsing/intents, bridge action validation, workflow outcomes, display rows, health labels, and regression tests.
- allowed tools/actions: focused Rust edits, targeted UI/desktop tests, cargo check.
- forbidden actions: app/protocol/agent/tracker/memory edits, provider invocation, worker execution, terminal/process execution, direct file/editor mutation, proposal apply, raw payload rendering, or dependency additions.
- verification criteria: UI tests, desktop command-center tests, and desktop check pass; unknown ids are denied; proposal actions route through app/proposal authority; autonomous merge remains unsupported.
- final result format: write `.planning/phases/13-legion-workflow-orchestration/13-06-RESULT.md` with status, files changed, command outputs, decisions, issues, and errors.

</execution_contract>

## Task 1: Add Projection Fields And UI Intents

### Read Targets

- `crates/devil-ui/src/ui.rs`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-ui/src/ui.rs`

### Implementation Sequence

1. Add Legion workflow projection fields to `ShellProjectionSnapshot` and shell state using Plan 13-02 projection DTOs. Initialize empty projections with metadata-only redaction.
2. Add command intents for inspecting workflow sessions, opening linked proposal preview/details, recording verification/sign-off requests, resolving conflicts by id, and requesting merge readiness. These intents must be request metadata only.
3. Update text rendering/row projection helpers to include worker counts, dependency/conflict counts, verification/sign-off state, merge readiness, and approval-gated/autonomous-merge-unsupported labels.
4. Add UI tests named with `legion_workflow` for empty projection, populated projection, command parsing, and metadata-only labels.
   > verification: `rg -q "LegionWorkflow" crates/devil-ui/src/ui.rs`
   > verification: `cargo test -p devil-ui legion_workflow -- --nocapture`

### Edge And Error Cases

- Empty projection must render safely without panics.
- Unknown or malformed ids must not create privileged actions.
- UI must not store raw worker prompts, source, diffs, provider payloads, or logs.

<verify>

```powershell
rg -q "LegionWorkflow" crates/devil-ui/src/ui.rs
cargo test -p devil-ui legion_workflow -- --nocapture
```

</verify>

### BLOCKED Conditions

- Protocol projection DTOs are missing or cannot be carried by `devil-ui` without adding authority dependencies.

- `crates/devil-desktop/src/workflow.rs`
- `crates/devil-desktop/src/view.rs`
- `crates/devil-desktop/src/health.rs`

### Implementation Sequence

1. Add desktop actions for workflow session inspection, workflow proposal review, verification/sign-off request, conflict resolution request, and merge-readiness request. Bridge translation must validate ids against the current snapshot before emitting app requests.
2. Add workflow outcomes/status labels that distinguish inspection, verification requested, sign-off requested, conflict resolution requested, merge readiness blocked, and merge readiness waiting for approval.
3. Render command-center rows in `view.rs` with display-safe counts and labels: sessions, workers, model backend labels, dependencies, conflicts, verification gates, sign-off, linked proposals, and merge readiness.
4. Update health text to include `Autonomous merge: unsupported until approval`.
   > verification: `rg -q "legion workflow command center" crates/devil-desktop/src/view.rs`
   > verification: `rg -q "Autonomous merge" crates/devil-desktop/src/health.rs`

### Edge And Error Cases

- Unknown session, worker, conflict, verification, sign-off, or proposal ids must return explicit bridge errors.
- Desktop cannot call `execute_legion_workflow` directly unless Plan 13-05 exposes an app request intended for that route.
- Merge readiness request is not apply; it must route to app-owned readiness/approval.

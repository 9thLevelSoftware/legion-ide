## Task 1: Add Coordinator State And Worker Scheduling

### Read Targets

- `crates/devil-agent/src/lib.rs`
- `crates/devil-agent/Cargo.toml`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-agent/src/lib.rs`

### Implementation Sequence

1. Add `LegionWorkflowCoordinator` that owns only metadata state: session id, ordered worker assignments, completed worker ids, blocked worker ids, conflict summaries, provider route requests, proposal outputs, and verification labels.
2. Add constructor accepting a validated `LegionWorkflowSession` or equivalent Plan 13-02 session DTO. Store no raw prompt/source/log/provider payloads.
3. Implement dependency scheduling so a worker can start only after all declared predecessor workers have completed or been explicitly skipped by a blocked state. Cycles or missing dependencies must produce a blocked coordinator result.
4. Implement deterministic `next_ready_workers()` and `mark_worker_completed()` methods; they must be pure metadata operations and must not spawn processes, invoke providers, write files, or apply proposals.
   > verification: `rg -q "LegionWorkflowCoordinator" crates/devil-agent/src/lib.rs`

### Edge And Error Cases

- Empty worker list is blocked.
- Dependency cycles are blocked.
- Missing worker ids in dependency edges are blocked.
- Duplicate worker ids are blocked.
- A worker cannot complete twice.

</verify>

### BLOCKED Conditions

- Coordinator state cannot be represented without storing raw task content.

## Task 2: Add Provider Route And Conflict Metadata Handling

### Read Targets

- `crates/devil-agent/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-agent/src/lib.rs`

### Implementation Sequence

1. For local workers, reuse existing delegated-task sandbox/proposal DTO flow where proposal output is needed. Do not add broad new filesystem walking beyond the Phase 12 generator boundary.
2. For provider-backed workers, emit `AssistedAiProviderRouteRequest` or the Plan 13-02 route metadata required by the protocol; do not invoke providers.
3. Add conflict metadata detection based on display-safe target labels and affected target summaries: two workers touching the same target label in overlapping dependency order must produce an unresolved conflict record until app composition marks it resolved.
4. Add coordinator outputs for blocked, conflict, provider-route-required, proposal-ready, and completed states.
   > verification: `cargo test -p devil-agent legion_workflow -- --nocapture`

### Edge And Error Cases

- Provider-backed worker without route metadata blocks.
- Local worker proposal output with invalid correlation/causality blocks.
- Conflicts across workers in the same dependency layer block merge readiness but do not delete worker outputs.
- Conflict records must not include raw diffs or source text.

</verify>

### BLOCKED Conditions

- Provider route metadata cannot be generated without invoking an external provider.

## Task 1: Add Tracker Workflow Ledger Records

### Read Targets

- `crates/devil-tracker/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-tracker/src/lib.rs`

### Implementation Sequence

1. Add `LegionWorkflowTrackerRecord` or similarly named metadata record with workflow session id, worker id optional, linked proposal ids, verification gate ids, conflict ids/counts, sign-off ids/counts, merge readiness state, correlation id, causality id, event sequence, risk/privacy labels, redaction hints, and schema version.
2. Add `validate` logic that rejects zero/nil identifiers, schema version zero, empty workflow session ids, event sequence zero, and `RedactionHint::None`.
3. Extend or add a tracker ledger type with append, `by_workflow_session_id`, `by_worker_id`, `by_proposal_id`, `conflicts_for_session`, and `verification_records_for_session` lookup helpers.
4. Keep records metadata-only; store summary hashes and display-safe labels instead of raw command bodies, source, logs, prompts, diffs, or provider payloads.
   > verification: `rg -q "LegionWorkflow" crates/devil-tracker/src/lib.rs`

### Edge And Error Cases

- Duplicate record ids should be rejected or deterministically replaced only if existing tracker style already supports replacement.
- A record with merge-ready state but unresolved conflict count > 0 must be invalid.
- Records without required verification/sign-off counts for completed workflow state must be invalid.

## Task 2: Add Consent-Aware Workflow Memory Candidates

### Read Targets

- `crates/devil-memory/src/lib.rs`
- `crates/devil-tracker/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-memory/src/lib.rs`

### Implementation Sequence

1. Add `LegionWorkflowOutcomeCandidate` or extend `MemoryCandidateRecord` with workflow-specific metadata: workflow session id, worker role/backend labels, outcome label, verification state, sign-off state, conflict count, proposal count, summary hash, consent state, redaction hints, and schema version.
2. Add helper logic that creates a memory candidate from tracker/protocol metadata only. Do not retain raw prompts, raw generated code, raw command output, raw provider payload, or source text.
3. Ensure retention requires explicit consent. Consent denied or unknown must leave the candidate proposed but not retained, matching existing memory behavior.
4. Add delete and lookup helpers for retained workflow candidates by workflow session id and worker role/backend label.
   > verification: `rg -q "LegionWorkflow" crates/devil-memory/src/lib.rs`

### Edge And Error Cases

- Unknown consent blocks retention.
- Empty workflow id or summary hash blocks candidate creation.
- Raw payload retention flags must default false.


</verify>

- Memory retention cannot be represented without retaining raw output bodies.

# Plan 07-03: Operational Health And Privacy-Safe Diagnostics

## Objective

Make GUI-visible operational health and diagnostics export usable for beta readiness while keeping diagnostics metadata-only and honest about unsupported advanced surfaces.

<execution_contract>

- read-before-write: read `07-CONTEXT.md`, `07-02-RESULT.md`, `AGENTS.md`, `crates/devil-desktop/src/diagnostics.rs`, `workflow.rs`, `view.rs`, `platform.rs`, `beta.rs`, existing diagnostics/platform/view tests, `plans/evidence/gui-productization/phase-6-session-diagnostics-safety.md`, and `plans/evidence/gui-productization/phase-7-local-workflow-smoke.md` before editing.
- evidence-before-action: confirm existing diagnostics omit raw editor text and status-message bodies before adding Phase 7 health fields.
- minimal diff: add health summarization inside `devil-desktop`; do not change app/UI/protocol authority contracts.
- verify-before-report: run every frontmatter command and record outcomes in `07-03-RESULT.md`.
- role: diagnostics and health implementer.
- task: expose metadata-only health rows and diagnostics fields for beta workflows.
- scope: desktop health helper, diagnostics export, view model/render rows, tests, evidence, and result file.
- allowed tools/actions: focused Rust/Markdown edits, targeted tests, beta smoke command, and `rg`.
- forbidden actions: adding raw payload fields, persisting status-message bodies, editing app/protocol/UI contracts, or claiming unsupported plugin/collaboration/remote/autonomy readiness.
- verification criteria: health rows are visible in the desktop view model; diagnostics export includes counts/status labels only; secret-like dirty text is absent from diagnostics/evidence.
- final result format: write `.planning/phases/07-fully-functional-local-ide-beta/07-03-RESULT.md` with changed files, command outcomes, redaction checks, and any `BLOCKED` items.

</execution_contract>

<stop_gates>

- BLOCKED if operational health requires adding state ownership to `devil-ui` or bypassing `ShellProjectionSnapshot`.
- BLOCKED if useful diagnostics cannot be produced without raw editor text, terminal payloads, provider payloads, prompts, source bodies, or secrets.
- BLOCKED if health rows would claim plugin, collaboration, remote, hosted provider, signed installer, or autonomous apply support that Phase 7 local beta has not accepted.

</stop_gates>

<recovery>

- If diagnostics export fields grow too broad, reduce them to counts, booleans, status labels, ids, schema versions, and redaction labels.
- If view rendering becomes noisy, keep the health surface as compact rows in `DesktopProjectionViewModel` with stable test assertions.
- If beta smoke cannot regenerate diagnostics on the host, verify diagnostics through integration tests and record the smoke blocker separately.
- On interruption, inspect `git diff`, compare modified files against `files_modified` and `files_forbidden`, rerun verification, and continue only from unverified work.

</recovery>

<verification>

- [ ] `DesktopOperationalHealthSnapshot` or equivalent contains only metadata labels/counts.
- [ ] Diagnostics markdown includes `## Operational Health` and unsupported-surface labels.
- [ ] Desktop view model exposes health rows derived from projections.
- [ ] Tests prove secret-like dirty text, source bodies, prompts, terminal payload markers, and status-message bodies are absent from diagnostics.
- [ ] Evidence file records command results and residual limitations.

</verification>

<success_criteria>

- GUI beta users can see operational health from the desktop surface.
- Diagnostics export is useful for beta triage without leaking raw payloads.
- Unsupported advanced surfaces are visible as limitations rather than hidden claims.

</success_criteria>

<output>

Write `.planning/phases/07-fully-functional-local-ide-beta/07-03-RESULT.md`.

</output>

## Task 1: Add Metadata-Only Operational Health Snapshot

### Read Targets

- `crates/devil-desktop/src/diagnostics.rs`
- `crates/devil-desktop/src/workflow.rs`
- `crates/devil-desktop/src/view.rs`
- `crates/devil-desktop/src/platform.rs`
- `crates/devil-desktop/src/beta.rs`

### Write Targets

- `crates/devil-desktop/src/health.rs`
- `crates/devil-desktop/src/lib.rs`
- `crates/devil-desktop/src/diagnostics.rs`
- `crates/devil-desktop/src/workflow.rs`

### Implementation Sequence

1. Add `health.rs` with a `DesktopOperationalHealthSnapshot` that includes workspace label, open/dirty tab counts, status count, last outcome label, search status, language status, terminal status, proposal row count, assisted AI run/proposal counts, diagnostics/session configured booleans, and `unsupported_surfaces` labels.
   > verification: `test -f crates/devil-desktop/src/health.rs`
2. Build the snapshot from `ShellProjectionSnapshot`, `DesktopWorkflowOutcome`, and config booleans. Do not store raw status messages, buffer previews, source bodies, terminal payloads, provider payloads, or prompt text.
   > verification: `rg -q "DesktopOperationalHealthSnapshot" crates/devil-desktop/src/health.rs`
3. Add the health snapshot to `DesktopDiagnosticsExport` and render a `## Operational Health` markdown section with stable key/value labels.
   > verification: `rg -q "Operational Health" crates/devil-desktop/src/diagnostics.rs`

### Edge And Error Cases

- Unknown or absent projections should produce `not_configured`, `not_observed`, or `default_denied` labels rather than false failures.
- The workspace path may be displayed, but file contents and buffer previews must not be displayed.
- Unsupported advanced surfaces must be listed as limitations, not errors.

<verify>

```powershell
test -f crates/devil-desktop/src/health.rs
rg -q "DesktopOperationalHealthSnapshot" crates/devil-desktop/src/health.rs
rg -q "Operational Health" crates/devil-desktop/src/diagnostics.rs
```

</verify>

### BLOCKED Conditions

- The health snapshot cannot be built without raw payloads or new app/protocol contracts.

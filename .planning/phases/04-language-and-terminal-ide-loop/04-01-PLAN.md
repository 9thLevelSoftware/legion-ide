# Plan 04-01: Governance And Projection Contract Rebaseline

## Objective

Establish the policy, dependency, protocol, and projection-only UI contracts that all Phase 4 language and terminal implementation must obey.

<execution_contract>

- read-before-write: read `.planning/phases/04-language-and-terminal-ide-loop/04-CONTEXT.md`, `.planning/PROJECT.md`, `.planning/ROADMAP.md`, `plans/dependency-policy.md`, `xtask/src/main.rs`, `crates/devil-app/Cargo.toml`, `crates/devil-protocol/src/lib.rs`, `crates/devil-protocol/tests/dto_contracts.rs`, and `crates/devil-ui/src/ui.rs` before editing.
- evidence-before-action: confirm the legacy Phase 4 AI evidence path in `xtask/src/main.rs` and the older policy numbering before changing policy text or checks.
- minimal diff: modify only the listed files and only enough to authorize GUI Phase 4 composition and expose projection DTOs/intents.
- verify-before-report: run every frontmatter command and record the command outcomes in `04-01-RESULT.md`.
- role: policy/protocol implementer with senior review discipline.
- task: rebaseline governance and add Phase 4 projection contracts without activating renderer, app behavior, LSP process launch, or terminal process launch.
- scope: dependency policy, optional `xtask` parser/check updates, app dependency declarations, protocol DTOs, DTO contract tests, and projection-only UI contracts.
- allowed tools/actions: focused Rust/Markdown edits, grep checks, cargo check/test, and `xtask check-deps`.
- forbidden actions: mutating editor/workspace/storage behavior, starting terminal processes, launching language servers, adding renderer dependencies to `devil-ui`, or changing root workspace dependencies.
- verification criteria: policy accepts GUI Phase 4 edges without breaking existing checks, protocol DTOs round-trip, UI remains projection-only, and no forbidden files change.
- final result format: write `.planning/phases/04-language-and-terminal-ide-loop/04-01-RESULT.md` with changed files, decisions, command outcomes, and any `BLOCKED` items.

</execution_contract>

<stop_gates>

- BLOCKED if `xtask check-deps` cannot be preserved while authorizing GUI Phase 4 app composition.
- BLOCKED if UI contract work requires `devil-ui` to depend on app, desktop, editor, index, terminal, project, storage, or renderer crates.
- BLOCKED if a language or terminal projection needs raw buffer text, raw terminal command text, or unbounded output in durable structures.
- BLOCKED if adding app dependencies requires changing root workspace dependency policy without a matching governance update.

</stop_gates>

<recovery>

- If `xtask` changes break legacy evidence checks, restore the previous checks and add only the smallest compatibility branch needed for GUI Phase 4 policy validation.
- If DTO additions duplicate existing protocol types, reuse or wrap the existing DTOs rather than creating parallel representations.
- If a needed UI field cannot be represented safely, add an explicit denial/error/status projection and stop implementation behavior for the later plan that would need it.

</recovery>

## Task 2: Add Protocol DTO Contract Surface

### Read Targets

- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-protocol/src/lib.rs`
- `crates/devil-protocol/tests/dto_contracts.rs`

### Implementation Sequence

1. Add or reuse DTOs for diagnostics/problems, hover, completion, definition, references, outline, formatting proposal status, rename proposal status, organize-imports proposal status, code-action proposal status, and cancellable operation metadata.
   > verification: `rg -q "LanguageToolingProjection" crates/devil-protocol/src/lib.rs`
2. Add or reuse DTOs for terminal runtime status, policy decision, launch/input/resize/kill status, bounded output page, scrollback/search status, denial/error state, and metadata-only audit correlation.
   > verification: `rg -q "TerminalPanelProjection" crates/devil-protocol/src/lib.rs`
3. Add focused DTO tests named with the `language_terminal_projection` filter that serialize/deserialize both projection groups and assert no raw source text or raw command text is required by the projection contracts.
   > verification: `cargo test -p devil-protocol --test dto_contracts language_terminal_projection -- --nocapture`

### Edge And Error Cases

- Prefer existing `Lsp*`, `Terminal*`, `CorrelationId`, `CausalityId`, and proposal DTOs.
- Encode denial/error states as data, not panics.
- Add operation ids where cancellation or stale-response filtering is needed.

## Task 3: Add Projection-Only UI Intents And Snapshot Fields

### Read Targets

- `crates/devil-ui/src/ui.rs`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-ui/src/ui.rs`

### Implementation Sequence

1. Add language tooling projection fields to `ShellProjectionSnapshot` and empty defaults. The projections must contain display/status data only.
   > verification: `rg -q "language_tooling_projection" crates/devil-ui/src/ui.rs`
2. Add terminal panel projection fields to `ShellProjectionSnapshot` and empty defaults. The projections must expose bounded display rows, status, search/scrollback metadata, and denial/error state.
   > verification: `rg -q "terminal_panel_projection" crates/devil-ui/src/ui.rs`
3. Add `CommandDispatchIntent` variants for hover, completion, definition, references, outline refresh, formatting proposal, rename proposal, organize imports proposal, code-action proposal, language operation cancel, terminal launch, input, resize, kill, close, output poll, and terminal search.
   > verification: `rg -q "RequestCompletion" crates/devil-ui/src/ui.rs; rg -q "TerminalLaunch" crates/devil-ui/src/ui.rs`

### Edge And Error Cases

- Do not add runtime handles, buffers, or terminal sessions as UI-owned state.
- Default projections must render as empty/disabled states in existing CLI shell output.
- Command variants must carry the minimum IDs and metadata needed for app routing.

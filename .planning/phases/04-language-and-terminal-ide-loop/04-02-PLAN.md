# Plan 04-02: App Language Tooling Composition And Proposal Routing

## Objective

Compose app-owned language tooling workflows from existing protocol and semantic-index contracts, with every edit-producing action converted into a proposal preview before mutation.

<execution_contract>

- read-before-write: read `.planning/phases/04-language-and-terminal-ide-loop/04-CONTEXT.md`, `04-01-RESULT.md`, `crates/devil-app/src/lib.rs`, `crates/devil-index/src/lib.rs`, `crates/devil-protocol/src/lib.rs`, `crates/devil-ui/src/ui.rs`, `plans/adrs/ADR-0018-lsp-runtime-supervision.md`, and existing app proposal/save tests before editing.
- evidence-before-action: identify the existing index query APIs and proposal conversion helpers before adding any new app service methods.
- minimal diff: keep changes inside app source and one focused app integration test file.
- verify-before-report: run every frontmatter command and record command outcomes in `04-02-RESULT.md`.
- role: language tooling implementer with proposal-boundary responsibility.
- task: route diagnostics, hover, completion, definition, references, outline, formatting, rename, organize imports, and code actions through app authority.
- scope: app state, command dispatcher routing, projection building, index/LSP DTO adaptation, cancellation/stale result filtering, proposal conversion, and app tests.
- allowed tools/actions: Rust edits in listed files, deterministic fake/index fixtures, cargo check/test, and source-boundary greps.
- forbidden actions: launching real language servers, adding renderer code, editing UI/protocol contracts, mutating editor buffers directly from language actions, or writing workspace files directly.
- verification criteria: language surfaces project into the shell snapshot, edit-producing actions create proposal previews, failed/cancelled/stale operations are visible, and tests prove editor text/disk are unchanged until accepted proposal authority runs.
- final result format: write `.planning/phases/04-language-and-terminal-ide-loop/04-02-RESULT.md` with implementation summary, command outcomes, and unresolved risks.

</execution_contract>

<stop_gates>

- BLOCKED if a formatting, rename, organize-imports, or code-action path cannot be converted through `convert_lsp_edit_to_workspace_proposal` or an equivalent existing proposal service.
- BLOCKED if `devil-index` requires direct editor/workspace ownership instead of snapshot/protocol inputs.
- BLOCKED if app routing requires adding dependencies outside those authorized by Plan 04-01.
- BLOCKED if cancellation cannot prevent stale operation results from replacing newer projection state.

</stop_gates>

<recovery>

- If full LSP supervision is not yet executable, use deterministic app-owned adapters over current protocol/index DTOs and expose unavailable process status explicitly.
- If a language operation lacks data for the active buffer, produce a typed projection error instead of panicking or silently clearing prior state.
- If proposal conversion rejects an edit, keep the editor dirty state unchanged and project the rejection reason.

</recovery>

## Task 1: Add App-Owned Language State And Read-Only Surfaces

### Read Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-index/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`
- `crates/devil-ui/src/ui.rs`

### Write Targets

- `crates/devil-app/src/lib.rs`

### Implementation Sequence

1. Add an app-owned language tooling state object that stores operation metadata, latest diagnostics/problems, hover, completion, definition, references, and outline projections for the active buffer.
   > verification: `rg -q "LanguageTooling" crates/devil-app/src/lib.rs`
2. Route new language `CommandDispatchIntent` variants through `CommandDispatcher` and `CommandExecutionService` into app-owned service methods.
   > verification: `rg -q "RequestCompletion" crates/devil-app/src/lib.rs`
3. Use `devil-index` semantic queries and existing protocol DTOs for read-only operations. Do not add index ownership of editor/workspace state.
   > verification: `rg -q "SemanticIndex" crates/devil-app/src/lib.rs`

### Edge And Error Cases

- Unknown active buffer returns a visible projection error.
- Unsupported language/server state projects as unavailable, not success.
- Read-only language actions do not mark buffers dirty.

## Task 2: Route Edit-Producing Language Actions Through Proposals

### Read Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`
- existing app proposal ledger and save workflow tests

### Write Targets

- `crates/devil-app/src/lib.rs`

### Implementation Sequence

1. Implement formatting proposal routing using existing LSP edit-to-workspace proposal conversion and app proposal preview plumbing.
   > verification: `rg -q "formatting_proposal" crates/devil-app/src/lib.rs`
2. Implement rename, organize-imports, and code-action proposal routing using the same conversion/preview path. If an action has no edit, project a no-op or command-denied status rather than mutating directly.
   > verification: `rg -q "rename_proposal" crates/devil-app/src/lib.rs; rg -q "organize_import" crates/devil-app/src/lib.rs; rg -q "code_action" crates/devil-app/src/lib.rs`
3. Add operation id, correlation, causality, and stale-result checks so cancelled or superseded edit-producing operations cannot replace current proposal status.
   > verification: `rg -q "CancelLanguageOperation" crates/devil-app/src/lib.rs`

### Edge And Error Cases

- Proposal rejection preserves dirty editor text.
- Edit conversion failure is projected with typed status.
- Command-like LSP code actions remain denied unless they can be represented as proposal previews.

<verify>

```powershell

cargo check -p devil-app --all-targets
```

</verify>

### BLOCKED Conditions

- A required edit-producing path cannot be represented as a proposal preview with existing proposal contracts.

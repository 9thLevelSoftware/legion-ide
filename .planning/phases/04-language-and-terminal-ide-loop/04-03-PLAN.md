# Plan 04-03: Policy-Gated Terminal App Workflow

## Objective

Route terminal lifecycle through app and security authority so the desktop can display a local IDE terminal panel without gaining direct PTY, filesystem, editor, or workspace mutation authority.

<execution_contract>

- read-before-write: read `.planning/phases/04-language-and-terminal-ide-loop/04-CONTEXT.md`, `04-01-RESULT.md`, `crates/devil-app/src/lib.rs`, `crates/devil-terminal/src/lib.rs`, `crates/devil-security/src/lib.rs`, `crates/devil-platform/src/lib.rs`, `crates/devil-protocol/src/lib.rs`, and `crates/devil-ui/src/ui.rs` before editing.
- evidence-before-action: confirm terminal default-disabled behavior and security capability checks before wiring app commands.
- minimal diff: keep changes inside app source and one focused app integration test file.
- verify-before-report: run every frontmatter command and record command outcomes in `04-03-RESULT.md`.
- role: terminal workflow implementer with security-boundary responsibility.
- task: compose policy-gated terminal launch, input, resize, kill, close, output polling, search, scrollback status, and denial/error projection through app authority.
- scope: app state, command routing, terminal runtime adapter selection, security broker use, bounded output projection, search/scrollback metadata, cancellation/kill outcomes, and app tests.
- allowed tools/actions: Rust edits in listed files, deterministic terminal fixture usage, cargo check/test, and source-boundary greps.
- forbidden actions: changing terminal runtime internals, weakening security policy, storing raw terminal transcripts in durable audit state, letting desktop/UI launch processes directly, or modifying editor/workspace/disk through terminal workflows.
- verification criteria: denied launch is visible, allowed fixture launch works through policy, input/resize/kill/output/search states project correctly, and tests prove no editor buffer or workspace file mutation happens directly.
- final result format: write `.planning/phases/04-language-and-terminal-ide-loop/04-03-RESULT.md` with implementation summary, command outcomes, and unresolved risks.

</execution_contract>

<stop_gates>

- BLOCKED if terminal launch cannot be checked through existing security policy before runtime invocation.
- BLOCKED if terminal output must be stored unbounded or unredacted to satisfy projection requirements.
- BLOCKED if desktop/UI would need direct access to `TerminalRuntime` or `PtyService`.
- BLOCKED if implementing app workflow requires weakening `devil-security` default-deny behavior.

</stop_gates>

<recovery>

- If native PTY launch is unavailable in the local environment, keep fixture-backed tests and project native runtime unavailability explicitly.
- If output search cannot scan full scrollback safely, expose bounded-window search status and truncated markers.
- If kill/close fails, preserve the terminal session projection with a visible failure state and do not mark the session cleanly closed.

</recovery>

<verification>

- [ ] Terminal workflow app tests pass.
- [ ] `devil-app` all-target check passes.
- [ ] Greps confirm app owns terminal workflow routing and uses terminal runtime only behind policy.
- [ ] No forbidden file is modified.

</verification>

<success_criteria>

- Terminal launch is denied by default and the denial is visible in the shell projection.
- Allowed terminal launch, input, resize, output poll, close, and kill are routed through app authority.
- Output projection is bounded and carries search/scrollback status.
- Terminal workflows do not mutate editor buffers or disk directly.
- Terminal failures are projected with correlation/causality metadata.

</success_criteria>

## Task 1: Add App-Owned Terminal State And Policy Routing

### Read Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-terminal/src/lib.rs`
- `crates/devil-security/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`
- `crates/devil-ui/src/ui.rs`

### Write Targets

- `crates/devil-app/src/lib.rs`

### Implementation Sequence

1. Add app-owned terminal workflow state for session id, runtime status, policy decision, output cursor, bounded display rows, search/scrollback status, and last error/denial.
   > verification: `rg -q "TerminalWorkflow" crates/devil-app/src/lib.rs`
2. Route terminal launch through existing security capability policy before invoking `devil-terminal`. Denied launch must update projection state and not call runtime launch.
   > verification: `rg -q "TerminalLaunchPolicy" crates/devil-app/src/lib.rs`
3. Use the existing terminal fixture/runtime adapter shape from `devil-terminal` for app-owned lifecycle. Do not expose runtime handles to UI or desktop.
   > verification: `rg -q "TerminalRuntime" crates/devil-app/src/lib.rs`

### Edge And Error Cases

- Untrusted workspace projects a denial.
- Missing command metadata projects a denial.
- Runtime unavailable projects an error state, not a panic.

<verify>

```powershell
rg -q "TerminalWorkflow" crates/devil-app/src/lib.rs
cargo check -p devil-app --all-targets
```

</verify>

### BLOCKED Conditions

- Existing security policy cannot evaluate terminal launch before runtime invocation.

## Task 2: Implement Terminal Lifecycle Commands And Projection Updates

### Read Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-terminal/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `crates/devil-app/src/lib.rs`

### Implementation Sequence

1. Implement app command routing for terminal input, resize, output poll, search, close, and kill.
   > verification: `rg -q "TerminalInput" crates/devil-app/src/lib.rs; rg -q "TerminalKill" crates/devil-app/src/lib.rs`
2. Convert terminal output chunks into bounded projection rows with truncation/search/scrollback status. Do not persist raw transcripts in app evidence or audit state.
   > verification: `rg -q "scrollback" crates/devil-app/src/lib.rs`
3. Attach non-zero correlation/causality metadata to terminal outcomes and errors where existing protocol constructors require it.
   > verification: `rg -q "CorrelationId" crates/devil-app/src/lib.rs`

### Edge And Error Cases

- Input to a closed or denied session projects an error.
- Resize before launch projects disabled/unavailable status.
- Kill is idempotent at the app projection boundary but must still report runtime errors.

## Task 3: Add Terminal Workflow Regression Tests

### Read Targets

- `crates/devil-app/tests/daily_editing_contracts.rs`

- `crates/devil-security/src/lib.rs`

### Write Targets

- `crates/devil-app/tests/terminal_workflow.rs`

### Implementation Sequence

1. Add tests proving default/untrusted launch denial is visible and runtime launch is not invoked.
   > verification: `cargo test -p devil-app --test terminal_workflow terminal_denial_is_visible_and_fail_closed -- --exact`
2. Add tests for allowed fixture launch, input, resize, output polling, search/scrollback status, close, and kill.
   > verification: `cargo test -p devil-app --test terminal_workflow terminal_fixture_lifecycle_projects_status -- --exact`
3. Add tests proving terminal workflows do not mutate active editor text or workspace files directly and output projection is bounded.
   > verification: `cargo test -p devil-app --test terminal_workflow terminal_workflow_cannot_mutate_editor_or_disk -- --exact`

### Edge And Error Cases

- Use deterministic terminal fixture paths; do not rely on host shell behavior.
- Assert projection state through public app APIs.
- Avoid sleeps; poll deterministic fixture output.

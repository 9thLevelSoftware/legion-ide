# Plan 04-05: Cross-Boundary Safety And Failure Tests

## Objective

Prove the Phase 4 IDE loop preserves critical boundaries across app, UI, desktop, terminal, language tooling, security, and proposal workflows.

<execution_contract>

- read-before-write: read `.planning/phases/04-language-and-terminal-ide-loop/04-CONTEXT.md`, `04-02-RESULT.md`, `04-03-RESULT.md`, `04-04-RESULT.md`, existing app/desktop/terminal/security tests, and `plans/evidence/gui-productization/phase-3-daily-editing-mvp.md` before editing.
- evidence-before-action: identify already-covered language and terminal tests before adding cross-boundary tests to avoid duplicate assertions.
- minimal diff: add only integration/workflow tests and targeted evidence Markdown; do not change product code in this plan.
- verify-before-report: run every frontmatter command and record command outcomes in `04-05-RESULT.md`.
- role: verification implementer with API/security/performance focus.
- task: add cross-boundary tests and evidence for no direct mutation, visible failures, cancellation, bounded output, policy denial, and non-blocking editor/save behavior.
- scope: app integration tests, desktop workflow tests, terminal/security package verification, workspace check, and targeted safety evidence.
- allowed tools/actions: test additions, evidence Markdown, cargo test/check, grep, and deterministic fixtures.
- forbidden actions: product-code fixes, weakening existing tests, skipping terminal/security gates, adding sleeps/flaky timing assertions, or creating broad manual-only verification.
- verification criteria: all targeted tests pass, cargo check passes, evidence file records commands/results, and any product bug is reported as `BLOCKED` rather than patched here.
- final result format: write `.planning/phases/04-language-and-terminal-ide-loop/04-05-RESULT.md` with test/evidence summary and any `BLOCKED` defects.

</execution_contract>

<stop_gates>

- BLOCKED if a test exposes product-code behavior that violates proposal mediation, terminal policy, bounded output, metadata audit, or projection-only ownership.
- BLOCKED if a required failure/cancellation state is not observable through public projections.
- BLOCKED if performance evidence cannot distinguish bounded projection behavior from unbounded terminal or language output growth.
- BLOCKED if a failing terminal/security package test is causally related to Phase 4 app/desktop routing.

</stop_gates>

<recovery>

- If a product defect appears, stop this plan and document the failing command, observed behavior, and owning earlier plan rather than editing product code.
- If a broad workspace check fails for an unrelated reason, keep targeted tests running and document the separate failure with evidence.
- If deterministic fixture output is insufficient for scrollback/search status, create fixture data inside the test file rather than depending on host shell output.

</recovery>

### Read Targets

- `crates/devil-app/tests/language_tooling_workflow.rs`

- `crates/devil-app/tests/workspace_vfs_integration.rs`

### Write Targets

- `crates/devil-app/tests/language_terminal_integration.rs`

### Implementation Sequence

1. Add a test that opens a workspace file, runs read-only language actions, and proves editor dirty state and disk content do not change.
   > verification: `cargo test -p devil-app --test language_terminal_integration language_read_only_actions_do_not_mutate -- --exact`
2. Add a test that runs formatting/rename/organize/code-action proposal workflows and proves mutation is absent until accepted through the existing proposal path.
   > verification: `cargo test -p devil-app --test language_terminal_integration language_edit_actions_require_proposals -- --exact`
3. Add a test that runs terminal denied and allowed fixture workflows while verifying active editor text and workspace file contents remain unchanged.
   > verification: `cargo test -p devil-app --test language_terminal_integration terminal_actions_cannot_mutate_editor_or_disk -- --exact`

### Edge And Error Cases

- Use temporary workspaces and deterministic fixture output.
- Assert disk contents explicitly before and after workflows.
- Do not call private app internals.

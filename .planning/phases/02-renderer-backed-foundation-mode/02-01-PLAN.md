# Plan 02-01: Desktop Crate Scaffold And Renderer Dependency Wiring

## Objective

Add the `devil-desktop` adapter crate and renderer dependencies in the only policy-approved location, while keeping the crate compile-safe for later Phase 2 waves.

<execution_contract>

- read-before-write: read `Cargo.toml`, `plans/dependency-policy.md`, `xtask/src/main.rs`, ADR-0002, ADR-0030, and current app/UI manifests before editing.
- evidence-before-action: add only dependencies already authorized by ADR-0002 and the dependency policy unless the policy is updated in the same minimal change.
- minimal diff: scaffold `devil-desktop` plus workspace/dependency wiring only; do not implement renderer workflows in this plan.
- verify-before-report: run `xtask check-deps`, package check, and the exact renderer-gate regression test before reporting completion.
- role: act as a Rust workspace integrator preserving architecture boundaries.
- task: create the adapter crate that later plans will fill in.
- scope: workspace manifest, desktop manifest, compile-safe desktop source stubs, and dependency policy wording if the planned crate wording must become active crate wording.
- allowed tools/actions: edit manifests/source stubs, run cargo metadata/check/test, run `xtask`.
- forbidden actions: editing `devil-ui`, moving app/editor/workspace logic, adding renderer dependencies to any non-adapter crate, or weakening `xtask`.
- verification criteria: the new package exists, compiles by itself, and dependency gates still fail closed for non-adapter renderer dependencies.
- final result format: write a plan result with changed files, verification commands, and any dependency-policy residual risk.

</execution_contract>

<stop_gates>

- BLOCKED if `plans/dependency-policy.md` no longer authorizes `devil-desktop` as the only renderer dependency host.
- BLOCKED if `cargo run -p xtask -- check-deps` fails because adding `devil-desktop` requires broader dependency-policy or `xtask` redesign.
- BLOCKED if `eframe`/`egui` cannot be added without placing renderer/windowing dependencies in `devil-ui` or core crates.

</stop_gates>

<recovery>

- If dependency policy still says `planned` after the crate is added, make the smallest wording update to describe `devil-desktop` as the active Phase 2 adapter while preserving all denials.
- If `cargo check -p devil-desktop --all-targets` fails from unused stub modules, use minimal public no-op types/functions in each module instead of suppressing warnings globally.
- If cargo resolves additional renderer stack crates, confirm `xtask` allows them only through `devil-desktop`; do not add them to non-adapter manifests.
- If work is interrupted, capture `git diff`, modified files, verification state, and the plan result artifact path before stopping.

</recovery>

<verification>

Run the frontmatter verification commands in order. Treat any failure as plan-blocking unless it is fixed within this plan's write targets.

</verification>

<success_criteria>

- `devil-desktop` is a workspace package with a binary entry point.
- `devil-desktop` depends on `devil-app`, `devil-ui`, `devil-protocol`, `eframe`, and `egui`; no non-adapter crate gains renderer dependencies.
- `cargo run -p xtask -- check-deps` and `cargo check -p devil-desktop --all-targets` pass.

</success_criteria>

Write `.planning/phases/02-renderer-backed-foundation-mode/02-01-RESULT.md` with scaffold summary, dependency list, verification results, and any cargo-deny warnings observed.

- `Cargo.toml`
- `Cargo.lock`

### Write Targets

- `Cargo.toml`
- `Cargo.lock`
- `plans/dependency-policy.md`
- `crates/devil-desktop/Cargo.toml`

### Implementation Sequence

1. Add `crates/devil-desktop` to the root `[workspace].members` list near `crates/devil-app` or `crates/devil-ui`.
   > verification: `rg -q "crates/devil-desktop" Cargo.toml`
2. Add shared workspace dependencies for `eframe` and `egui` using exact crate names authorized by Phase 1; use current compatible versions resolved by Cargo and no default-feature suppression unless required by compilation.
   > verification: `rg -q "eframe" Cargo.toml` and `rg -q "egui" Cargo.toml`
3. Create `crates/devil-desktop/Cargo.toml` with package metadata inherited from the workspace and dependencies on `anyhow`, `devil-app`, `devil-protocol`, `devil-ui`, `eframe`, `egui`, and `thiserror`.
   > verification: `rg -q "devil-app" crates/devil-desktop/Cargo.toml`
4. Run a metadata/check command that updates `Cargo.lock` after adding the package and dependencies.
   > verification: `Test-Path Cargo.lock`
5. If the dependency policy still describes `devil-desktop` only as a future planned crate, update that wording to describe the active Phase 2 adapter without widening permissions.
   > verification: `rg -q "devil-desktop" plans/dependency-policy.md`

### Content Structure

- Keep root workspace dependency declarations grouped with other external crates.
- Keep `devil-desktop` dependency declarations in the desktop crate manifest; do not add renderer dependencies to `devil-ui`.
- Preserve existing dependency-policy section numbering and parser-compatible wording.

### Edge And Error Cases

- If `eframe` pulls `accesskit`, `winit`, or `wgpu`, this is allowed only through `devil-desktop` and must remain caught by non-adapter deny checks.
- If Cargo selects a version that conflicts with Rust 2024 workspace settings, use the latest compatible version that compiles locally and record it in the result.
- Do not add `rfd`, `tauri`, `wry`, `tao`, `slint`, or `gpui` in this plan.

- Dependency policy changes would need to authorize renderer dependencies outside `devil-desktop`.

### Implementation Sequence

1. Create `src/lib.rs` with public modules `view`, `bridge`, `workflow`, `metrics`, and `smoke`; export only desktop-adapter types, not editor/workspace/storage internals.
   > verification: `rg -q "pub mod view" crates/devil-desktop/src/lib.rs`
2. Create `src/main.rs` that delegates to a `workflow::run_from_env()` function and returns `anyhow::Result<()>`.
   > verification: `rg -q "run_from_env" crates/devil-desktop/src/main.rs`
3. Create stub modules with compile-safe public placeholders named `ProjectionView`, `DesktopCommandBridge`, `DesktopRuntime`, `FrameTimingRecorder`, and `run_smoke_from_args`.
   > verification: `rg -q "DesktopCommandBridge" crates/devil-desktop/src/bridge.rs`
4. Ensure stubs depend only on `devil-app`, `devil-ui`, `devil-protocol`, `eframe`, `egui`, and standard library APIs already declared in the manifest.
   > verification: `cargo check -p devil-desktop --all-targets`

### Content Structure

- `lib.rs`: module declarations and concise crate-level adapter-boundary comment.
- `main.rs`: no renderer or app authority logic beyond delegating to workflow.
- Stub modules: minimal structs/functions with explicit `todo`-free no-op behavior that compiles and can be replaced by later waves.

### Edge And Error Cases

- Stubs must not call `AppComposition::new()` yet except where a compile-safe constructor needs type references.
- Do not add `#[allow(dead_code)]` at crate scope. Prefer small public placeholders or tests that exercise stubs.
- Do not introduce mock editor/workspace state in `devil-desktop`.

### Verification Commands

```powershell

```

<verify>

The scaffold is complete only when the package compiles without weakening lint or architecture checks.

</verify>

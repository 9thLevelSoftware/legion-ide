# Plan 02-06: Phase 2 Evidence And Acceptance Gate

## Objective

Close Phase 2 with a source-backed evidence package that records whether the renderer-backed foundation shell is accepted, blocked, or accepted with explicitly named residual risks.

<execution_contract>

- read-before-write: read every Phase 2 result artifact, all Phase 2 code touched by prior plans, Phase 1 readiness, smoke evidence, dependency policy, and current command outputs before writing acceptance evidence.
- evidence-before-action: record a command as `passed` only after running it in this checkout and seeing exit code 0.
- minimal diff: write the final Phase 2 evidence document only.
- verify-before-report: run all required gates or mark the document blocked with exact failures.
- role: act as skeptical release/evidence reviewer.
- task: determine whether Phase 2 success criteria are satisfied by artifacts and live verification.
- scope: evidence inventory, command results, Phase 2 success criteria, residual risks, and Phase 3 entry criteria.
- allowed tools/actions: read artifacts/source, run verification commands, write final evidence.
- forbidden actions: editing code, fixing tests, changing policy, weakening acceptance criteria, or marking blocked checks as accepted.
- verification criteria: evidence document includes all required command results and rejects acceptance if native smoke or gates fail.
- final result format: write plan result with acceptance verdict and exact verification table.

</execution_contract>

<stop_gates>

- BLOCKED if any upstream Phase 2 result artifact is missing.
- BLOCKED if the desktop crate cannot compile or the renderer dependency gate fails.
- BLOCKED if smoke evidence is missing required timing/platform fields.
- BLOCKED if the final document cannot truthfully state every Phase 2 roadmap success criterion is met or explicitly blocked.

</stop_gates>

<recovery>

- If a verification command fails because of an in-scope Phase 2 issue, mark final evidence blocked and route back to the owning plan.
- If a verification command fails due an unrelated pre-existing issue, include evidence, explain why it is unrelated, and do not mark the command passed.
- If work is interrupted, capture `git diff`, modified files, verification state, and `02-06-RESULT.md` status.

</recovery>

<verification>

Run the frontmatter verification commands plus the broader phase gates listed in Task 2. The plan is complete only when the evidence document truthfully reflects command results.

</verification>

<success_criteria>

- Final evidence names every Phase 2 artifact, command result, and success criterion.
- Acceptance is not claimed unless `devil-desktop` compiles, workflow tests pass, renderer smoke evidence exists, and dependency gates pass.
- Residual risk distinguishes foundation-mode limitations from later Phase 3/6 obligations.

</success_criteria>

<output>

Write `.planning/phases/02-renderer-backed-foundation-mode/02-06-RESULT.md` with final verdict, verification results, and next recommended Legion command.

</output>

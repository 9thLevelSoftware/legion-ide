# Plan 02-02: Projection Renderer Panels

## Objective

Render the current `ShellProjectionSnapshot` in `devil-desktop` using egui panels without moving projection state or business authority into UI.

<execution_contract>

- read-before-write: read `crates/devil-ui/src/ui.rs`, `crates/devil-app/src/lib.rs` projection builders, ADR-0030, and the desktop stubs from Plan 02-01.
- evidence-before-action: every displayed field must come from `ShellProjectionSnapshot` or adapter-local presentation state.
- minimal diff: edit only the renderer view module and its tests.
- verify-before-report: run the projection-rendering test and desktop package check.
- role: act as frontend implementer preserving the projection boundary.
- task: turn protocol/UI projection DTOs into egui-visible panels and a testable view model.
- scope: layout, explorer, active buffer, status, proposal, context manifest, privacy, permission budget, approval checklist, checkpoint/rollback, assisted AI, plugin, delegated task, and collaboration projection summaries.
- allowed tools/actions: create pure view-model helpers, egui rendering functions, and renderer tests.
- forbidden actions: opening files, saving, mutating editor buffers, dispatching commands, storing full source text outside `ShellProjectionSnapshot`, or editing `devil-ui`.
- verification criteria: tests prove all required Phase 2 surfaces appear in the view model for populated and empty snapshots.
- final result format: write a plan result with rendered surfaces, tests, and residual visual limitations.

</execution_contract>

<stop_gates>

- BLOCKED if `ShellProjectionSnapshot` lacks enough fields to render one of the required Phase 2 surfaces and adding projection fields would be required.
- BLOCKED if egui rendering requires modifying `devil-ui` or app internals.
- BLOCKED if tests cannot construct representative projection data without copying private app state.

</stop_gates>

<recovery>

- If a projection field is empty in current app output, render a clear empty state derived from the projection type rather than fabricating app state.
- If egui widget APIs differ from assumptions, keep the pure `DesktopProjectionViewModel` stable and adjust only the egui draw helper.
- If work is interrupted, capture `git diff`, modified files, verification state, and `02-02-RESULT.md` status.

</recovery>

<verification>

Run the frontmatter verification commands. The projection renderer is not complete until required surfaces are asserted by tests and package check passes.

</verification>

<success_criteria>

- A native window can call one renderer function with a `ShellProjectionSnapshot` and display all required Phase 2 surfaces.
- Empty/degraded states are visible and do not panic.
- The renderer does not dispatch commands or own product state.

</success_criteria>

<output>

Write `.planning/phases/02-renderer-backed-foundation-mode/02-02-RESULT.md` with surfaces implemented, screenshots or smoke notes if available, and exact test output.

</output>

## Task 1: Build A Testable Projection View Model

### Read Targets

- `crates/devil-ui/src/ui.rs`
- `crates/devil-app/src/lib.rs`
- `plans/desktop-adapter-boundary-v0.1.md`
- `crates/devil-desktop/src/view.rs`

### Write Targets

- `crates/devil-desktop/src/view.rs`

### Implementation Sequence

1. Define `DesktopProjectionViewModel` with explicit sections: `layout_title`, `explorer_rows`, `active_buffer_lines`, `status_rows`, `proposal_rows`, `trust_rows`, `assistant_rows`, `plugin_rows`, `collaboration_rows`, and `empty_or_degraded_flags`.
   > verification: `rg -q "DesktopProjectionViewModel" crates/devil-desktop/src/view.rs`
2. Add `DesktopProjectionViewModel::from_snapshot(snapshot: &ShellProjectionSnapshot) -> Self`.
   > verification: unit/integration tests can call it without an egui context.
3. Map explorer nodes from `ExplorerProjection.nodes`, active buffer content from `ActiveBufferProjection::small_buffer_text()` or viewport line slices, and dirty/degraded/large-file status from projection fields.
   > verification: tests assert active buffer line text, dirty marker, and degraded marker.
4. Map proposal and trust summaries from `proposal_ledger_projection`, `context_manifest_projection`, `privacy_inspector_projection`, `permission_budget_projection`, `approval_checklist_projection`, and `checkpoint_rollback_projection`.
   > verification: tests assert a proposal row and at least one trust row for populated fixtures.
5. Map plugin, delegated-task, collaboration, and assisted-AI projections as read-only summary rows.
   > verification: tests assert rows remain empty for empty projections and non-empty for supplied fixtures.

### Content Structure

- Keep the view-model structs in `view.rs`; do not add them to `devil-ui`.
- Use owned strings and simple booleans in the view model so tests do not depend on egui internals.
- Use helper functions by projection domain only when they reduce repeated formatting.

### Edge And Error Cases

- Empty explorer: render an empty-state row, not a panic.
- Missing active buffer: render `<no active buffer>`.
- Large/degraded active buffer: show the visible viewport lines and degraded marker; do not attempt to reconstruct hidden full text.
- Redaction hints: surface redaction labels as metadata-only rows; do not print secrets or raw provider payloads.

### Verification Commands

```powershell
rg -q "DesktopProjectionViewModel" crates/devil-desktop/src/view.rs
```

<verify>

The view model is complete only when each required Phase 2 surface has a named field and a deterministic conversion from the snapshot.

</verify>

### BLOCKED Conditions

- Required surfaces cannot be represented from current projection DTOs.

## Task 2: Render Egui Panels From The View Model

### Read Targets

- `crates/devil-desktop/src/view.rs`
- `plans/adrs/ADR-0030-desktop-adapter-boundary.md`

### Write Targets

- `crates/devil-desktop/src/view.rs`

### Implementation Sequence

1. Define `ProjectionView` with adapter-local UI state limited to scroll/focus hints and collapsed-panel flags.
   > verification: `rg -q "ProjectionView" crates/devil-desktop/src/view.rs`
2. Implement `ProjectionView::render(&mut self, ctx: &egui::Context, snapshot: &ShellProjectionSnapshot) -> ProjectionViewOutput`.
   > verification: `cargo check -p devil-desktop --all-targets`
3. Use egui top/bottom/side/central panels for title/status, explorer, active buffer viewport, proposal/trust summaries, and auxiliary projection summaries.
   > verification: tests assert the view model; cargo check proves egui API use.
4. Return a `ProjectionViewOutput` containing only adapter-local flags such as `needs_repaint` or selected display ids; do not return app mutations from this plan.
   > verification: `rg -q "ProjectionViewOutput" crates/devil-desktop/src/view.rs`

### Content Structure

- Top panel: title, dirty marker, current path if present, save/conflict summary text supplied by workflow later.
- Left panel: explorer rows and selection marker.
- Central panel: active buffer visible text or empty/degraded state.
- Bottom/right panels: status, proposal ledger, trust/privacy/permission/checkpoint summaries, and compact auxiliary projections.

### Edge And Error Cases

- Long text must wrap or clip inside egui scroll areas; avoid layout code that grows unbounded.
- No nested product-state caches. Scroll offsets and collapsed flags are allowed; file ids, buffer ids, save status, and dirty state remain projection data.
- Do not create command buttons that dispatch app actions in this plan; Plan 02-03 owns intent translation.

### Verification Commands

```powershell
cargo check -p devil-desktop --all-targets
```

<verify>

The renderer is complete only when egui drawing compiles and all user-visible content is derived from `ShellProjectionSnapshot` or adapter-local display flags.

</verify>

### BLOCKED Conditions

- egui panel implementation requires storing editor-owned state in `ProjectionView`.

## Task 3: Add Projection Rendering Regression Tests

### Read Targets

- `crates/devil-ui/src/ui.rs`
- Existing `devil-ui` tests in `crates/devil-ui/src/ui.rs`
- `crates/devil-desktop/src/view.rs`

### Write Targets

- `crates/devil-desktop/tests/projection_rendering.rs`

### Implementation Sequence

1. Build fixture helpers that create `ShellProjectionSnapshot` values using public `devil-ui` and `devil-protocol` types.
   > verification: tests compile without importing app/editor/workspace internals.
2. Add `projection_rendering_populates_required_phase2_surfaces` to assert layout, explorer, active buffer, status, proposal, and trust rows.
   > verification: `cargo test -p devil-desktop projection_rendering --test projection_rendering`
3. Add `projection_rendering_handles_empty_and_degraded_snapshots` to assert empty explorer, missing active buffer, and degraded active buffer states.
   > verification: same targeted test command.
4. Add a boundary assertion that tests use only `devil-ui`, `devil-protocol`, and `devil-desktop` imports.
   > verification: inspect imports and keep `devil-app` out of this test file.

### Content Structure

- Use one helper for an empty snapshot and one helper for a populated snapshot.
- Keep expected text stable and user-visible; avoid asserting egui internals.

### Edge And Error Cases

- If constructing full proposal/trust fixtures is verbose, use the existing public empty/default projection helpers where available and explicitly populate one row per required surface.
- Do not duplicate large fixture blocks from `devil-ui` tests if a smaller public projection fixture is enough.

### Verification Commands

```powershell
cargo test -p devil-desktop projection_rendering --test projection_rendering
cargo check -p devil-desktop --all-targets
```

<verify>

Projection rendering is complete only when tests fail if any required Phase 2 surface disappears from the view model.

</verify>

### BLOCKED Conditions

- Public projection constructors are insufficient to build test snapshots without app-private state.

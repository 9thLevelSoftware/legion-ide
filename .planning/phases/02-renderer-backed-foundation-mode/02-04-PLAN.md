# Plan 02-04: App Composition Desktop Workflow

## Objective

Connect the egui desktop shell to `AppComposition` so users can open this repository, open a file, edit a small buffer, save, see rejection/conflict state, and quit through app-owned authority.

<execution_contract>

- read-before-write: read `AppComposition` open/save/dispatch APIs, `Shell` projection replacement, `DesktopCommandBridge`, `ProjectionView`, and CLI entry point behavior.
- evidence-before-action: all mutation requests must route through `AppComposition::dispatch_ui_intent`, `AppComposition::open_workspace`, `AppComposition::open_file`, or `AppComposition::save_active_buffer`.
- minimal diff: edit desktop workflow/main/test files only.
- verify-before-report: run desktop workflow tests and package check.
- role: act as application integrator for the desktop adapter.
- task: own the renderer loop and app composition boundary without moving app authority into UI.
- scope: startup arguments, trusted workspace open, optional file open, projection refresh, command dispatch, save outcome display, quit handling, and test-only workflow harness.
- allowed tools/actions: use `AppComposition`, `Shell`, `DesktopCommandBridge`, `ProjectionView`, egui/eframe, and test temp files.
- forbidden actions: direct filesystem saves from desktop, direct editor mutation outside `dispatch_ui_intent`, app code edits, or renderer dependencies outside `devil-desktop`.
- verification criteria: tests prove open/edit/save success and external-overwrite save rejection while dirty text remains app-owned and visible.
- final result format: write result with workflow paths, commands, and known UI limitations.

</execution_contract>

<stop_gates>

- BLOCKED if `AppComposition` cannot open a workspace and file from desktop without changing app APIs.
- BLOCKED if save rejection cannot be surfaced without treating rejected saves as success.
- BLOCKED if eframe requires renderer callbacks that would own editor text or save state.

</stop_gates>

<recovery>

- If the real eframe loop is hard to exercise in tests, create a non-rendering `DesktopRuntime` harness that uses the same command dispatch and projection refresh code as the eframe app.
- If external overwrite tests are flaky due file timestamp resolution, use content/fingerprint-changing writes and the existing app save outcome instead of sleeps.
- If work is interrupted, capture `git diff`, modified files, verification state, and `02-04-RESULT.md` status.

</recovery>

<verification>

Run the frontmatter verification commands. The workflow is complete only when the non-rendering harness proves app-owned open/edit/save/reject paths and the eframe app compiles.

</verification>

<success_criteria>

- `cargo run -p devil-desktop -- . Cargo.toml` opens a native window for this repo and file.
- User actions in the window route through `DesktopCommandBridge` and `AppComposition`.
- Save success and save rejection are visible as adapter status while editor dirty state remains app-owned.

</success_criteria>

<output>

Write `.planning/phases/02-renderer-backed-foundation-mode/02-04-RESULT.md` with run command, tested workflows, and rejection evidence.

</output>

## Task 1: Implement Desktop Runtime Startup

### Read Targets

- `crates/devil-app/src/lib.rs`
- `crates/devil-app/src/main.rs`
- `crates/devil-ui/src/ui.rs`
- `crates/devil-desktop/src/workflow.rs`
- `crates/devil-desktop/src/main.rs`

### Write Targets

- `crates/devil-desktop/src/workflow.rs`
- `crates/devil-desktop/src/main.rs`

### Implementation Sequence

1. Define `DesktopLaunchConfig` with `workspace_root: PathBuf`, `initial_file: Option<String>`, `principal: PrincipalId`, and smoke/test flags reserved for Plan 02-05.
   > verification: `rg -q "DesktopLaunchConfig" crates/devil-desktop/src/workflow.rs`
2. Implement `DesktopLaunchConfig::from_env_args()` with positional args: first arg workspace root defaulting to current directory, second arg initial file defaulting to `None`.
   > verification: `cargo check -p devil-desktop --all-targets`
3. Implement `DesktopRuntime::open(config)` that creates `AppComposition::new()`, calls `open_workspace(root, WorkspaceTrustState::Trusted, principal)`, optionally calls `open_file(initial_file)`, builds `shell_projection_snapshot`, and creates `Shell`.
   > verification: desktop workflow tests can open a temp workspace and file.
4. Update `main.rs` to call `workflow::run_from_env()` and return errors instead of panicking.
   > verification: `rg -q "run_from_env" crates/devil-desktop/src/main.rs`

### Content Structure

- Keep argument parsing small and dependency-free; do not add `clap` unless later phase needs richer CLI.
- Keep startup errors typed or wrapped in `anyhow::Result`.
- Use `PrincipalId("desktop".to_string())` for the default desktop principal.

### Edge And Error Cases

- Missing initial file should still open the workspace and show an empty active buffer state.
- Invalid workspace path must return an error; do not silently open current directory.
- File-open failure must display or return an app error; do not fabricate an empty buffer unless `open_new_file` is explicitly used later.

### Verification Commands

```powershell
cargo check -p devil-desktop --all-targets
```

<verify>

Startup is complete only when `DesktopRuntime::open` produces a fresh projection through `AppComposition::shell_projection_snapshot`.

</verify>

### BLOCKED Conditions

- `AppComposition::open_workspace` or `open_file` cannot be called from the desktop crate under policy-approved dependencies.

## Task 2: Wire Eframe Update Loop Through Bridge And App Authority

### Read Targets

- `crates/devil-desktop/src/view.rs`
- `crates/devil-desktop/src/bridge.rs`
- `crates/devil-desktop/src/workflow.rs`
- `crates/devil-app/src/lib.rs`

### Write Targets

- `crates/devil-desktop/src/workflow.rs`

### Implementation Sequence

1. Implement `DesktopRuntime::handle_action(action: DesktopAction) -> DesktopWorkflowOutcome` that calls `DesktopCommandBridge::translate`, then dispatches intents through `AppComposition::dispatch_ui_intent`.
   > verification: `rg -q "dispatch_ui_intent" crates/devil-desktop/src/workflow.rs`
2. On `CommandDispatchIntent::Quit`, mark an adapter-local `quit_requested` flag and do not call app mutation APIs.
   > verification: workflow tests assert quit outcome.
3. On save outcomes, store adapter-local display status `Saved`, `SaveRejected`, or `CommandError`; do not override editor dirty state manually.
   > verification: workflow tests assert rejected save status and projection dirty flag.
4. After every handled command, rebuild `ShellProjectionSnapshot` through `AppComposition::shell_projection_snapshot` and call `Shell::replace_projection_snapshot`.
   > verification: workflow tests assert projection changes after edit.
5. Implement `eframe::App` for the desktop app using `ProjectionView::render`, menu/key/path prompt controls, and `DesktopRuntime::handle_action`.
   > verification: `cargo check -p devil-desktop --all-targets`

### Content Structure

- `DesktopRuntime`: owns `AppComposition`, `Shell`, `DesktopCommandBridge`, `ProjectionView`, last workflow status, open-path prompt state, and quit flag.
- `DesktopWorkflowOutcome`: records `Opened`, `Edited`, `Saved`, `SaveRejected`, `Noop`, `QuitRequested`, or `Error`.
- `eframe::App::update`: render first from current projection, collect adapter actions, then dispatch and request repaint.

### Edge And Error Cases

- Save rejection must be visible and must not mark dirty text clean.
- Command errors must be visible but must not imply success.
- Path prompt cancellation emits no app mutation.
- Clipboard/IME actions should route through bridge abstractions added in Plan 02-05 if they are not needed for basic editing in this plan.

### Verification Commands

```powershell
rg -q "dispatch_ui_intent" crates/devil-desktop/src/workflow.rs
cargo check -p devil-desktop --all-targets
```

<verify>

The update loop is complete only when every mutation path crosses `DesktopCommandBridge` and `AppComposition`.

</verify>

### BLOCKED Conditions

- Eframe update handling requires bypassing bridge/app authority to mutate buffer text.

## Task 3: Add Open/Edit/Save/Rejection Workflow Tests

### Read Targets

- Existing app integration tests involving workspace overwrite/save conflict
- `crates/devil-app/src/lib.rs`
- `crates/devil-desktop/src/workflow.rs`
- `crates/devil-desktop/src/bridge.rs`

### Write Targets

- `crates/devil-desktop/tests/desktop_workflow.rs`

### Implementation Sequence

1. Create temp workspace helpers using `std::env::temp_dir()` plus a unique directory name; clean up only the exact directory created by the test.
   > verification: tests create paths under the temp root, not the repository.
2. Add `desktop_workflow_opens_repo_file_and_edits_small_buffer` that opens a file, dispatches insert text, verifies projection dirty state, dispatches save, and verifies on-disk text changed.
   > verification: `cargo test -p devil-desktop desktop_workflow --test desktop_workflow`
3. Add `desktop_workflow_external_overwrite_save_rejects_and_preserves_dirty_projection` that opens a file, edits it, externally overwrites the file, dispatches save, asserts `SaveRejected`, and asserts the active buffer projection remains dirty.
   > verification: same targeted test command.
4. Add `desktop_workflow_quit_sets_quit_flag_without_app_mutation` to prove quit is adapter-local.
   > verification: same targeted test command.

### Content Structure

- Keep tests in `crates/devil-desktop/tests/desktop_workflow.rs`.
- Use the public desktop runtime harness, not eframe internals.
- Assert app-visible outcomes, projection dirty state, and file content.

### Edge And Error Cases

- Test cleanup must never recursively delete a computed path unless it is verified to be under `std::env::temp_dir()` and has the unique test prefix.
- Avoid time-based sleeps for conflict detection unless source evidence proves timestamps are required.
- If an external overwrite does not trigger conflict, inspect current app save preconditions before changing test expectations.

### Verification Commands

```powershell
cargo test -p devil-desktop desktop_workflow --test desktop_workflow
cargo check -p devil-desktop --all-targets
```

<verify>

The workflow is complete only when tests demonstrate both successful save and rejected save paths through app-owned authority.

</verify>

### BLOCKED Conditions

- App save workflow cannot expose dirty/rejection state through public projection/outcome APIs.

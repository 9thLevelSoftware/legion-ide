# Plan 01-04: Dependency Policy And Xtask Renderer Gate

## Objective

Authorize the desktop adapter dependency boundary in policy and enforce it with `xtask` so renderer dependencies cannot enter `devil-ui` or core crates.

<execution_contract>

- read-before-write: read ADR-0002, ADR-0030, dependency policy, current manifests, and `xtask/src/main.rs` before editing.
- evidence-before-action: only add renderer dependency allowances that ADR-0002 explicitly authorizes.
- minimal diff: edit dependency policy and `xtask` tests/checks only.
- verify-before-report: run the exact targeted `xtask` test and `xtask check-deps` before reporting completion.

</execution_contract>

<stop_gates>

- BLOCKED if ADR-0002 has no accepted renderer path and no explicit renderer dependency names or categories.
- BLOCKED if ADR-0030 does not define the desktop adapter boundary.
- BLOCKED if enforcing the renderer gate requires modifying existing crates or manifests in this plan.

</stop_gates>

<recovery>

- If the selected renderer dependency list is not final, add a policy section that permits renderer dependencies only in the planned adapter after a named ADR update; keep `devil-ui` denial active.
- If an existing `xtask` parser cannot express the new rule cleanly, add the smallest dedicated test/check helper rather than weakening current checks.
- If `cargo test -p xtask` exposes unrelated failures, capture evidence and still ensure the new exact test passes or reports BLOCKED.

</recovery>

## Task 1: Update Dependency Policy For Desktop Adapter

### Read Targets

- `plans/dependency-policy.md`
- `plans/adrs/ADR-0002-ui-editor-rendering.md`
- `plans/adrs/ADR-0030-desktop-adapter-boundary.md`
- `Cargo.toml`
- `crates/devil-ui/Cargo.toml`
- `crates/devil-app/Cargo.toml`

### Write Targets

- `plans/dependency-policy.md`

### Implementation Sequence

1. Add a `devil-desktop` policy entry in the same style as existing crate policy entries.
2. State allowed internal dependencies: `devil-app`, `devil-ui`, and `devil-protocol`.
3. State renderer dependency allowance using the exact names or categories accepted by ADR-0002.
4. State forbidden directions: no core crate may depend on `devil-desktop`; `devil-ui` must not depend on renderer crates, app, editor, project, storage, workspace, provider, or telemetry storage crates.
5. Add a policy note that renderer crates are adapter-only until a later ADR and dependency-policy update says otherwise.

### Content Structure

- Keep existing dependency policy section numbering stable unless the file already uses generated numbering.
- Add the adapter entry near existing crate dependency rules.
- Add a short enforcement note under the section parsed by `xtask check-deps` if the parser requires it.

### Edge And Error Cases

- If the policy parser depends on exact headings, update `xtask` and its tests together.
- Do not authorize wildcard renderer dependencies if ADR-0002 selected a specific crate set.
- Do not add `devil-desktop` to workspace manifests in this plan.

### Verification Commands

```powershell
rg -q "devil-desktop" plans/dependency-policy.md
rg -q "renderer" plans/dependency-policy.md
```

### BLOCKED Conditions

- ADR-0002 does not provide enough specificity to write an enforceable policy.

## Task 2: Add Xtask Renderer Dependency Gate

### Read Targets

- `xtask/src/main.rs`
- `plans/dependency-policy.md`
- `crates/devil-ui/Cargo.toml`
- `Cargo.toml`

### Write Targets

- `xtask/src/main.rs`

### Implementation Sequence

1. Locate existing dependency-policy constants, parser helpers, and tests.
2. Add a check or test named `renderer_dependency_gate_preserves_projection_boundary`.
3. Enforce that `crates/devil-ui/Cargo.toml` does not include renderer/windowing dependencies named by ADR-0002/policy.
4. Enforce that the policy documents the desktop adapter renderer boundary.
5. Keep the existing `ui_shell_remains_projection_only` behavior intact.

### Content Structure

- Prefer a focused helper near existing dependency-policy checks.
- Prefer an exact unit test near current architecture policy tests.
- Use explicit dependency names or policy-derived constants; do not use fragile broad string matching unless the existing file already does.

### Edge And Error Cases

- If renderer dependency names include optional platform crates, include every policy-authorized renderer crate in the deny list for `devil-ui`.
- If no renderer crate is finalized, test for the policy gate and a conservative known-renderer deny list.
- Do not introduce `as any`-style type escapes or broad error swallowing.

### Verification Commands

```powershell
rg -q "renderer_dependency_gate_preserves_projection_boundary" xtask/src/main.rs
cargo test -p xtask renderer_dependency_gate_preserves_projection_boundary -- --exact
```

### BLOCKED Conditions

- The current `xtask` architecture cannot test manifest boundaries without a broader rewrite.

# Plan 01-05: Phase 1 Evidence And Readiness Gate

## Objective

Close Phase 1 with evidence that baseline reconciliation, renderer decision, adapter boundary, and dependency gates are complete enough for Phase 2 GUI shell implementation.

<execution_contract>

- read-before-write: read all Phase 1 artifacts and current gate outputs before creating readiness evidence.
- evidence-before-action: record command results only after running the commands in this checkout.
- minimal diff: add the readiness document only.
- verify-before-report: do not mark readiness accepted unless every required gate passes and every upstream Phase 1 artifact is present.

</execution_contract>

<stop_gates>

- BLOCKED if any upstream Phase 1 plan artifact is missing.
- BLOCKED if renderer decision, adapter boundary, or dependency gate remains unresolved.
- BLOCKED if required verification commands fail and the failure is in scope for Phase 1 closure.

</stop_gates>

<recovery>

- If a gate fails, record the exact command and failure summary in the readiness document and mark `Phase 1 readiness: Blocked`.
- If an upstream artifact exists but lacks required evidence, route the executor back to the owning plan instead of filling the gap here.
- If broad workspace checks are slow, keep the command running until it exits or capture timeout details honestly.

</recovery>

## Task 2: Run And Record Phase 1 Gates

### Read Targets

- `Cargo.toml`
- `plans/dependency-policy.md`
- `xtask/src/main.rs`
- `crates/devil-ui/Cargo.toml`
- `crates/devil-app/Cargo.toml`

### Write Targets

- `plans/evidence/gui-productization/phase-1-renderer-readiness.md`

### Implementation Sequence

1. Run `cargo run -p xtask -- check-deps`.
2. Run `cargo fmt --all --check`.
3. Run `cargo check --workspace --all-targets`.
4. Run `cargo test -p xtask`.
5. Run `cargo check -p devil-ui --all-targets`.
6. Run `cargo check -p devil-app --all-targets`.
7. Record each command as `passed` only if it exited successfully in this checkout.

### Content Structure

- Under `## Gate Results`, include one line per command in the form `<command>: passed`.
- If a command fails, include `<command>: failed` and a concise failure summary.

### Edge And Error Cases

- If `cargo check --workspace --all-targets` fails for an unrelated pre-existing issue, record the failure and decide whether it blocks Phase 1 readiness; do not hide it.
- If command output contains sensitive local paths only, paths are acceptable in evidence; never include secrets.
- Do not edit source code in this plan to make gates pass.

### Verification Commands

```powershell
rg -q "cargo run -p xtask -- check-deps: passed" plans/evidence/gui-productization/phase-1-renderer-readiness.md
rg -q "cargo check -p devil-ui --all-targets: passed" plans/evidence/gui-productization/phase-1-renderer-readiness.md
```

### BLOCKED Conditions

- Any required gate fails and there is no approved owner plan to fix it.

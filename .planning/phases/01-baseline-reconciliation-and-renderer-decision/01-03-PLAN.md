# Plan 01-03: Desktop Adapter Boundary Specification

## Objective

Specify the GUI desktop adapter boundary so renderer work can start without moving session, text, workspace, save, provider, or telemetry ownership into UI.

<execution_contract>

- read-before-write: read the baseline, renderer decision matrix, ADR-0002, `devil-ui`, `devil-app`, and dependency policy before writing.
- evidence-before-action: base every boundary rule on an existing source responsibility or a documented renderer requirement.
- minimal diff: add the new ADR and boundary spec only.
- verify-before-report: run every verification command and do not report success if adapter ownership is ambiguous.

</execution_contract>

<stop_gates>

- BLOCKED if plan 01-01 did not establish an accepted baseline.
- BLOCKED if plan 01-02 did not identify a renderer path or explicit `BLOCKED` renderer reason.
- BLOCKED if the adapter spec cannot preserve projection-only UI and proposal-mediated saves.

</stop_gates>

<recovery>

- If the renderer decision is blocked, still write the adapter boundary using renderer-neutral protocol terms and mark renderer-specific entries as blocked inputs.
- If a boundary rule conflicts with current code, cite the code and stop rather than inventing a new ownership model.
- If security review identifies a privilege boundary gap, add it to the spec's stop gates instead of burying it in prose.

</recovery>

### Implementation Sequence

1. Create ADR-0030 with status `Accepted` only if plans 01-01 and 01-02 are accepted; otherwise use `Blocked` with the exact missing input.
2. Define the planned desktop adapter crate name as `devil-desktop`.
3. State that `devil-desktop` may depend on `devil-app`, `devil-ui`, `devil-protocol`, and renderer crates authorized by ADR-0002 and dependency policy.
4. State that core crates must not depend on `devil-desktop`.
5. State that `devil-desktop` must not own editor text, workspace state, save decisions, provider credentials, telemetry storage, or persistence policy.

- `## Verification`

### Edge And Error Cases

- If `devil-desktop` already exists by execution time, inspect it and write the ADR against actual code.
- If the renderer decision names a different adapter crate, use the ADR-0002 name consistently and explain the deviation.
- Do not create or edit the crate in this plan.

## Task 2: Write Adapter Boundary Contract

### Read Targets

- `plans/adrs/ADR-0030-desktop-adapter-boundary.md`
- `plans/adrs/ADR-0002-ui-editor-rendering.md`
- `plans/evidence/gui-productization/gui-productization-baseline.md`
- `crates/devil-ui/src/lib.rs`
- `crates/devil-app/src/lib.rs`
- `crates/devil-protocol/src/lib.rs`

### Write Targets

- `plans/desktop-adapter-boundary-v0.1.md`

### Implementation Sequence

1. Define startup flow from process entry to trusted workspace open, initial projection, render loop, command dispatch, app state update, and re-projection.
2. Define the inbound projection type family using existing protocol names such as `ShellProjectionSnapshot`.
3. Define outbound intent flow through `CommandDispatchIntent`.
4. Define allowed side effects: window lifecycle, renderer resources, native input, clipboard integration, file dialog request forwarding, and accessibility tree publication.
5. Define forbidden side effects: direct file saves, editor text ownership, workspace mutation, provider runtime execution, telemetry storage mutation, secret persistence, and bypassing save proposals.

### Content Structure

- `# Desktop Adapter Boundary v0.1`
- `## Scope`
- `## Startup Flow`
- `## Projection In`
- `## Intent Out`
- `## Allowed Side Effects`
- `## Forbidden Side Effects`
- `## Error And Recovery Semantics`
- `## Test Harness Requirements`
- `## Phase 2 Entry Criteria`

### Edge And Error Cases

- File dialogs must request app/workspace actions; they must not directly mutate the workspace.
- Clipboard and IME integration must preserve command correlation where an app command is emitted.
- Renderer crashes must not imply successful save or accepted command execution.

### Verification Commands

```powershell

rg -q "CommandDispatchIntent" plans/desktop-adapter-boundary-v0.1.md
rg -q "projection-only|must not own editor|must not own workspace" plans/desktop-adapter-boundary-v0.1.md
```

### BLOCKED Conditions

- Existing protocol types cannot represent the minimum shell projection and intent flow.

- `plans/desktop-adapter-boundary-v0.1.md`

### Implementation Sequence

1. Review the adapter boundary for secret, filesystem, telemetry, save, and provider execution risks.
2. Add explicit stop gates for any risk that cannot be handled in Phase 2 implementation.
3. Confirm the spec routes all persistent mutations through existing app/workspace services.
4. Confirm the ADR links dependency-policy enforcement in plan 01-04.

### Content Structure

- Add a `Security And Persistence Constraints` subsection if missing.
- Add a `Stop Gates` subsection if any boundary risk remains.

### Edge And Error Cases

- Do not give the renderer direct authority to save files for convenience.
- Do not grant the adapter durable storage unless a later ADR authorizes it.

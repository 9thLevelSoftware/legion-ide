# Plan 01-01: Baseline Ledger Reconciliation And GUI Baseline

## Objective

Reconcile the planning record with accepted substrate evidence, then create the baseline evidence document that all GUI productization work must preserve.

<execution_contract>

- read-before-write: read the ledger, Phase 8 evidence, ADR-0002, SPIKE-001A, dependency policy, and current crate manifests before editing.
- evidence-before-action: every status change must cite current repository evidence, not memory or assumption.
- minimal diff: update only the ledger text needed for Phase 8 reconciliation and add the new GUI baseline document.
- verify-before-report: run every verification command in this plan before reporting this plan complete.

</execution_contract>

<stop_gates>

- BLOCKED if Phase 8 acceptance evidence cannot be found under `plans/evidence/phase-8/`.
- BLOCKED if ledger evidence and Phase 8 evidence cannot be reconciled without changing the roadmap or code.
- BLOCKED if any verification command fails after one focused correction pass.

</stop_gates>

<recovery>

- If the ledger contains multiple conflicting Phase 8 entries, preserve all historical context and add a dated reconciliation note instead of deleting context.
- If the GUI baseline document cannot assert a fact from current files, mark that line as a gap with the exact file searched.
- If `cargo run -p xtask -- check-deps` fails, capture the failure verbatim and do not label the plan accepted.

</recovery>

## Task 1: Reconcile Phase 8 Ledger Status

### Read Targets

- `plans/phase-status-ledger.md`
- `plans/evidence/phase-8/phase-8-architecture-map.md`
- `plans/evidence/phase-8/release-readiness-review.md`
- `plans/evidence/phase-8/platform-matrix-evidence.txt`
- `plans/adrs/ADR-0002-ui-editor-rendering.md`
- `plans/spikes/SPIKE-001A-result.md`

### Write Targets

- `plans/phase-status-ledger.md`

### Implementation Sequence

1. Locate the Phase 8 ledger entry and any later acceptance notes.
2. Compare the ledger wording with Phase 8 evidence files and ADR acceptance status.
3. Update the Phase 8 ledger entry so it states `Phase 8 acceptance: Accepted`.
4. Add a short dated note that GUI productization is a new post-substrate track and does not reopen accepted Phase 8 runtime hardening.
5. Preserve any historical caveats about renderer-backed latency, IME, clipboard, focus, and accessibility as GUI follow-up evidence, not as current substrate blockers.

### Content Structure

- Keep the ledger's existing structure.
- Add one concise reconciliation paragraph under the Phase 8 area or nearest status summary.
- Include explicit references to the Phase 8 evidence filenames.

### Edge And Error Cases

- If the ledger already has an accepted Phase 8 line, do not duplicate it; update the contradictory line instead.
- If the evidence files disagree, add a `BLOCKED` note in the ledger and stop this plan.
- Do not modify roadmap phase scope in this task.

### Verification Commands

```powershell
rg -q "Phase 8 acceptance: Accepted" plans/phase-status-ledger.md
rg -q "GUI productization" plans/phase-status-ledger.md
```

### BLOCKED Conditions

- No current Phase 8 evidence file supports accepted status.
- Ledger edits would require changing code or roadmap scope to be truthful.

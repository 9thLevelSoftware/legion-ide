# Plan 08-02: Plugin Management And Contribution GUI Workflow

## Objective

Expose plugin management and plugin command contribution workflows in the desktop GUI while preserving app-owned plugin runtime authority, sandbox limits, capability checks, quota checks, and metadata-only audit behavior.

<execution_contract>

- read-before-write: read `08-CONTEXT.md`, `08-01-RESULT.md`, `crates/devil-ui/src/ui.rs`, `crates/devil-app/src/lib.rs`, `crates/devil-desktop/src/bridge.rs`, `crates/devil-desktop/src/workflow.rs`, `crates/devil-desktop/src/view.rs`, `crates/devil-desktop/tests/projection_rendering.rs`, `crates/devil-app/tests/workspace_vfs_integration.rs`, `plans/evidence/gui-productization/phase-8-advanced-platform-gui-ga.md`, and `AGENTS.md` before editing.
- evidence-before-action: confirm `CommandDispatchIntent::InvokePluginCommand` already exists and `AppComposition::invoke_plugin_command` owns command execution before adding desktop plugin actions.
- minimal diff: edit only desktop bridge/workflow/view plugin-management code, a focused desktop regression test file, this plan evidence, and the result file.
- verify-before-report: run every frontmatter command and record outcomes in `08-02-RESULT.md`.
- role: plugin GUI implementer with security-boundary ownership.
- task: make plugin management usable from desktop projections without giving UI or desktop plugin-host authority.
- scope: plugin rows, plugin command routing, invalid plugin/command denial, metadata-only command outcomes, and evidence.
- allowed tools/actions: focused Rust/Markdown edits, targeted desktop/app tests, `rg`, and `xtask` dependency checks.
- forbidden actions: changing plugin runtime contracts, adding dependencies, persisting raw plugin payloads, granting plugin authority to UI/desktop, editing legacy Phase 8 substrate evidence, or weakening existing plugin sandbox/capability/quota tests.
- verification criteria: valid projected plugin commands route to `CommandDispatchIntent::InvokePluginCommand`; unknown plugin ids and unknown command ids are rejected before dispatch; rows expose status and contribution metadata without raw payloads; app plugin tests still prove runtime authority boundaries.
- final result format: write `.planning/phases/08-advanced-platform-gui-ga/08-02-RESULT.md` with changed files, command outcomes, and any `BLOCKED` items.

</execution_contract>

<stop_gates>

- BLOCKED if desktop plugin command routing requires direct calls into `devil-plugin` or bypasses `CommandDispatchIntent`.
- BLOCKED if the plugin manager needs new plugin runtime authority in `devil-ui` or `devil-desktop`.
- BLOCKED if a required plugin row cannot be rendered from `PluginContributionProjection` without adding raw plugin payloads to UI snapshots.
- BLOCKED if invalid plugin or command ids cannot be rejected deterministically from projected state.
- BLOCKED if any change weakens existing sandbox, capability, quota, storage, or metadata-only audit checks.

</stop_gates>

<recovery>

- If projected plugin data lacks enough command identity for safe validation, add a narrow desktop-side validation fallback that denies command dispatch until the projection includes the required command id; record the missing projection field in `08-02-RESULT.md`.
- If app plugin tests expose a runtime regression, stop desktop work and fix the app-owned plugin boundary only if the failing code is causally related to this plan; otherwise record the unrelated failure with evidence.
- If workflow outcome mapping becomes ambiguous, add a specific `DesktopWorkflowOutcome` variant for plugin command invocation and denial rather than reusing a generic success label.
- On interruption, inspect `git diff`, compare modified files against `files_modified` and `files_forbidden`, rerun targeted verification, and continue only from unverified work.

</recovery>

## Task 1: Add Desktop Plugin Command Routing

### Read Targets

- `crates/devil-ui/src/ui.rs`
- `crates/devil-desktop/src/bridge.rs`
- `crates/devil-app/src/lib.rs`
- `crates/devil-app/tests/workspace_vfs_integration.rs`

### Write Targets

- `crates/devil-desktop/src/bridge.rs`
- `crates/devil-desktop/tests/plugin_management.rs`

### Implementation Sequence

1. Add a desktop action for projected plugin command invocation with plugin id, command id, and display-safe metadata label.
   > verification: `rg -q "InvokePluginCommand" crates/devil-desktop/src/bridge.rs`
2. Validate the plugin id and command id against `ShellProjectionSnapshot::plugin_contribution_projections` before dispatch. Unknown plugin ids and unknown command ids must return typed bridge errors and no app intent.
   > verification: `cargo test -p devil-desktop plugin_management_unknown -- --nocapture`
3. Route accepted desktop plugin commands to the existing `CommandDispatchIntent::InvokePluginCommand` only. Do not import `devil-plugin` into `devil-desktop`.
   > verification: `cargo test -p devil-desktop plugin_management_command_routes_to_app_intent -- --nocapture`

### Edge And Error Cases

- Reject empty plugin ids or empty command ids.
- Reject commands not present in the current projection even when the plugin id exists.
- Keep labels display-safe and do not copy raw plugin output, payloads, or manifest bodies into errors.

<verify>

```powershell
rg -q "InvokePluginCommand" crates/devil-desktop/src/bridge.rs
cargo test -p devil-desktop plugin_management_unknown -- --nocapture
cargo test -p devil-desktop plugin_management_command_routes_to_app_intent -- --nocapture
```

</verify>

### BLOCKED Conditions

- A safe command id cannot be obtained from the current projection.

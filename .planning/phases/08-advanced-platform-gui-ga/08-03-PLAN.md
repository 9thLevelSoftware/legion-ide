# Plan 08-03: Collaboration Presence And Shared Proposal GUI Workflow

## Objective

Make collaboration presence, session status, reconnect/offline indicators, conflict visibility, and shared proposal review usable in the GUI while keeping collaboration mutation and proposal lifecycle authority in app/editor/proposal services.

<execution_contract>

- read-before-write: read `08-CONTEXT.md`, `08-02-RESULT.md`, `crates/devil-protocol/src/lib.rs`, `crates/devil-ui/src/ui.rs`, `crates/devil-app/src/lib.rs`, `crates/devil-desktop/src/bridge.rs`, `crates/devil-desktop/src/workflow.rs`, `crates/devil-desktop/src/view.rs`, `crates/devil-app/tests/workspace_vfs_integration.rs`, `plans/evidence/gui-productization/phase-8-plugin-management.md`, and `AGENTS.md` before editing.
- evidence-before-action: confirm existing app-owned collaboration entry points and proposal lifecycle actions before adding any GUI shared-proposal review surface.
- minimal diff: add only projection data, action routing, view rows, tests, evidence, and result content needed for collaboration GUI workflows.
- verify-before-report: run every frontmatter command and record outcomes in `08-03-RESULT.md`.
- role: collaboration GUI implementer and regression verifier.
- task: expose collaboration presence and review workflows through projections and app-owned actions.
- scope: collaboration session/presence rows, reconnect/offline rows, conflict rows, shared proposal review links, bridge validation, workflow outcomes, and evidence.
- allowed tools/actions: focused Rust/Markdown edits, targeted app/desktop tests, `rg`, and `xtask` dependency checks.
- forbidden actions: direct editor mutation from UI/desktop, direct collaboration runtime ownership in UI/desktop, direct proposal state mutation outside app proposal actions, raw transport payload logging, legacy Phase 8 evidence edits, dependency additions, and test weakening.
- verification criteria: join/leave/presence actions route through existing app intents; shared proposal review uses proposal preview/approve/reject/apply routing; conflict and reconnect states are visible as metadata; tests prove no direct editor mutation path was added.
- final result format: write `.planning/phases/08-advanced-platform-gui-ga/08-03-RESULT.md` with changed files, command outcomes, and any `BLOCKED` items.

</execution_contract>

<stop_gates>

- BLOCKED if collaboration conflict or reconnect status cannot be represented without raw operation payloads or raw editor text.
- BLOCKED if shared proposal review requires UI or desktop ownership of proposal lifecycle state.
- BLOCKED if collaboration operations would mutate editor buffers directly from UI or desktop code.
- BLOCKED if app-owned collaboration runtime must be enabled by default to make GUI tests pass.
- BLOCKED if existing app collaboration tests fail due to a boundary regression caused by this plan.

</stop_gates>

<recovery>

- If current projections lack reconnect/conflict/shared-proposal summary fields, add the smallest DTOs in `devil-protocol` and thread them through `devil-ui` and `devil-app`; do not add raw operation bodies.
- If desktop action validation cannot prove a session exists, deny the action and render a visible no-session status instead of dispatching an unchecked intent.
- If proposal review already has sufficient desktop actions, reuse existing proposal actions and add collaboration-context rows instead of duplicating proposal lifecycle controls.
- On interruption, inspect `git diff`, compare modified files against `files_modified` and `files_forbidden`, rerun targeted verification, and continue only from unverified work.

</recovery>

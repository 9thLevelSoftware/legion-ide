# Codebase Map: Legion IDE

Generated by `/legion:map --refresh` from the current checkout. Current source files remain the source of truth if this map conflicts with live code.

## Architecture

Legion IDE is a Rust 2024 Cargo workspace for a local-first IDE. The current product track is GUI productization: the repo now contains both the legacy `legion-app` CLI shell proof and the newer `legion-desktop` renderer-backed adapter. Product authority remains outside the renderer.

Core boundary:

- `legion-protocol` is the shared DTO, validation, and port-contract crate. It is the highest fan-in crate and contains editor, workspace, proposal, trust, LSP, semantic, terminal, plugin, collaboration, remote, telemetry, retention, and storage contracts.
- `legion-app` is the composition root. It owns application orchestration, active document state, proposal-mediated saves, language tooling projection refresh, default-deny terminal workflow routing, plugin/collaboration/remote composition, search, storage, and observability wiring.
- `legion-ui` remains projection-only. It accepts `ShellProjectionSnapshot` and emits `CommandDispatchIntent`; it must not own editor text, workspace state, proposal state, storage, terminal sessions, providers, plugins, collaboration, or remote state.
- `legion-desktop` owns native windowing, egui/eframe rendering, adapter-local presentation state, session JSON read/write, input-to-intent translation, smoke harnesses, and display models derived from projections.
- `legion-editor`, `legion-text`, `legion-project`, `legion-platform`, `legion-security`, `legion-observability`, and `legion-storage` provide the current local authority substrate.
- `legion-index`, `legion-terminal`, `legion-ai`, `legion-ai-providers`, `legion-agent`, `legion-memory`, `legion-tracker`, `legion-plugin`, `legion-collaboration`, `legion-remote`, `legion-remote-transport`, `legion-telemetry`, and `legion-retention` provide activated or gated runtime surfaces that must stay behind protocol, policy, proposal, and privacy boundaries.

The high-risk implementation pattern is a deliberate anti-layering rule: renderer and UI code project and dispatch, while `legion-app` and lower authority crates perform stateful work and mutation mediation.

## Code Patterns

### Projection-only UI pattern

`legion-ui` exposes display DTOs and command intents. `legion-desktop` consumes those projections and maps UI actions back into intents; neither should own editor/workspace/proposal state.

### App-owned authority routing pattern

`AppComposition::dispatch_ui_intent` routes `CommandDispatchIntent` into app command requests. Direct edits go through editor authority; saves go through `SaveWorkflowService`; terminal and language operations update projections through app-owned workflows.

### Proposal-mediated mutation pattern

File saves and edit-producing language/AI actions become proposals or proposal previews before mutation. Stale/conflict/denial outcomes preserve dirty text and return rejected outcomes rather than pretending success.

### Metadata-only audit pattern

Observability, storage, terminal, plugin, AI, collaboration, remote, telemetry, and retention surfaces use metadata-only records with non-zero correlation/event/causality expectations where applicable.

### Default-deny capability pattern

Terminal, provider, plugin, collaboration, remote, telemetry, and retention behavior should begin denied or inert, then become available only through explicit policy contracts and tests.

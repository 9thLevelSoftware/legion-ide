# Codebase Map: Devil IDE

Generated by `/legion:map` from the current checkout. Current source wins if any section conflicts with later edits.

## Architecture

Devil IDE is a Rust 2024 Cargo workspace organized as a local-first IDE core with explicit authority boundaries. The current runnable application is `devil-app`, a CLI shell proof that opens the current directory as a trusted workspace and supports `:w` and `:q`. It is not a real GUI renderer.

The architecture is projection-led:

- `devil-ui` is projection-only. It accepts `ShellProjectionSnapshot` values and emits `CommandDispatchIntent` values. It must not own editor sessions, workspace state, file text, save state, or renderer-side business logic.
- `devil-app` is the composition root. It wires workspace, editor, proposal, security, storage, observability, AI, plugin, collaboration, remote, and UI projection surfaces.
- `devil-project` owns workspace VFS, root-boundary path checks, tree discovery, watcher recovery, proposal-mediated mutations, and file-save preconditions.
- `devil-editor` owns buffers, transactions, undo/redo, save request DTOs, snapshot leases, viewport projections, and dirty state.
- `devil-text` owns rope-backed text buffers, line indexes, immutable snapshots, UTF-8/UTF-16 conversion, and the 5 MiB full-cache budget.
- `devil-protocol` is the shared DTO and port contract layer. It is very large and is the highest fan-in crate.
- `devil-security`, `devil-observability`, and `devil-storage` enforce policy, metadata-only audit posture, and persistence/replay boundaries.
- `devil-index`, `devil-ai`, `devil-agent`, `devil-memory`, `devil-tracker`, `devil-plugin`, `devil-collaboration`, `devil-remote`, `devil-terminal`, `devil-telemetry`, `devil-retention`, and `devil-remote-transport` contain activated or partially gated feature surfaces that must remain behind ADR, dependency policy, and contract evidence.

The product gap for a fully functional GUI IDE is not "build the editor core from zero"; it is selecting and integrating a renderer-backed desktop shell while preserving the existing projection/intent boundary.

## Code Patterns

- `#![warn(missing_docs)]` appears across key crates; public APIs are documented.
- Boundary-oriented DTOs and ports live in `devil-protocol`; implementation crates depend inward on the protocol layer.
- Save mutations are proposal-mediated and fail closed on stale/conflict/denial.
- UI surfaces consume snapshots and emit typed intents instead of owning mutable domain state.
- Metadata-only audit is the default. Raw source, terminal output, provider payloads, and hosted egress are explicit exceptions requiring later gates.
- Large-file behavior is bounded. Text snapshots materialize a full cache only within the 5 MiB budget; viewport/chunk APIs carry degraded state.
- `xtask` encodes governance checks and must be kept in sync with dependency policy and protocol symbols.
- Naming is crate/domain-driven with `devil-*` packages and snake_case Rust modules/functions.
